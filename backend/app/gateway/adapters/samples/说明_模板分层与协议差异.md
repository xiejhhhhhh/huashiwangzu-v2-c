# 网关响应模板库 — 分层约定与协议差异

这里存**真实探针打回来的裸响应体**,给后端做适配时 1:1 对照,不猜。

## 命名约定(华哥定:服务商→模型,别乱放)

```
{模型}_via_{服务商}[_协议][_流式SSE].json
```

- `via_{服务商}`:同一个模型,不同服务商响应可能不一样,必须按服务商分开存
- 协议后缀:`ant`(Anthropic /v1/messages)、`res`(GPT /v1/responses)、不写=chat(老 OpenAI /v1/chat/completions)
- 流式加 `_流式SSE`,里面按 `[{event,data},...]` 顺序存事件序列

## 三种协议(华哥 2026-07-16 定)

| 协议 | 端点 | 请求体 | 响应体 | 流式必须同步的模板 |
|------|------|--------|--------|--------------------|
| chat | `/v1/chat/completions` | `messages[]` | `choices[].message.content` | `choices[].delta.content` |
| res  | `/v1/responses` | `input[]`+`instructions` | `object=="response"`,`output[]` | `output`增量事件 |
| ant  | `/v1/messages` | `messages[]`+`max_tokens`必填 | `content[]`(block数组) | `content_block_delta` |

**流式铁律**:后端返回什么协议,请求者就得用什么协议模板吃实时数据。不能 chat 的解析器去吃 ant 的 SSE。

## 同协议、不同服务商的差异(保留冗余度的原因)

同样是 **ant 协议**,两个服务商响应结构就不一样:

### deepseek官方 (api.deepseek.com/anthropic)
- `content[]` 里**带 thinking block**:`{"type":"thinking","thinking":"...","signature":"..."}` 在前,`text` 在后
- 流式:先一串 `thinking_delta`,`content_block_stop` 后换 index,再 `text_delta`
- usage 带 `service_tier`
- 认证:`x-api-key`,直连可通
- 模型:deepseek-v4-flash / deepseek-v4-pro

### jayce中转站 (jayce.sky1818.com,claude真身)
- `content[]` **只有 text**(不主动带 thinking)
- usage **无 service_tier**,input_tokens 很大(中转站塞了系统提示,6000+)
- 认证:`x-api-key` 或 `Bearer` 都行
- **必须走代理 127.0.0.1:4780**,直连 HTTP 000
- 模型:claude-opus-4-8(贵,企业用不上,仅备)

## 网关职责边界(华哥定)

- 网关只**收信息、返信息**,不管模型选哪个
- **模型底座(adapter层)负责完美适配**:把各服务商各协议的差异,归一成统一的 ModelResponse/StreamEvent
- 调用方只报标识符(用途),底座看标识符查 models.json 决定 服务商+协议+降级链

## 三协议归一现状(2026-07-16 补完)

前端/网关出口永远只吃统一事件 `{type: token/thinking/tool_call/done/error}`,
不用管后端选了谁——这就是"输入→返回,同频复用"的双向同步。

| 协议 | 非流式归一 | 流式归一 | 位置 |
|------|-----------|---------|------|
| chat | ✓ | ✓ | base.py `_build_stream_chunk` choices[].delta 分支 |
| res  | ✓ `_build_responses_response` | ✓(本次补) | base.py `_build_stream_chunk` `response.*` 分支 |
| ant  | ✓(本次补) | ✓(本次补) | adapters/anthropic.py `AnthropicAdapter` |

**本次修复的真 bug**:res 协议流式之前只按 chat 的 `choices[].delta` 解析,
res 事件是 `response.output_text.delta`(无 choices)→ 每个 token 被当空丢掉。
gptstore-text/local 都是 res,流式必丢字。已补 `response.*` 分支,实测 token 不丢。

**ant 接线**:AnthropicAdapter 已注册(claude→AnthropicAdapter),非流式+流式都通。
但还差一个 AnthropicProvider(构造 /v1/messages 请求 + 读 ant SSE),
因这两个服务商现在不参与运行,provider 等接入时再补(adapter 已就位,即插即用)。
