## 每处改动的根因

### 1. modules/knowledge/manifest.json
- 清空 `supported_formats` 为 `[]`
- **根因**: 知识库声明了几乎所有文件格式，导致 `getAppByFileFormat` 把所有双击文件路由到知识库。知识库是 AI 分析工具，不应是文件打开器。清空后不影响上传→ingest 管线（该管线不依赖 supported_formats）。

### 2. frontend/src/desktop/app-registry/file-association-registry.ts
- 移除 legacy 格式（doc/xls/ppt）硬编码返回 `filePreview` 的短路逻辑
- 更新 `inferFormatCategory` 以识别新 viewer 模块的 appKey
- **根因**: `filePreview` 应用不存在（悬空），legacy 格式应走正常匹配逻辑路由到对应的新查看器（doc→doc-viewer, ppt→ppt-viewer, xls→excel-engine）。移除后，新 viewer 模块的 `supported_formats` 声明即生效。

### 3. frontend/vite.config.ts
- 新增 `pdfjs-dist` resolve alias 指向 `frontend/node_modules/pdfjs-dist`
- **根因**: 模块文件位于 `modules/pdf-viewer/frontend/`，Vite Rolldown 无法从该路径解析 `pdfjs-dist`。与已有 `marked`/`dompurify`/`highlight.js` 别名同一模式，是框架允许的公共能力扩展。

### 4. frontend/src/desktop/app-registry/component-key-map.generated.ts
- 自动生成，新增 5 个 viewer 模块的组件入口
- **根因**: `npm run scan:modules` 扫描新模块并注册组件键映射。

### 5. frontend/src/shared/components/viewer-shell.vue
- 新建统一 WPS 风格查看器外壳组件
- **根因**: 所有新建查看器复用统一壳，保证风格一致（浅色高对比、品牌色 #2395bc、苹方字体）。顶部工具栏（文件名 + 操作按钮）+ 中间内容区插槽 + 底部状态栏插槽。

### 6. modules/image-viewer/ (图片查看器)
- 纯前端模块，只读
- 通过 `GET /api/files/download/{id}` 加载图片，缩放/拖动/平移/适应窗口/下载
- 支持的格式: png/jpg/jpeg/gif/bmp/webp/svg/ico

### 7. modules/text-editor/ (文本/MD 编辑器)
- 前后端分离，纯前端，可编辑
- 加载文本内容 → textarea 编辑 → 上传覆盖保存
- MD 文件支持"预览/源码"切换（简易 Markdown→HTML 渲染）
- 支持的格式: txt/md/log/json/yaml/yml/xml/ini/cfg；可编辑: txt/md/log/json/yaml/yml/xml/ini/cfg

### 8. modules/pdf-viewer/ (PDF 查看器)
- 纯前端，使用项目自带的 pdf.js 渲染
- 翻页/缩放/适配/页码跳转/搜索 UI
- 支持的格式: pdf

### 9. modules/doc-viewer/ (Word 文档查看器)
- 纯前端，通过跨模块能力调用 `docx-parser:parse` 获取 JSON 结构
- 渲染为可读文档（段落 + 表格）
- 支持的格式: docx, doc（leagcy doc 尽力而为）

### 10. modules/ppt-viewer/ (PPT 演示文稿查看器)
- 纯前端，通过跨模块能力调用 `pptx-parser:parse` 获取 JSON 结构
- 缩略图侧栏 + 主区域逐张渲染幻灯片
- 支持的格式: pptx, ppt

## JSON 中间层与 Agent 编辑

| 能力 | 实现方式 |
|---|---|
| docx 解析 JSON | 已有 `docx-parser:parse` 能力（doc-viewer 前端调用） |
| pptx 解析 JSON | 已有 `pptx-parser:parse` 能力（ppt-viewer 前端调用） |
| JSON→docx 生成 | 已有 `office-gen:docx` 能力（通过 `/api/modules/call`） |
| JSON→pptx 生成 | 已有 `office-gen:pptx` 能力 |
| text 读写 | text-editor 前端直接读写（下载/上传接口） |
| 补丁系统 | 已有 `JsonPatchService`（backend/app/services/office/json_patch_service.py） |

Agent 编辑路径：Agent 通过 `platform.modules.call` → `/api/modules/call` 网关 → 调用各 parser/office-gen 能力，已完成闭环。本期未新增能力，全复用现有注册能力。

## 已知局限

1. PDF 文本搜索未实现（需 pdf.js textContent API 异步获取，当前为简化 placeholder）
2. DOC 老格式（.doc）解析依赖 docx-parser 兼容能力，可能失败
3. PPT 老格式（.ppt）同理
4. text-editor 保存创建新文件（上传接口），而非覆盖原文件记录
5. doc-viewer/ppt-viewer 为只读，编辑功能留待后续（可通过 JSON patch 系统实现）
6. 图片查看器不支持多图片翻页（无批量打开场景）

## 试过但没成的方案

1. **动态 import pdfjs-dist from 模块内**: 直接 `import('pdfjs-dist')` 在模块内无法被 Rolldown 解析 → 需在 vite.config.ts 加 resolve alias（与 marked/dompurify 同模式）
2. **模块放 backend router**: 考虑过给 doc-viewer/ppt-viewer 加 backend router 做中间层 → 最终决定纯前端直接调 `/api/modules/call`，省掉无意义的转发层
