# 实施计划：受控投递 owner 4 的高收益 GPT 批次

## 核心原则

本次不会直接调用现有 `enqueue_incomplete_documents`，因为它只能处理调用者 owner 1、会从 `source_validate` 启动完整 DAG，并可能继续产生 Profile / Graph / Relations 任务，不符合今晚“只跑有收益的云端阶段”的要求。

采取两层处理：

1. **先补一个安全的 Knowledge 管理入口**，允许管理员准确指定 owner 和单一阶段，但默认只预览、强制确认、自动防重，并且执行完该阶段后不再自动向下游扩散。
2. **再做 5～20 份 owner 4 PDF 小批试点**，确认 GPT Cockpit 的实际质量与稳定性后才逐级扩大。

## 二、增加“单阶段受控批量投递”能力

新增 admin Capability：`enqueue_pipeline_stage_batch`。

### 能力参数

- `owner_id`：目标数据所有者，今晚明确使用 owner 4；未传时才回落到调用者。
- `stage`：严格只允许：
  - `raw_ocr`
  - `raw_vision`
  - `fusion`
- `dry_run`：默认 `true`。
- `confirm`：真实投递必须等于固定确认字串 `ENQUEUE_KNOWLEDGE_STAGE_BATCH`。
- `audit_reason`：记录本次投递原因。
- `limit`：默认 20，并设置安全上限。
- `document_ids`：支持精确文档 ID，便于试点和复跑。
- `extensions`、`filename_contains`：可选筛选条件。
- `priority`：有界优先级。

### 安全和收益筛选

- 查询始终包含 `KbDocument.owner_id == owner_id`，绝不做无 owner 范围扫描。
- 过滤 deleted、源文件不存在或物理路径不可用的文档。
- `raw_ocr/raw_vision`：只选视觉文档，并要求页面资产已存在；已有成功页面由现有阶段逻辑直接复用，不重复调用 GPT。
- `fusion`：只选 parse 已就绪、所需 raw rounds 完整且 Fusion 尚未完成的文档；不把必然 blocked 的文档塞进队列。
- 已完成阶段默认跳过，不提供无差别强制重跑。
- 复用现有 `enqueue_pipeline_stage_task` 的 `(document_id, stage)` advisory lock 和 active-task 去重。
- dry-run 不创建任务、不 commit，并返回候选、跳过原因和预计投递数。
- apply 时先检查平台 maintenance gate，再一次性提交有界批次。

### 防止自动扩散

扩展 Knowledge DAG 任务参数，加入可选 `stop_after_stage`：

- 管理批次总是设置为当前指定阶段。
- Dispatcher settlement 发现完成阶段等于 `stop_after_stage` 时：
  - 不发布 Profile、Graph、Relations 或其他 successor；
  - 正确结束本次 pipeline run，并记录 `stopped_after_requested_stage` 诊断。
- 普通文档完整 DAG 不传该参数，现有行为保持不变。

### 审计归属

- 队列 `owner_id` 和任务内 `user_id` 保持目标 owner 4，确保阶段读取和产物归属正确。
- `requested_by` 记录实际管理员 actor，而不是伪装成 owner 4。
- `audit_reason`、目标 owner、阶段和筛选条件进入任务/返回结果，便于追查。

## 四、测试重点

1. 默认 dry-run 不写库、不调用 enqueue、不 commit。
2. 错误或缺少 confirm 时拒绝真实执行。
3. 只允许 OCR / Vision / Fusion，拒绝 Embedding / Profile / Graph / Relations / 任意字符串。
4. owner 4 查询不会混入 owner 1 或其他 owner 文档。
5. actor 与 target owner 分开记录。
6. 非视觉文档、缺页面资产、源文件不可用、Fusion 上游不完整、阶段已完成均有准确 skip reason。
7. 同文档同阶段在途任务不会重复创建；不同阶段仍可独立创建。
8. `stop_after_stage=fusion` 后不会产生 Profile / Graph；Vision/OCR 同理不会自动扩散。
9. 普通完整 DAG 未传 stop 参数时保持现有 successor 语义。
10. runtime capability 与 manifest 参数无漂移。

## 六、owner 4 小批试点与自动停止门槛

### 第一批：Fusion 优先

1. 用新 Capability 对 owner 4 做 dry-run。
2. 精确选择 5～20 份：
   - PDF；
   - parse ready；
   - raw rounds 完整；
   - Fusion 缺失；
   - 页面数适中、内容有业务价值；
   - 无 active duplicate。
3. 人工检查 dry-run 返回的文件名、页数、状态和 skip 原因。
4. 携带 confirm 做真实投递，且每项任务都设置 `stop_after_stage=fusion`。
5. 监控队列、stage runs、analysis artifacts、页面 Fusion 和后端错误日志。

### 只有缺原料时才跑 VLM

- 若 Fusion 候选因 round 2/3 缺失而不足，再分别 dry-run：
  - 缺 OCR 的页面 → `raw_ocr`
  - 缺视觉理解的页面 → `raw_vision`
- 每个阶段单独停止；两轮都齐后，再显式投递 Fusion。
- 已有成功 VLM/OCR 页面不重跑。
- 今晚不启用本地 VLM；必要 OCR 仍走 GPT Cockpit 云端。CLIP 本轮不需要启动，因为当前试点已有明确候选，不为“可能有用”额外消耗本机资源。

### 扩量 Gate

小批满足以下条件才扩到 50，再扩到 200 份文档；这里的 200 是批量规模/并发上限，不是必须跑满：

- GPT 调用成功率 ≥ 90%。
- 401/403：出现一次立即停止。
- 429、5xx、超时或失败率 > 10%：暂停扩量并诊断。
- fallback 必须为 false；发现 DeepSeek 或本地模型立即停止。
- 目标页面 Fusion 非空覆盖率 ≥ 95%。
- 抽查至少 5 份，摘要与原文/图片一致，无明显张冠李戴、空洞套话或大段重复。
- 没有产生 Profile、Graph、Relations、Embedding successor。
- 本机资源保持正常；不启动本地 Embedding/VLM。

若质量提升无法从页面 Fusion 内容和后续检索样例中观察到，则停止扩量，不因 Token 充足而继续跑。
