- V2 当前：
  - `modules/agent/backend/handlers/chat.py:101-571`
  - `modules/agent/backend/engine/engine.py:32-188`

所以 V2 的 runtime 升级，不能只学 Claude 的 plan agent，也要把状态恢复和工作区边界一起纳入。

### 2. 记忆系统

- V2 当前：
  - `modules/agent/backend/engine/layered_memory.py:436-675`

交叉结论：

- Hermes 让 V2 学会“会后整理和演进”

- Claude 让 V2 学会“召回不等于注入”

所以 V2 记忆升级的核心，不是再加召回层，而是：

1. 记忆分层更清楚
2. surfacing 更克制
3. 文件 / 证据 / 记忆之间的界面更明确

1. query rewrite
2. retrieval grading

1. 什么文件在 working set 里
2. 什么文件只是证据，不进主上下文
3. 什么输出只是 draft，不能自动 publish

### C1. Document IR 统一层 — Docling × Unstructured × V2 现有 JSON 包 交叉印证

#### 真相矩阵

| 维度 | V2 当前 (`backend/app/models/office.py`) | Docling (`docling-core/.../document.py`) | Unstructured (`elements.py`) |

2. **但 V2 的内容 schema 太简单** — `{manifest, content}` 缺乏元素类型、坐标、资源引用、元数据扩展
3. **Docling 的 DoclingDocument 是最适合 V2 的 IR 模型** — 同时有树结构(给 viewer/workflow) 和平面列表(给 search/embedding)，Pydantic model 天然可序列化
4. **Unstructured 的 ElementMetadata 的"稀疏字段 + 合并策略"设计可直接用于知识库 chunk 元数据**

## 节点 1：Agent 运行时主循环

### 当前 V2 逻辑

- 入口在 `modules/agent/backend/handlers/chat.py:101-478`
- 当前链路是：

  - 已经有降级链
  - 已经有事件持久化
  - 已经有 tool loop
- 当前不足：
  - 编排职责过于集中在 `chat.py`
  - 主循环、流式输出、权限判断、工具恢复、持久化没有彻底分层

### 我们怎么改

1. 把 `handlers/chat.py` 里的运行时编排外提成统一 runtime service
2. 把主会话、review、subagent、理解环都统一挂到 execution context

### 执行锚点

### 提精度抓手

1. 先做任务复杂度分级，再决定是否进入理解环 / review fork

3. 把“只读规划”“证据检索”“执行落地”拆成显式 runtime role，避免一个循环里混做所有事

### 目标收益

- 改一条链不会牵动整个 chat handler
- review fork、理解增强环、批量 subagent 都能共用一套 runtime
- 返工成本显著下降

- 是否走 understanding loop
- 是否切换模型模板

- `fast`
- `balanced`
- `deep`

- 多检索一步

### E. 风险确认开关

如果任务有不可逆动作，就强制：

- critic 一轮

## 结论先行

如果只盯 Agent，本结论只够一半。结合 Hermes、Claude 和 OpenClaw 之后，我认为下一阶段应该同时看到两层：

- **Agent 层**：持续自改进、技能演进、委派与记忆
- **平台层**：控制平面、能力注册、任务/会话治理、安全策略、插件化与成熟度度量

所以主线不该只剩两条，而是至少四条：

1. **把 Agent 做成会持续自改进的系统**
2. **把技能系统从 prompt 注入升级成可治理、可演进、可审计的能力层**
3. **把平台层做成真正的控制平面，而不是模块拼盘**
4. **把会话、任务、权限、沙箱、插件和质量度量收进统一治理面**

如果只选一个最高优先级方向，我的建议是：

**先做 `background review fork + skill lifecycle` 这一组联动改造。**

原因很简单：这组改造能把 Hermes 最有价值的“持续学习”能力，接到我们已经有的 V2 运行时骨架上；而 Claude 提供的价值，主要是把这件事做得更工程化、更可控、更可观测。

1. 后台 review fork
2. 技能 lifecycle
3. 统一 runtime/tool 治理

MemoryProvider 插件化更像第二阶段的结构性增强，不是第一阶段最赚钱的改造。

### C. Hook 跨 worker 不再是下一主线

这一条现在也应该降级。

根据你收回来的补丁结果和我对当前方向的判断：

- DB 级 leader election 已补
- leader 续租语义已补
- observer stop 写回边界已补
- 诊断字段语义已补

这条主线现在更像“持续观察项”，不是下一阶段核心升级抓手。

## 4. 我自己复核后新增强调的几条问题

这些不是外部回件没提，而是我认为应该在下一阶段决策里加重权重。

### A. 当前 gateway 的模型治理能力仍然偏弱

我实读确认：

- `backend/app/gateway/config.py`
- `backend/app/gateway/service.py`
- `modules/agent/backend/engine/budget_allocator.py`

当前模型层已经完成了服务层纯净化，但治理能力仍然偏轻：

- profile 是静态配置
- fallback chain 是静态线性链
- `context_budget` 只是单值读取
- 没有 task-aware model routing
- 没有 budget / latency / reliability 综合策略
- 没有 local/cloud 动态裁决

这说明“模型网关纯净化”是对的，但它只是边界清理，不是模型治理完成。

### B. 上下文装配已经很丰富，但缺二次选择层

我复核 `engine.py assemble_context()` 后，判断当前 V2 的问题不是“信息太少”，而是“信息越来越多，但缺统一选择层”。

当前 system prompt 会叠加：

- static memory
- system prompt
- enterprise prompt
- agent config prompt
- user profile
- skills
- three-layer memory
- workflow strategy
- experience memory

这说明当前瓶颈已经从“没有注入源”转成：

**缺统一的 relevance / weighting / pruning 机制。**

如果不补这层，后面再接 review fork、skill lifecycle、memory refinement，只会继续增噪。

### C. `handlers/chat.py` 已经承担过多运行时编排职责

我复核后比较明确的一点是：

`handlers/chat.py` 现在已经不只是 endpoint handler，它实际上承担了：

- tool loop
- streaming
- approval 处理
- stuck detector
- diminishing budget
- inline tool recovery
- persistence 节点

这说明下一阶段如果做 runtime 治理统一化，入口很明确：

**先把 `handlers/chat.py` 里的运行时编排职责继续外提。**

## 5. 这轮问题汇总后的升级方向结论

如果把“外部回件结论”“我自己的源码复核结论”以及这轮新增的 OpenCode / Dify / Coze / OpenClaw / Hermes 源码实读一起压平，下一阶段最值得投带宽的方向已经从四条扩成六条：

1. **Background Review Fork**
2. **技能系统工业化**
3. **运行时治理统一化**
4. **平台控制面升级**
5. **模型治理控制面**
6. **理解增强型多代理循环**

其中真正的起手式，我仍然建议是：

**先做 `Background Review Fork + skill_manage 最小 CRUD + 技能写审批门`。**

因为这组改造能同时验证三件事：

- Hermes 的学习闭环值不值得接
- V2 的技能系统能不能从注入层升级成资产层
- 当前 runtime 骨架能不能承住后台自改进流程

## OpenCode、Dify、Coze 补充后的新增判断

这三条线补进来之后，原方案还有明显深化空间，而且是结构性深化，不是边角料补丁。

## 1. OpenCode 最值得补给 V2 的，是“模型治理控制面”

这轮实读后，我认为 OpenCode 最有价值的，不是它也有 agent，而是它已经把模型相关对象拆得比 V2 成熟很多：

- provider
- model
- variant
- agent role
- provider budget
- model TPM limiter
- runtime permission / agent mode

尤其是它把“同一个模型的不同运行形态”做成了 variant：

- `fast`
- `low`
- `high`
- `max`
- `thinking`

这对 V2 很关键，因为我们现在虽然有 `models.json`，但本质还是：

- 静态 profile
- 静态主模型
- 静态线性 fallback

它还不是一个真正的模型治理控制面。

### 这意味着什么

V2 下一阶段不能再把模型层只当“换个 provider 地址”的问题处理了。

应该把它升级成下面这些一级对象：

1. `provider`
2. `model`
3. `variant/template`
4. `routing policy`
5. `budget / rate / health`
6. `agent role binding`

## 2. Dify 最值得补给 V2 的，是“应用化工作流平台”

Dify 给我的最大提醒不是 agent 技巧，而是：

**它已经把 workflow / RAG / model management / observability / worker queues 做成了产品平台。**

这对 V2 的提醒很直接：

- 未来不能只有 Agent 对话入口
- 还要有 workflow app / chatflow / dataset / pipeline 这类“可发布业务形态”
- 观测不能只停留在 agent_events，要走向 trace / ops / run history / annotation / eval

我不建议现在把 V2 直接改造成 Dify 那种“大而全平台”，但 Dify 证明了一件事：

**如果后面要做企业级 AI 平台，workflow、dataset、plugin、observability 迟早会变成一等公民。**

## 3. Coze Studio 最值得补给 V2 的，是“资源图谱 + 低代码工作流面”

Coze Studio 最打动我的，不是“也能做 Agent”，而是它把资源对象做得很清楚：

- agent
- workflow
- plugin
- knowledge
- database
- prompt
- connector

它不是把这些能力藏在一堆工具调用里，而是把它们明确建模成资源。

这对 V2 很重要，因为我们现在虽然有模块和 capability，但还没有把“Agent 所依赖的资源图谱”显式建模出来。

从长期看，V2 需要逐步具备：

1. 资源对象层
2. 工作流编排层
3. 发布/连接器层
4. 运行/追踪层

这条线不一定是下一批马上开做，但应该进入总图，而不是继续停在“对话 Agent”视角里。

## Hermes、Claude、OpenClaw、V2 综合判断

## 1. Hermes 最值得借的，不是“记忆插件”，而是“后台回顾学习闭环”

Hermes 真正高价值的东西有两个：

1. **background review fork**
2. **完整的 skill lifecycle**

其中第一项更重要。

原因：

- V2 已经有 memory、experience、hook、subagent、task queue
- 我们缺的是“谁来在对话后主动复盘，然后驱动 memory/skill 改进”
- Hermes 的 background review 正好补的是这个调度层

相比之下，`MemoryProvider` 插件接口是有价值，但**不应排在第一位**。因为 V2 已经不是纯文件记忆，它本身有独立 memory 模块和跨模块 capability；现在更缺的是学习闭环，而不是多 provider 抽象。

## 2. Claude 最值得借的，不是某个单点功能，而是“统一运行时治理”

Claude 这条线最强的是工程化完整度，主要体现在：

- `QueryEngine.ts` 统一维护会话生命周期、memory prompt、tool 权限、usage、abort、partial message、snip/compact
- `tools.ts` 有一个清晰的全局工具注册面，并且跟 feature flag、permission mode、preset、environment 紧密耦合
- `skills/loadSkillsDir.ts` 不只是“扫 markdown”，而是连 frontmatter、hooks、argument、execution context、model override、loadedFrom、plugin source 一起收编
- `memdir/findRelevantMemories.ts` 说明 Claude 的 memory surfacing 更强调“选择哪些记忆给主模型看”，而不是盲目全塞
- `DreamTask.ts` 说明它会把后台子流程显式映射成 task，可见、可取消、可诊断

Claude 的启发不是“照抄工具清单”，而是：

**把 Agent 的运行时、技能、权限、记忆、后台任务，收进一个统一的治理面。**

## 3. OpenClaw 最值得借的，不是某个 agent 技巧，而是“控制平面化”

OpenClaw 跟 Hermes、Claude 的气质不一样。

它最值得学的不是“对话更聪明”，而是：

- Gateway 是长期运行的控制平面
- channels / nodes / web / automation / agent / session 都挂在同一控制面下
- 插件不是散落功能，而是 capability model
- sandbox、tool policy、elevated、session route、thread binding 都是一级对象
- 后台任务不是附属日志，而是 task ledger
- 文档里显式有 maturity scorecard，而不是只靠体感说“成熟”

这对 V2 的意义很大，因为我们现在虽然有框架和模块边界，但还没有形成真正的平台控制平面。

OpenClaw 这轮实读后，我认为最值得借的五件事是：

1. **控制平面视角**
2. **插件/能力注册模型**
3. **session / thread / route 建模**
4. **sandbox / tool policy / elevated 分层**
5. **background task ledger + maturity scorecard**

## 4. V2 当前最弱的几块，正好落在三家强项交集里

这次看下来，不止两块，而是六块：

### A. 技能生命周期

V2 当前只有：

- `skill_list`
- `skill_describe`
- `skill_use`
- `skills_loader.py` 的 markdown 发现与 prompt 注入

缺的是：

- skill create/edit/patch/delete
- 写审批门
- 安全扫描
- provenance
- usage 统计
- 低质量技能清理
- bundle / preload / fork context

这块 Hermes 强于我们很多，Claude 在 metadata / load path / hook / context 上也更成熟。

### B. 后台自改进闭环

V2 当前有：

- `PostTurnHooks`
- `profile_evolve`
- `memory_distill`
- `experience` 蒸馏

但没有：

- 受限工具集的 review fork
- review 结果的 task surfacing
- review 产物审批/溯源
- review 驱动 skill 改进

这里 Hermes 给了学习闭环，Claude 给了任务可视化和运行时治理。

### C. 平台级能力注册与插件化

V2 当前的跨模块能力调用已经比普通项目规整，但仍然偏“总线式调用”，还不是完整 capability platform。

缺的是：

- capability taxonomy
- capability owner / shape / metadata
- startup activation planning
- metadata snapshot / registry cache boundary
- 对“谁拥有什么能力”的稳定约束

这块 OpenClaw 最强，Claude 次之，Hermes 较弱。

### D. 会话 / 线程 / 路由控制面

V2 当前 Agent 还是偏单入口心智。  
OpenClaw 明显更强的是：

- 多 session
- 多 thread
- 多 route
- 多 delivery path
- 多 requester / delegate identity

这不是为了照搬多聊天渠道，而是提醒我们：

**Desktop、Agent、IM、Scheduler、Docs-open 最后都需要统一的 session/thread/task 语义。**

### E. 安全与运行边界分层

Claude 和 OpenClaw 共同的优点是：

- 不把所有安全收敛成一个开关
- 而是把 permission、tool policy、sandbox、elevated、source trust 分层治理

V2 这块有意识，但还没体系化。

### F. 背景任务与成熟度治理

OpenClaw 给我的额外启发很强：

- detached work 要有 task ledger
- task 要可通知、可取消、可审计、可追踪
- 平台要有成熟度评分，不只是单次验收绿了就算完

这两块在 V2 当前仍然很弱。

### G. 模型治理控制面

这条是这轮新增强调的主问题。

V2 当前模型层的问题已经不是“有没有 fallback”，而是：

- 没有 provider registry
- 没有 model capability schema
- 没有 variant / template
- 没有 task-aware routing
- 没有 budget / TPM / latency / health 联动
- 没有 local/cloud/privacy 策略裁决
- 没有按 agent role 绑定模型职责

OpenCode、Dify、Coze 都从不同侧面证明了这一层的重要性。

### H. 理解增强型多代理循环

这条也是这轮新增强调的主问题。

用户提的方向本身是成立的：

**不是所有问题都必须靠顶级模型解决，很多时候真正缺的是“理解过程”和“分工过程”。**

Claude 的 plan/general agent、Hermes 的 delegation、OpenClaw 的多 agent route、Coze/Dify 的 workflow 都在证明：

- 复杂任务可以拆阶段
- 不同阶段可以用不同 agent 角色
- 不是每一步都要最强模型

所以“弱一些的模型 + 更好的理解编排”是可行路线，但必须做成受控流程，而不能变成无边界循环。

## 下一阶段建议路线

## 第一主线：Background Review Fork

### 建议级别

**最高优先级，建议单独开重构任务。**

### 目标

在每次对话后，由一个受限子 Agent 在后台做 review，只允许：

- memory 相关工具
- skill 管理工具
- 少量只读项目工具

它的职责不是继续对话，而是：

- 判断哪些内容该写入 stable rule / chunk / experience
- 判断是否该创建新技能
- 判断是否该改已有技能
- 输出结构化 review 结果

### 借鉴源

- Hermes：`background_review.py`
- Claude：`DreamTask.ts`、任务显式可视化思路

### 建议改造点

- `modules/agent/backend/engine/post_turn_hooks.py`
- `modules/agent/backend/handlers/tool.py`
- `modules/agent/backend/handlers/tasks.py`
- `modules/agent/backend/engine/engine.py`
- 新增 review task / event / result schema

### 预期收益

- Agent 获得持续自改进能力
- memory/skill 不再只靠人工维护
- 用户每次纠错都可能沉淀成长期资产

### 风险

- 低质量 review 可能污染记忆或技能库
- 后台 fork 会增加 token 成本

### 必须搭配的防护

- review 只允许受限工具集
- 默认先走“建议/审批”而不是直接写入
- review 结果要有 provenance 和 task 可追踪

## 第八主线：子 Agent / 委派 / 后台任务可观测

### 建议级别

**中优先级。**

- 轮数上限明确
- 工具循环明确
- 但缺少更强的任务化、可观测、批量委派、事件汇聚

### 借鉴源

- Hermes：delegation / async delegation 思路

### 建议方向

- 子 Agent 执行结果进入统一 task/event 面板
- 支持批量委派
- 支持结构化结果聚合
- 支持更明确的 cancellation / timeout / provenance

### 这条线的前提

先把“统一 tool/runtime 治理”做起来，否则越做越散。

## 第十主线：模型治理控制面

### 建议级别

**高优先级。**

这条现在应该被正式抬到主线，而不是继续作为 gateway 清理附属项。

### 目标

把当前 V2 的模型配置，从“静态 profile + 线性 fallback”升级成真正的模型治理控制面。

### 借鉴源

- OpenCode：provider / model / variant / provider budget / model TPM limiter / agent role
- Dify：大规模 model provider 支持 + model management
- Coze Studio：后台 model management + 资源化配置

### 当前 V2 的真实短板

当前仓库里我已经确认：

- `backend/data/config/models.json` 仍是静态配置中心
- `backend/app/gateway/service.py` 只负责静态取 profile/provider
- `modules/agent/backend/engine/fallback_chain.py` 仍是线性降级
- `backend/app/gateway/opencode_provider.py` 里的 `opencode` 只是一个 provider endpoint 包装，不是完整模型治理层

### 我建议新增的一级对象

1. `provider`
   - cloud / local
   - auth profile
   - health
   - rate limit
   - budget policy
2. `model`
   - context window
   - reasoning support
   - tool calling
   - vision / attachment
   - structured output
   - latency class
   - privacy class
   - cost class
3. `variant/template`
   - `fast`
   - `balanced`
   - `deep`
   - `longctx`
   - `local_private`
   - `vision`
   - `planner`
   - `reviewer`
   - `summarizer`
4. `routing policy`
   - task type
   - complexity
   - ambiguity
   - privacy requirement
   - context size
   - modality
   - latency SLO
   - cost ceiling
5. `fallback policy`
   - 不再只是线性链
   - 应该是按失败类型和约束做条件降级

### 建议落地方式

第一阶段先不做 UI，先做后端控制面：

1. 扩展 `models.json` 为 `provider + model + variant + policy` 结构
2. 抽 `ModelRouter`
3. 抽 `ModelCapabilitySchema`
4. 抽 `ProviderHealth/BudgetTracker`
5. 让 `AgentConfig` 不再只绑单一 `model`，而是允许绑 `model template` 或 `role policy`

### 预期收益

- 本地模型 + 云端模型能共存
- 不同任务能自动匹配不同模型
- 便宜模型、隐私模型、强模型能各司其职
- 后面做 subagent / review / compression / summarizer 时不需要全都吃同一个模型

## 第十一主线：理解增强型多代理循环

### 建议级别

**中高优先级。**

它不是第一枪，但很值得进入第二批骨架设计。

### 结论先行

用户提出的方向在工程上是可行的，而且经常是有效的。  
但它的价值主要在：

- 提高需求理解质量
- 提高复杂任务的顾虑覆盖率
- 提高计划质量

它**不是**对所有任务都划算，也**不是**无限多轮就一定更准。

### 借鉴源

- Claude：`plan` / `general` 这类专职 agent
- Hermes：delegation / async delegation / review fork
- OpenClaw：多 agent route / task ledger
- Coze / Dify：workflow 分阶段执行

### 正确做法

我建议把它建成“理解增强环”，而不是“盲目多开几个 agent”。

推荐的受控流程：

1. `intent_clarifier`
   - 复述用户真实目标
   - 标出歧义点
2. `concern_miner`
   - 找用户隐含顾虑、边界、禁忌、成本敏感点
3. `plan_critic`
   - 质疑主计划的漏洞、遗漏、风险
4. `retrieval/evidence`
   - 只补事实和证据
5. `main_orchestrator`
   - 汇总成 `understanding packet`
   - 决定是否进入第二轮理解
   - 再进入执行

### 必须加的边界

1. **只在高歧义 / 高复杂 / 高代价任务触发**
2. **最多 2-3 轮**
3. **每个子 agent 角色固定**
4. **优先只读工具**
5. **产物必须结构化回收**
6. **要有 stop condition**

### 什么时候值得触发

- 用户需求模糊
- 用户表达里有多重目标冲突
- 任务代价高
- 执行动作有不可逆性
- 之前已经出现过误解或返工
- 主模型置信度低 / 分歧高

### 什么时候不该触发

- 简单问答
- 低风险工具调用
- 明确的单步执行
- 用户只要直接结果，不需要长理解链

### 为什么这条路值得做

因为它能把“模型不够强”的问题，部分转化成“流程是否足够聪明”的问题。

对于不少任务：

- 用一个顶级模型硬顶
- 不如用一个较便宜模型做主会话，再配几个便宜/中档角色模型做理解增强

真正提升的不是 token 神话，而是：

- 语义对齐率
- 风险覆盖率
- 计划完备度

### V2 当前的差距

V2 虽然已经有 `agent:spawn_subagent`，但还缺：

- 角色化子 agent
- 批量委派
- 结构化回收
- 结果裁决
- 事件树可观测
- 多轮理解 stop condition

所以这条线不能直接靠现在的 `spawn_subagent` 裸扩。

应该先基于统一 runtime 治理，做一个：

**`understanding loop orchestrator`**
