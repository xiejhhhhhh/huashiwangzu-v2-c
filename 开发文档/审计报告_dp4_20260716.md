## 1. 🔴 BLOCKER — 需立即处理

### B1. `profile_evolve` 持续性失败 — 数据类型不匹配

**严重度**: CRITICAL — 用户画像演进管道完全中断

**根因定位**:

```
文件: modules/agent/backend/services/profile_evolve.py:312
代码: profile.profile_data = merged  # merged 是 dict {}
```

`profile_data` 列类型是 `text`（`agent_user_profile` 表），但 handler 将 Python `dict` 直接赋值给 ORM 模型的 `profile_data` 字段。当 SQLAlchemy 尝试执行：
```sql
UPDATE agent_user_profile SET profile_data=$1::VARCHAR, ...
```
asyncpg 收到 `{'tone': '...', 'focus': [...]}`（dict）而非字符串 → 编码失败 → DataError。

**证据链**:
- task_handlers.log: `Task 315263 (profile_evolve) handler failed`
- 完整错误: `invalid input for query argument $1: {'tone': '正式、直接…' (expected str, got dict)`
- `agent_user_profile` 表 6 条记录全部 `profile_data='{}'`（版本号 0，从未成功演进）

**修复方向**: 第 312 行 `profile.profile_data = merged` → `profile.profile_data = json.dumps(merged, ensure_ascii=False)`；或修改模型列类型为 `JSONB`。

### B2. 数据库连接池事务泄漏

**严重度**: CRITICAL — API 接口抛 500

**根因定位**:

```
文件: backend/app/database.py:32
代码: AsyncSessionLocal = async_sessionmaker(engine, expire_on_commit=False, class_=AsyncSession)
```

多个 handler（尤其 `handlers.log` 中 `POST /api/modules/call`）中的 session 在 `__aexit__` 时尝试 `Transaction.rollback()`，但底层 asyncpg 连接已被池回收或关闭，导致二次异常：

```
sqlalchemy.exc.InterfaceError: cannot call Transaction.rollback():
the underlying connection is closed
```

**证据链**:
- `handlers.log`: 13:41:48、13:42:17 各爆发一次，间隔约 30 秒（或与连接池 `pool_recycle` 配置有关）
- `modules.log` 同时段也有同类 Traceback
- 同时刻数据库 `idle` 连接 204 个（system_resource_snapshot），暗示连接未及时归还

**配置现状**:
```python
pool_size: settings.DB_POOL_SIZE        # 当前值待确认
max_overflow: settings.DB_MAX_OVERFLOW  # 当前值待确认
pool_recycle: settings.DB_POOL_RECYCLE_SECONDS  # ≥60秒
pool_pre_ping: True
```

**修复方向**: 在 session 上下文中加入 `session.is_active` 守卫；检查 `DB_IDLE_IN_TRANSACTION_TIMEOUT_MS` 配置是否过于宽松；考虑将 `expire_on_commit` 改回 `True`。

### B3. 知识库管道 9138 条"运行中"孤立记录

**严重度**: HIGH — 管道状态不可信

**证据**:
```sql
SELECT status, COUNT(*) FROM kb_pipeline_runs GROUP BY status;
-- done=19279, running=9138, degraded=6012, failed=299, paused=62, skipped=2829
```

`running=9138` 远超并发上限（lane 最大 40），几乎确定是历史孤立记录未被清理。上次 reconcile（2026-07-13）仅发现 4 条 orphan diagnostic running。

**修复方向**: 用 `reconcile_orphan_pipeline_runs` 全量清理（limit=10000, dry_run=false）；为 pipeline_runs 加 TTL heartbeat 超时自动 stale 标记。

## 7. 🕵️ 第二波深度审计 — 补充发现

> 第二轮审计覆盖：安全认证链、命令执行沙盒、API 密钥硬编码扫描、跨模块权限路由、EventBus 健康、前端模块加载

### S1. 🔴 认证系统 — 登录限流内存级 + Token localStorage 存储

**文件**: `backend/app/services/auth.py:16`
```python
_login_attempts: dict[str, list[float]] = {}  # 进程内纯内存
MAX_LOGIN_ATTEMPTS = 5
LOGIN_LOCKOUT_MINUTES = 15
```

**问题**:
- **限流非持久** — `_login_attempts` 是进程内 dict，重启后端即清零。攻击者可通过反复重启绕过限流
- **Token 存 localStorage** — 所有前端模块统一用 `v2_auth_token` key 存储在 `localStorage`。标准 SPA 做法，但 XSS 可窃取；无 httpOnly cookie 方案兜底
- **密码 bcrypt** ✅ — 使用 bcrypt 哈希，密码不自曝

**风险等级**: MEDIUM（有重器登录爆破面，但内网环境 + 密码足够复杂时风险可控）

### S2. 🟢 terminal-tools 命令执行安全 — 设计合理

**文件**: `modules/terminal-tools/backend/handlers/exec.py` + `sandbox.py`

**审计结论**:
- **macOS 沙盒保护** ✅ — 使用 `sandbox-exec` 内核级沙盒隔离，只读写用户工作区
- **危险命令拦截** ✅ — `_check_dangerous_command()` 拦截 rm -rf / 等
- **路径逃逸检查** ✅ — `_check_path_escape()` 拦截 ../ 逃逸

**注意**: 执行路径是 `["sandbox-exec", "-p", profile, "/bin/sh", "-c", command]`，即沙盒化 sh -c。如果沙盒配置有漏洞仍可能逃逸，但属 macOS 内核级，非应用层可控。

### S4. 🟡 跨模块权限路由 — `_resolve_caller_role` 设计需注意

**文件**: `backend/app/services/module_registry.py:193-218`

```python
def _resolve_caller_role(caller, caller_role, *, actor=None, trusted_user_role=False):
    if caller.startswith("user:"):
        if not trusted_user_role and caller_role != "viewer":
            raise PermissionDenied("user caller role must be resolved...")
        return caller_role
```

**分析**:
- 默认 `call_capability()` 传入 `caller_role="viewer"`、`trusted_user_role=False`
- 实际授权走 **SQL 数据库**（`_authorize_user_capability` → `assert_capability_authorized`），不依赖此返回值
- 此函数仅做格式校验 + 日志记录角色
- **关键安全保证**: handler 收到的 `caller` 是 `"user:{id}"` 格式，不附带角色信息，handler 自己如果要做额外权限检查必须自己查 DB

**结论**: 设计正确，无权限绕过风险 ✅

### T4. 🟡 TimeoutMiddleware 异常类型覆盖不完整

**文件**: `backend/app/middleware/timeout_middleware.py:48`

```python
try:
    return await asyncio.wait_for(call_next(request), timeout=TIMEOUT_SECONDS)
except asyncio.TimeoutError:
    return JSONResponse(status_code=504, ...)
```

只捕获 `asyncio.TimeoutError`，未捕获 `asyncio.CancelledError`（task 被取消时抛）或其他异常。如果 handler 在被 `wait_for` 取消后抛其他异常，中间件不会拦截，返回 500。

**建议**: 加 `except Exception` 兜底。

### T7. 🟡 `url_safety` DNS 解析同步阻塞事件循环

**文件**: `backend/app/core/url_safety.py:117`
```python
addr_info = socket.getaddrinfo(hostname, None, socket.AF_UNSPEC, socket.SOCK_STREAM)
```

`socket.getaddrinfo` 是同步阻塞调用，在异步 FastAPI handler 中执行会阻塞整个事件循环。如果上游 DNS 响应慢，会拖慢所有并发请求。
