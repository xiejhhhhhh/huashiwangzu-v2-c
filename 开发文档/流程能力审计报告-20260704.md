# 流程能力审计报告（2026-07-04）

## 审计目标

本次审计按“整套能力链路”取证，不做无限发散式代码扫描，也不做代码修复。审计顺序从数据库与持久化结果反推，到后端能力/API/任务，再到前端显示与用户下一步操作，最后检查发布门禁是否能拦住同类问题。

本轮使用 5 个只读子代理并行审计：

- 数据库/持久化健康
- 后端统一通路与能力链
- 前端用户可见流程
- 模块接入与 sandbox 门禁
- Agent + Knowledge + Content IR 主流程

- 已读入口文档：`开发文档/README.md`、`开发文档/01_框架开发文档/README.md`、`开发文档/02_底层开发文档/README.md`、`开发文档/03_模块开发文档/README.md`

- 后端 smoke：`smoke_all(skip_ui=true)` 29 场景，28 绿、1 跳过，结论 `PASS_WITH_DEBT`

## 总体结论

项目主运行态不是“全局坏掉”：健康检查、模块注册、sandbox、大部分 happy path 都能跑通。真正的问题在流程闭环：数据能写进去，任务能完成，接口能返回 200，但用户最终看到的状态、下一步操作、发布门禁和清理治理没有完全闭合。

本轮没有发现必须立即停机的 P0；但发现多条 P1 级流程问题，尤其是文件生命周期、Knowledge/Content 派生数据、后端假绿语义和前端空态吞错。

## P1 问题

### 1. 文件删除后 Knowledge/ContentPackage 形成半孤儿

链路：

`文件上传 -> file.uploaded -> Knowledge/Content 生成派生数据 -> 文件删除进回收站 -> Knowledge 标 source_file_deleted 但不归档 -> ContentPackage 无删除事件处理 -> 用户列表看不到，后台看板爆红`

证据：

```sql
-- Knowledge active 文档里 163/165 指向已删除源文件
SELECT COUNT(*) active_docs,
       COUNT(*) FILTER (WHERE fi.deleted=false) visible_source_docs,
       COUNT(*) FILTER (WHERE fi.deleted=true) hidden_source_deleted_docs
FROM kb_documents d JOIN framework_file_items fi ON fi.id=d.file_id
WHERE d.deleted=false;
-- 结果：165 / 2 / 163

-- ContentPackage 204 active，198 个 source_file 已删除
SELECT COUNT(*) packages,
       COUNT(*) FILTER (WHERE deleted=false) active_packages,
       COUNT(*) FILTER (WHERE source_file_id IS NOT NULL AND EXISTS (
         SELECT 1 FROM framework_file_items fi
         WHERE fi.id=framework_content_packages.source_file_id AND fi.deleted=true
       )) source_file_deleted
FROM framework_content_packages;
-- 结果：204 / 204 / 198
```

活系统表现：

- `/api/knowledge/documents?page=1&page_size=3` 只返回 `total=2`
- `/api/knowledge/dashboard/stats` 返回 `total_documents=165`、`failed_documents=163`、`source_unavailable_documents=163`
- `/api/health` 仍显示 task queue clean

代码证据：

- `backend/app/routers/files.py` 发出 `file.deleted`
- `modules/knowledge/backend/services/file_lifecycle_service.py` 只标记源文件不可用
- ContentPackage 未见对应 `file.deleted/file.restored/permanent_deleted` 生命周期处理

修复方向：

- 给 ContentPackage 补文件生命周期事件处理：源文件删除时软归档，恢复时恢复关联状态，永久删除时清理或冻结派生数据
- Knowledge 删除事件不要只暂停，应进入治理队列：恢复源文件 / 归档知识文档 / 删除派生数据
- 批量治理当前 163 个 source-deleted 文档、198 个 source-deleted ContentPackage

### 2. 测试数据只进回收站，持续污染用户文件域和知识库

链路：

`smoke/e2e 创建文件 -> 删除只进入回收站 -> Knowledge/Content 派生数据继续 active -> 用户回收站、知识库看板、统计长期污染`

证据：

```sql
SELECT COUNT(*) total_files,
       COUNT(*) FILTER (WHERE deleted=false) active_files,
       COUNT(*) FILTER (WHERE deleted=true) deleted_files
FROM framework_file_items;
-- 结果：303 / 6 / 297

SELECT COUNT(*) recycle_rows,
       COUNT(*) FILTER (WHERE item_type='file') file_recycle_rows
FROM framework_file_recycle_items;
-- 结果：297 / 297
```

样本包含大量 `smoke-*`、`recycle-*`、`e2e-*-to-delete` 文件。后端 smoke 本轮显示“清理 9 个测试文件”，但实际只是删除到回收站，未清理 Knowledge/Content 派生数据。

修复方向：

- smoke/e2e 使用独立测试 owner 或强 marker
- 测试清理必须永久删除测试文件，并同步清理 Knowledge/Content 派生表
- release gate 加 SQL 门禁：active/recycle/knowledge/content 中测试 marker 必须为 0

### 3. `/api/modules/call` 仍存在 error-only 假绿

链路：

`能力 handler 返回 {"error": "..."} -> /api/modules/call 只检查 success:false -> 外层 ApiResponse success:true -> 调用方误判成功`

证据：

- `backend/app/routers/modules.py` 只在 `result.get("success") is False` 时抛错
- `modules/agent/backend/handlers/tool.py` 多处返回 `{"error": ...}`，不是异常或 `success:false`
- 子代理活系统复现：`agent:update_system_prompt {}` 返回 HTTP 200 / `success:true` / `data: {"error": "content is required"}`

主线程另测：

- `codemap:search {"keyword": ""}` 经 `/api/modules/call` 返回 422
- 但直连 `/api/codemap/search` 返回 HTTP 200 + `success:false`

修复方向：

- 抽统一 `semantic_failure_reason(result)`
- 覆盖 `success:false`、顶层/内层 `error`、`status in failed/error`、legacy `code != 0`
- `/api/modules/call`、模块 HTTP 包装、smoke `_cap_ok`、任务 worker 共用同一判断

### 4. 前端加载失败会伪装成空态/正常态

链路：

`后端有数据 -> API 失败/权限/网络瞬断 -> 前端 catch 清空状态 -> 用户看到空桌面、空文件夹、反馈中心“没有问题”`

证据：

- `/api/files/list?folder_id=0` 活系统有根文件，但 `frontend/src/desktop/shell/use-desktop-root-files.ts` 失败时设为空数组
- 文件管理器 `fm-state.ts` 失败时也置空
- `use-notifications.ts` 多个 catch 把 unread/list/task/agent 状态置 0、空数组或 null
- `notification-panel.vue` 空态文案会显示“现在没有需要处理的反馈”

修复方向：

- 引入统一 `LoadState<T> = loading | ready | error | stale`
- 失败时保留上次成功数据，显示错误和重试
- 禁止核心列表/反馈信号在 catch 中无提示清空

## P2 问题

### 9. 活系统 capability registry 与 manifest/source 参数元数据漂移

链路：

`manifest public_actions -> 源码 register_capability -> 活系统 /api/modules/capabilities -> Agent/前端/工具台发现能力`

证据：

- `scripts/check-capability-drift.py` 静态检查通过：183 registered public capabilities
- 活系统 `/api/modules/capabilities` 返回 205 条能力
- 子代理发现 live registry 参数键有 10 个 mismatch，集中在 Agent workflow 能力；磁盘源码注册参数完整，说明可能是后端未重启或运行时注册状态旧
- `scheduler:create` runtime 比 manifest 多 `title`
- `terminal-tools:chart` runtime 比 manifest 多 `title`

修复方向：

- release gate 加 live registry vs manifest/source 参数键检查
- `/api/health` 或 system status 暴露 registry build/version/source hash
- 源码变更后 registry 未刷新时 degraded

### 10. 任务提交端允许不可执行任务先成功入队

`POST /api/tasks/submit -> 不校验 task_type handler -> 返回 success -> worker 后续才 No handler registered`

证据：

- `backend/app/routers/tasks.py` submit 直接创建 pending
- `backend/app/services/task_worker.py` 执行阶段才检查 handler

修复方向：

- submit 阶段校验 handler 是否注册
- 或响应明确区分 `accepted:true` 与 `executed/succeeded`
- 暴露 registered handler 列表供前端/模块预校验

### 11. `/api/health` 与 task audit 语义失败口径重复实现

链路：

证据：

- `backend/app/services/task_queue_audit_service.py` 用 JSON decode 后规则判断

修复方向：

- health 复用 `audit_task_queue()` 分类结果
- 删除 SQL 字符串匹配口径

### 12. API 错误被前端泛化，后端明确原因到不了用户

链路：

`后端返回 File not found / App not found / PermissionDenied / Conflict -> axios/组件泛化 -> 用户只看到“失败/不存在”`

证据：

- `/api/files/detail/999999999` 返回 `File not found`
- `/api/desktop/apps/no-such-app` 返回 `App not found`
- `shared/api/response-transform.ts` 对 404 固定成“内容不存在或已被删除”
- 多个文件操作 catch 只显示“上传失败/创建失败/删除失败”

修复方向：

- 统一 `ApiError`，保留 `http_status/code/backend_message/user_message`
- 组件层统一 `displayApiError(e, fallback)`
- 禁止空 catch 和无细节泛化

### 13. 反馈中心缺少可操作下一步

链路：

`任务审计/Agent 状态聚合 -> 面板只显示数字/文案 -> 用户无法跳到 Knowledge/Agent/任务详情处理`

证据：

- `notification-panel.vue` 任务问题只显示失败/债务数字
- 文案提示“请到对应模块查看详情”，但没有目标按钮
- 活系统 task audit 可返回 `module=knowledge` 等定位信息，前端未转成 action

修复方向：

- 反馈中心 signal 加 action model：`targetApp/module/taskId/actionLabel`
- 支持打开 Knowledge、Agent、任务详情
- 没有目标时说明“无法定位”的原因

### 14. 组件接入可被静默吞掉

链路：

`manifest.component_key -> scan-modules -> app_service sync -> app-loader -> 普通模块可能空窗口`

证据：

- `frontend/scripts/scan-modules.js` 对缺前端入口直接跳过
- `backend/app/services/app_service.py` 对缺组件文件清空 component_key
- `frontend/src/desktop/app-registry/app-loader.ts` 把 `!entryKey` 当 background-service 占位
- 现有测试固定了“普通模块无前端入口也变空 component_key”的行为

修复方向：

- 非 background-service 且可见模块必须有真实 `frontend/{component_key}`
- background-service 必须显式空 component_key
- normal app 空 entryKey 应显示注册错误，不能静默 placeholder

### 15. 发布门禁未覆盖文档可复现性和能力漂移

链路：

`项目文档要求每模块 README 写可复现验收命令 -> module_sandbox_matrix 不检查 README -> release_gate 不检查 capability drift -> PASS_WITH_DEBT 仍 release_safe`

证据：

- 模块文档要求 README 写 sandbox/main framework/测试数据清理等验收命令
- `module_sandbox_matrix.py` 只看 sandbox/test_module/backend/package，不看 README
- 缺 README 模块：`browser-tools`、`csv-parser`、`github-search`、`markdown-parser`、`structured-parser`
- `knowledge` README 未写 sandbox/test_module 验收口径
- release gate 当前未调用 `scripts/check-capability-drift.py`
- `PASS_WITH_DEBT` 的 `release_safe=true` 容易被误读成干净发布

修复方向：

- README 验收矩阵机器化：缺失为 DEBT，新模块/变更模块缺失为 BLOCKER
- release gate 增加 `Capability contract` 阶段
- 拆分 `deploy_allowed` 与 `clean_release_ready`

## 本轮未判为 bug 的点

- `memory_*` 空表：当前接口返回空，未发现用户使用后丢数据证据；暂按未使用/已清理状态处理
