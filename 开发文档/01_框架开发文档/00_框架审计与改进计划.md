# 00_框架审计与改进计划

## 审计范围

前端（`frontend/`）与后端（`backend/`）框架层的兼容性、性能、一致性审计。模块层（`modules/`）后置处理。

---

## A. 后端改进（backend/）

### A1. 统一模型配置源 — 消除 registry.py 与 models.json 双源问题

**问题**：`backend/app/services/model_watchdog/registry.py` 硬编码注册了 6 个模型（bge-m3、bge-reranker、qwen3-vl、gemma-4、opencode-go、mimo），同时 `backend/data/config/models.json` 又定义了一套相同的模型配置。看门狗用 registry.py，模型网关用 models.json。改一个模型参数需要在两个地方改。

**目标**：registry.py 改为只从 `models.json` 的 `watchdog_models` 段动态加载，消除硬编码。

**改动范围**：
- `backend/app/services/model_watchdog/registry.py` — 移除所有 `register(ModelRecord(...))` 调用，改为从 `models.json` 加载
- `backend/data/config/models.json` — 已经是单一事实源，不需要改
- `backend/app/services/model_watchdog/watchdog.py` — 确认不影响（已通过 `get_model()` / `list_models()` 调用）

**验收**：
```bash
cd backend && python -c "from app.services.model_watchdog.registry import list_models; print(len(list_models()))"
# 输出 6（与 models.json watchdog_models 条目数一致）
```

---

### A2. 数据库连接池加固

**问题**：`backend/app/database.py` 只设置了 `pool_size=20`，缺少 `pool_recycle`、`pool_pre_ping`、`pool_timeout`。PostgreSQL 长连接会被网关断开（通常 5-30 分钟），池中脏连接会导致偶发的 `SSL connection has been closed` 错误。

**目标**：添加连接保活与超时参数。

**改动范围**：
- `backend/app/database.py` — `create_async_engine()` 添加：
  ```python
  pool_size=20,
  max_overflow=10,
  pool_recycle=3600,     # 1 小时回收
  pool_pre_ping=True,     # 每次借用前检查连接活性
  pool_timeout=30,        # 等待连接的超时
  ```

**验收**：后端运行 2 小时后，检查日志没有 `SSL connection has been closed` 错误。

### A3. CORS 收紧

**问题**：`config.py` 中 `CORS_ORIGINS: list[str] = ["*"]` — 生产环境不应允许所有来源。

**目标**：默认只允许本机，运行时可通过 `.env` 配置。

**改动范围**：
- `backend/app/config.py` — 默认值改为 `["http://127.0.0.1:5173", "http://localhost:5173"]`

**验收**：
```bash
curl -v -X OPTIONS -H "Origin: https://evil.com" http://127.0.0.1:30004/api/health 2>&1 | grep "Access-Control-Allow-Origin"
# 应不包含 evil.com
```

## B. 前端改进（frontend/）

### B1. 消除转中文() 函数 — 统一使用英文字段名

**问题**：`frontend/src/shared/api/response-transform.ts` 中 `转中文()` 函数包含 ~100 条 `snake_case→中文` 硬编码映射，运行在每次 API 响应上。V2 是英文优先架构，不应再维护这套转换层。它同时与 `统一响应` 接口的 `success/data/error` 字段名冲突——消费者代码读 `res.成功` 但接口类型声明是 `success`。

**目标**：删除 `转中文()` 函数，所有组件直接使用后端返回的英文 `snake_case` 字段名（`success`、`data`、`error`、`items`、`total` 等）。

**改动范围**：
- `frontend/src/shared/api/response-transform.ts` — 删除 `转中文()` 函数
- 扫描所有 `res.成功`、`res.数据`、`res.错误` 访问，改为 `res.success`、`res.data`、`res.error`
- 扫描所有 `res.数据.xxx` 访问，改为 `res.data.xxx`
- 组件和方法调用中的中文属性名对齐为英文

**扫描命令**：
```bash
cd frontend && rg "\.成功\b\|\.数据\b\|\.错误\b" src/ -g "*.ts" -g "*.vue"
```

**验收**：
```bash
cd frontend && npx vue-tsc -b && npm run build
# 0 错误
```

### B2. Vite 代码分割优化 — 缩小主包体积

**问题**：`index-BwV2DhQF.js` 956KB（超过 500KB 阈值），`dist-B3PtPfTz.js` 171KB。主要是 Element Plus 全量引入 + pdfjs + highlight.js + marked。

**目标**：通过 Vite 的 `manualChunks` 配置将稳定依赖拆分为独立 chunk，减少主包体积。

**改动范围**：
- `frontend/vite.config.ts` — 在 `build.rollupOptions.output.manualChunks` 中拆分：
  ```typescript
  build: {
    rollupOptions: {
      output: {
        manualChunks: {
          'element-plus': ['element-plus', '@element-plus/icons-vue'],
          'pdf': ['pdfjs-dist'],
          'editor': ['highlight.js', 'marked'],
        },
      },
    },
  },
  ```

**验收**：
```bash
cd frontend && npm run build && ls -la dist/assets/*.js | awk '{print $5, $NF}'
# element-plus chunk 与主包分离，index-[hash].js < 500KB
```

### B3. 添加 Vue 错误边界

**问题**：模块内组件抛出的异常会冒泡至根组件，可能导致整个桌面壳白屏。没有 `<ErrorBoundary>` 兜底。

**目标**：在桌面壳的应用窗口容器层捕获渲染异常，保证一个模块崩溃不影响其他模块和桌面壳本身。

**改动范围**：
- 新建 `frontend/src/desktop/components/component-error-boundary.vue`
- 在桌面壳的应用窗口容器组件中包裹每个模块实例

**验收**：在模块组件中手动 `throw new Error('test')`，桌面壳不崩溃，错误组件显示降级 UI。

### B4. API_BASE_URL 环境变量可配

**问题**：`frontend/src/shared/api/index.ts` 硬编码 `export const API_BASE_URL = '/api'`。Vite 的 `server.proxy` 只在开发环境生效，生产环境需要改代码才能切换后端地址。

**目标**：通过 Vite 构建时环境变量配置 API 地址。

**改动范围**：
- `frontend/.env.production` 新建文件，添加 `VITE_API_BASE=/api`
- `frontend/src/shared/api/index.ts` — `API_BASE_URL` 改为 `import.meta.env.VITE_API_BASE || '/api'`

**验收**：
```bash
VITE_API_BASE=/api/v2 npm run build
grep -c "/api/v2" dist/assets/*.js  # 应包含新的 base URL
```

### B5. 应用注册表加载器零值处理

**问题**：`frontend/src/desktop/app-registry/app-loader.ts` 中 `transformApiToEntry` 使用 `取值()` 辅助函数同时尝试 `snake_case` 和 `camelCase`，这是一种次优的防御性编程方式。

**目标**：确认后端返回字段稳定后，直接使用 `snake_case` 字段名（Python 后端标准），删除 `取值()` 函数。

**改动范围**：
- `frontend/src/desktop/app-registry/app-loader.ts` — 直接读取确定的字段名：
  ```typescript
  appKey: app.app_id,
  appName: app.name,
  defaultWidth: app.default_width,
  ```
- 删除 `取值()` 辅助函数

**验收**：
```bash
cd frontend && npx vue-tsc -b # 0 错误
```

## C. 架构决策记录

请将以下决策记录加入 `01_框架开发文档/README.md` 的「目标模块扫描规则」段落后。

### C1. 模型配置数据流

```text
models.json  ← 单一事实源（编辑这里）
    │
    ├─→ watchdog_models → registry.py（动态加载）→ 看门狗
    │
    ├─→ model_types.llm.profiles → router.py（ModelGatewayRouter）
    │
    ├─→ model_types.embedding / rerank / vision → 对应服务
    │
    └─→ providers → 网关 Provider 工厂
```

1. 用户/开发者改模型只在 `models.json` 改一处。
2. 看门狗启动时从 `models.json` 的 `watchdog_models` 段读取。
3. `registry.py` 不得再硬编码 `ModelRecord`。
4. 添加新模型 = 在 `models.json` 的 `watchdog_models` + 对应 `model_types` 段 + `adapters/` 中各加一条。

### C2. API 字段命名约定

**后端**（Python Pydantic）：`snake_case`
- 例：`entry_component_key`、`default_width`、`created_at`

**前端**（TypeScript）：接收时保持 `snake_case`，内部使用时可按需转换为 `camelCase` 但必须在单一位置（如类型定义或 composable）处理，不得在每个组件中重复适配。

**不要使用 `转中文()`** 将 API 字段名翻译为中文键名。V2 不是多语言前端，不维护翻译层。

### C3. 错误处理策略

**后端异常链**：

```text
视图层 → 服务层抛 AppException（或子类）
       → register_exception_handlers 统一捕获
       → 返回 { success: false, data: null, error: "xxx", errors: null }
```

**前端错误处理**：

```text
api.interceptors.response.use → 捕获 HTTP 错误
                              → 401：重试 1 次 → 重定向到首页
                              → 其他：节流日志 + 显示 ElMessage
                              → Promise.reject(友好化错误信息)
```

组件内使用 `try/catch` 或 composable 级错误处理，不依赖全局状态传递错误对象。

### C4. Auth 令牌策略

- JWT 令牌：`HS256` 签名，默认 24 小时过期，嵌入 `session_version` 以便退出登录使旧令牌失效。
- 无 `refresh_token`：前端 401 后重试 1 次，仍失败则清除令牌并跳转首页。
- Token 存储：`localStorage`（`v2_auth_token` 键）。
- 登录成功：服务器 `session_version` +1，使该用户所有旧令牌同时失效。
