# 11 语义打齐引擎 + 图片多维分字段方案

## 一、背景与第一性原理

知识库召回不准,病根在**数据没语义清洗**:图片(1万+张)经 OCR/VLM 提取的实体名带错字
("华世王镞"被识别成"华世王族/链/嫒/璞"),同一实体裂成一堆变体,搜规范名带不出、搜变体各命中一点。

**第一性原理(华哥定)**:文本直接提取的文本层,**字100%正确**(除语序乱)。
所以文本层 = 唯一权威真值。图片 OCR/VLM 的错字,全靠**全库文本层权威**拉回。
数据脏源在图,权威锚永远是文本层。一次命中,命中不了才降级。

**数据现状(owner_id=4)**:
- 非图片文档 4175,其中 91.5% 有文本层 → 基础级一次命中
- 图片文档 11453,全部有 OCR/VLM 原始数据(kb_raw_data)+ 融合层(kb_page_fusions)
  - 知识**已提取**(不是锁在像素里),问题是**提取质量**(错字)
  - 图片无自己的文本层,不能自校准,只能靠全库文本层(3821篇)拉正

## 二、语义打齐引擎(第一层,已完成)

文件:`modules/knowledge/backend/services/semantic_align_service.py`

### 核心尺子:逐字位滑窗,文本层当权威
对一个实体名的每个汉字位置,遮住它,用左右窗口去干净文本层(base_parse 非图片)查这一位
该是啥字。文本层碾压者≠当前字 → OCR/VLM 错字,就地改;干净名每位都一致 → 不动。
**单个名字即可自纠,不需兄弟变体**,天然全自动、按位算、零硬编码、零LLM。

### 五重护栏(精度第一,防误伤)
1. **只纠汉字**:英文设计师码/标点/数字原样保留
2. **权威字也必须汉字**:不拿标点/字母替换汉字
3. **左右窗口都非空,且只取相邻汉字**:碰数字/标点即停(否则"广花X路237号"里门牌号把上下文
   锁死,掩盖竞争真值,把"广花二路"误并进"广花一路")
4. **当前字真值保护**:当前字在该位文本层出现 ≥3 次 = 真值(不是错字),不动
   (例:"广花X路"里"二"出现8次→广花二路是真路;"世王X集团"里"族"0次→是错字)
5. **改后完整名文本层命中**:改后的整名必须在干净文本层逐字命中≥1篇才接受
   (把"公司名纠错(华世王镞集团25篇)"与"合法词误改成另一合法词(肌肤生态臆造句0篇)"彻底分开)

### 便宜预筛(提速)
整名在文本层能逐字命中(≥3字走 trgm 索引,快)= 已是正确名,直接跳过逐字扫。
只有"查无此串"的才跑尺子。第一性原理的自然推论:整名在100%正确的文本层存在=它就是对的。

### 合并动作(变体并入锚点,双向可召回)
1. 锚点实体不存在→建;2. chunk_entities/evidence/图谱节点改指向锚点;
3. 变体名注册成锚点 alias(搜变体也命中锚点);4. 变体条目标 status=merged + canonical_id +
   semantic_meta 落打齐依据;5. 写 kb_entity_merge_log。变体条目保留可回溯。
**检索无需改动**:search_service 已用 kb_entity_aliases 命中,注册别名即通。

### 落地方式
- **增量(新文件自动修正)**:折入 graph 节点(`pipeline_stages/graph.py`),抽完实体后调
  `align_document_entities`,失败不拖垮主流程。已验证:族→镞改对、幂等、误伤已修。
- **存量(分轮回填)**:`批_文本层打齐分轮.py`。字典表加 `align_status` 字段(pending/done),
  每轮取一批 pending 跑尺子、跑完打 done,下轮只跑 pending,滚到清空。天然续跑/幂等。

### 新增数据结构
- `kb_entity_dictionary` 加字段:type_id(类型)、canonical_id(合并指向)、semantic_meta(依据)、
  align_status(打齐状态)
- 新表 `kb_authority_tokens`(全局权威字典,缓存已用权威字)、`kb_semantic_types`(类型本体)

## 三、图片多维分字段方案(第四层,待实施)

### 问题
一张图信息量大,VLM 读出多维信息(图里的字/画面描述/场景/标签/素材类型),但现在**糊在一坨**
存进 fused_text。且 fused_text 混了 round1 的**像素技术噪声**("JPEG 4032x2268px RGB 平均亮度
0.53…"),对 agent 回答零价值,还占召回权重、稀释命中。

### 理念(华哥定)
Agent 每次实时跑 VLM 不现实(7秒)。正解:**VLM/OCR 多维信息分类型拆成结构化文本存进库**,
Agent 命中对应那类文本就秒定位到图,永不实时跑 VLM。等于**用文本给图片建多维可检索索引**。

### 方案:不建新表,字段全现成(kb_page_fusions),问题是没用干净
| 维度 | 归入字段 | 现填充率 |
|---|---|---|
| 图里的字(OCR) | fused_text 主体(可被实体打齐纠错) | — |
| 画面描述(VLM注释) | page_summary / body_json | 99.7% / 72% |
| 标签+素材类型(VLM) | tags_json(命中"会议现场"即定位图) | 74% |
| 标题 | page_title | 50% |
| 像素技术元数据(round1) | attributes_json,**踢出 fused_text 不进召回** | — |

改**第四层融合逻辑**:VLM 三维度(素材类型/画面注释/标签)正确分字段 + fused_text 去噪。
效果:搜"培训现场照片"命中标签、搜图里公司名命中OCR文字 → 秒级定位,永不实时跑VLM。

## 四、第二层降级(纯图无文本层校准,探测中)
只在图里出现、全库文本层从没有过的实体(独一份,无权威可校)→ 需 OCR/VLM 重跑/超高清。
