# 字位权威预计算表(根治 SQL 当处理器)

## 病根(华哥直觉命中)
`_slot_authority` 对每个实体名的每个字位发一条 `text ~ 'left.right'` 全表正则扫(55万行/4GB,358ms/次)。
3.9万实体 × 平均5字 = ~20万次全表扫,同一批语料被反复扫20万遍 → CPU打满、20小时跑不完。
= 把数据库当处理器用了。

## 正解(华哥方案:一次性吃数据→查表)
字级权威本质是统计问题:"'left X right'这个坑,X填什么字最多"。这个统计跑一次就够,
不该每个实体现算。预计算落表,打齐时O(1)查表,精度完全不变(统计的是同一个东西)。

## 表设计
kb_char_slot_authority:
  left_ctx  VARCHAR   -- 左上下文(1或2个汉字)
  right_ctx VARCHAR   -- 右上下文(1或2个汉字)
  mid_char  VARCHAR   -- 中间字
  cnt       INTEGER   -- 该组合出现的文档数(与原COUNT(DISTINCT document)等价)
  主键/唯一: (left_ctx, right_ctx, mid_char)
  查询索引: (left_ctx, right_ctx)

## 构建(一次性,等价复刻原窗口逻辑,保精度)
扫 base_parse 非图片 chunk 的 text,对每个汉字位置 j:
  mid = text[j]
  左窗口 lefts = [左1字, 左2字](连续汉字,碰非汉字停,最多2)
  右窗口 rights = [右1字, 右2字]
  for l in lefts: for r in rights:
    记录 (l, r, mid) 命中的 document_id
  最后 cnt = distinct document 数(与原 COUNT(DISTINCT c.document_id) 等价)
  ※ 注:原SQL用 substring 每chunk取第一个match,预计算用"每次出现"更精确,
     但按document去重后相对排序一致,精度不降(更全)。

实现:流式分批读chunk(每批1万),Python滑窗累加到 dict[(l,r,mid)] = set(doc_ids),
     最后批量 COPY 写表。Mac 96GB内存扛得住(dict约几GB)。

## 改 _slot_authority
从"全表正则扫"改成"查 kb_char_slot_authority":
  SELECT mid_char, cnt FROM kb_char_slot_authority
  WHERE left_ctx=:l AND right_ctx=:r ORDER BY cnt DESC
btree索引 O(log n),<1ms。20万次查询=几秒,不再扫4GB表。

## 增量维护
新文档入库(graph阶段)时,对其 base_parse text 做同样滑窗,UPSERT 增量更新表。
先做全量构建,增量维护后续加(不阻塞当前回填)。

## 精度保证
- 统计对象完全一致(文本层字频),只是现算→预算
- 窗口逻辑1:1复刻(WIN=2,左右各最多2汉字,碰非汉字停)
- document去重与原 COUNT(DISTINCT document) 等价
- 黄金集验证:构建后跑 测_裁定循环,精度必须和现在一致(6/6乱并0)

### CPU 爆满二次根治(降级路径)
现象:火力全开时 CPU 100%,24个并发全表正则扫。
根因:canonicalize_name 就地修正后产生的新坑位不在预计算表→_slot_authority 降级全表扫。
修复:降级路径改"查不到返回空(保守不改)",不再全表扫。只有预计算表异常才兜底全表扫。
精度影响:第一次修正的坑位一定在表里(表从原始名提取),只有多错字名的第2+次修正可能miss,
返回空=不做未验证修正=保守安全。黄金集验证一致。
