# 实体治理实现方案（护肤品行业 / kb_terms + kb_entity_dictionary）

> 执行方案，非咨询报告。每步带 `文件:行号` 与可直接跑的代码/SQL。
> owner_id 默认 4。数据库为 PostgreSQL。

## 0. 现状根因（一句话）

两套并行"实体"体系，治理只做了半套：

- **体系B（词项）kb_terms**：`cognitive_index` 阶段纯正则分词产出，`term_type` 全是 `term`（68 万条无分类），`status` 只有 active/rejected，rejected 2089 条已被搜索召回排除。
- **体系A（实体图谱）kb_entity_dictionary**：`graph` 阶段 LLM 抽取，`category` 已有分类（术语/产品名/人名…共 10 万条），但全是 `candidate`（0 条 confirmed），rejected 1181 条**只在召回被排除、打分阶段仍在污染 7704 个文档**。
- **kb_governance_candidates**：24.2 万条 LLM 已分类候选，confidence=0.70，全部 pending 待审。分类模型其实已经跑过一轮，缺的是审核+入库+去噪。

治理目标：① 规则清噪 ② 消化存量候选 ③ 让 `term_type` 真正分类 ④ 堵住搜索打分漏洞 ⑤ 高频 LLM 精分。

## 1. 实体分类体系（护肤品行业枚举）

统一 12 类。写入 `kb_terms.term_type` 和 `kb_entity_dictionary.category` 都用同一套中文枚举，禁止英文混入（现候选表里有 `product/其它` 等脏值，先归一）。

| 枚举值 | 说明 | 典型 | 检索权重系数 |
|--------|------|------|------|
| `人名` | 自然人 | 讲师、代言人、专家 | 8 |
| `品牌` | 品牌名 | 华世王镞、娇薇诗、蔻诺 | 9 |
| `产品` | 产品/SKU | 面霜、精华液、修护乳 | 9 |
| `成分` | 原料成分 | 烟酰胺、神经酰胺、玻尿酸 | 9 |
| `功效` | 功效声称 | 补水、抗皱、修护屏障 | 6 |
| `机构` | 企业/组织 | 美容院、招商公司、检测机构 | 7 |
| `证书` | 资质证书 | 检测报告、备案凭证 | 8 |
| `规格` | 规格/包装 | 30ml、50g、5片装 | 5 |
| `检验标准` | 标准/法规 | GB/T、化妆品安全技术规范 | 8 |
| `备案批号` | 备案/批准文号 | 国妆特字、粤G妆网备字 | 9 |
| `术语` | 通用业务术语 | 生态门店、引爆新消美 | 3 |
| `噪声` | 版式/OCR碎片 | 主色为、分辨率、Sheet1 | reject（不召回）|

> `噪声` 是治理专用标签：命中即置 `status='rejected'`（term）或 `archived`（entity），彻底退出召回和打分。

## 2. 三层治理策略

### 2.1 规则层（零 LLM 成本，先跑）

正则直接判定，覆盖约 20 万条噪声 + 高置信度分类词。判定优先级：噪声 > 备案批号 > 规格 > 成分/术语。

**噪声识别规则（直接 reject）：**

- 结尾字：`为|约|是|的|了|吗|呢` 结尾（OCR 句片，1.5 万）
- 长度：`char_length(term) >= 20` 或 `< 2`（整句被当实体，4 万）
- 版式模板词：`主色为|图片格式|分辨率|边缘密度|平均亮度|Sheet\d|OCR|RGB|像素|色彩模式|DPI|JPEG|PNG`
- 纯数字 / 纯英文 ≤3 字符

**正则分类规则（高置信度，直接打 term_type）：**

- `备案批号`：`国妆|粤G妆网备|妆网备|国妆特字|卫妆准字` 或 `[A-Z]{1,4}妆.*字`
- `规格`：`^\d+(\.\d+)?(ml|g|mg|片|支|瓶|盒|袋|kg|L)$`（忽略大小写）
- `检验标准`：`GB/?T?\s*\d|化妆品安全技术规范|QB/T`
- `成分`：命中成分词典（见下 SQL，可用 kb_entity_dictionary 里 category=成分 的已分类词反哺）

其余交给 LLM 层。规则层预计直接处理 ~40% 存量。

### 2.2 LLM 层（deepseek-v4-flash 批量分类）

- **只分去噪后高频**（occurrence > 10，约 9265 条），50/批 = **186 批**。这是检索召回主力。
- 中频（3-10 次，4.1 万）等检索日志暴露"搜到没分类"再补，不盲跑。
- 低频（<3 次，42 万）不投 LLM，向量兜底。
- 模型：`deepseek-v4-flash`（models.json `module_routing.knowledge.agent_search_profile` 已配，快且便宜）。

### 2.3 合并层（复用 kb_entity_aliases）

- 同义合并走已有 `governance_service.merge_entities`（`governance_service.py:116`），把源实体名转目标别名、迁移 chunk 关联和证据。
- 别名产出复用 `derived_governance_service._backfill_entity_governance`（`derived_governance_service.py:540`）：括号内容/分隔符/CJK 紧凑变体。
- 分类后新增：**同 term_type 且 normalized 编辑距离近**的词做候选合并（人工确认，不自动落库）。

新增函数签名（贴进 `derived_governance_service.py`，`_backfill_entity_governance` 之后）：

```python
# derived_governance_service.py 新增，约在 615 行之后
_NOISE_SUFFIX = ("为", "约", "是", "的", "了", "吗", "呢")
_NOISE_TEMPLATE_RE = re.compile(
    r"主色为|图片格式|分辨率|边缘密度|平均亮度|Sheet\d|OCR|RGB|像素|色彩模式|DPI|JPEG|PNG"
)
_SPEC_RE = re.compile(r"^\d+(\.\d+)?(ml|g|mg|片|支|瓶|盒|袋|kg|l)$", re.I)
_FILING_RE = re.compile(r"国妆|粤G妆网备|妆网备|国妆特字|卫妆准字|[A-Z]{1,4}妆.{0,4}字")
_STANDARD_RE = re.compile(r"GB/?T?\s*\d|化妆品安全技术规范|QB/T")


def _rule_classify_term(term: str) -> str | None:
    """返回 term_type；'噪声' 表示应 reject；None 表示交给 LLM。"""
    t = str(term or "").strip()
    n = re.sub(r"\s+", "", t)
    if len(n) < 2 or len(n) >= 20:
        return "噪声"
    if n.endswith(_NOISE_SUFFIX) or _NOISE_TEMPLATE_RE.search(n):
        return "噪声"
    if re.fullmatch(r"\d+", n) or (re.fullmatch(r"[A-Za-z]+", n) and len(n) <= 3):
        return "噪声"
    if _FILING_RE.search(n):
        return "备案批号"
    if _SPEC_RE.match(n):
        return "规格"
    if _STANDARD_RE.search(n):
        return "检验标准"
    return None  # 交给 LLM
```

## 4. LLM 分类的具体实现

### 4.1 模型调用范式

复用知识库标准 LLM 范式（见 `fusion_service.py:110`），profile 用 `deepseek-v4-flash`（可直接 override，绕开 stage 配置）：

```python
# derived_governance_service.py 新增函数
from app.gateway.router import gateway_router
from .llm_diagnostics import timed_llm_chat
from .model_routing import resolve_knowledge_profile

async def _governance_llm_classify_terms(
    db, *, owner_id: int, dry_run: bool = True,
    min_occurrence: int = 10, batch_size: int = 50,
    max_terms: int = 500, previews: list | None = None,
) -> int:
    """高频未分类词投 deepseek，产出到 kb_governance_candidates（不直接改 kb_terms）。"""
    # deepseek-v4-flash：models.json module_routing.knowledge.agent_search_profile
    profile_key = resolve_knowledge_profile("entity_governance", override="deepseek-v4-flash")

    # 1. 取去噪后高频未分类词（term_type 仍是 'term'、active、occurrence>阈值）
    rows = (await db.execute(sa_text(
        """
        SELECT t.id, t.term
        FROM kb_terms t
        JOIN (
            SELECT term_id, count(*) AS c FROM kb_term_occurrences
            WHERE owner_id = :oid GROUP BY term_id HAVING count(*) > :minocc
        ) o ON o.term_id = t.id
        WHERE t.owner_id = :oid AND t.status = 'active' AND t.term_type = 'term'
          AND char_length(t.term) BETWEEN 2 AND 19
        ORDER BY o.c DESC
        LIMIT :maxn
        """
    ), {"oid": owner_id, "minocc": min_occurrence, "maxn": max_terms})).all()

    written = 0
    for i in range(0, len(rows), batch_size):
        batch = rows[i:i + batch_size]
        terms = [r[1] for r in batch]
        user_msg = _CLASSIFY_PROMPT_USER.format(
            terms="\n".join(f"{j+1}. {w}" for j, w in enumerate(terms))
        )
        result = await timed_llm_chat(
            logger=logger, stage="entity_governance", profile_key=profile_key,
            messages=[{"role": "system", "content": _CLASSIFY_PROMPT_SYS},
                      {"role": "user", "content": user_msg}],
            chat_func=gateway_router.chat,
        )
        parsed = _parse_classify_json(result.get("content"))  # [{"term":..,"type":..,"confidence":..}]
        for r, item in zip(batch, parsed):
            cat = _canonical_category(item.get("type"))  # 归一到 12 类
            conf = float(item.get("confidence") or 0.6)
            if previews is not None:
                previews.append({"term": r[1], "category": cat, "confidence": conf})
            if not dry_run:
                # 产出候选（走人工确认），不直接改 kb_terms.term_type
                await db.execute(sa_text(
                    """
                    INSERT INTO kb_governance_candidates
                        (owner_id, document_id, entity_name, category, excerpt, confidence, audit_status)
                    VALUES (:oid, 0, :name, :cat, 'term_llm_classify', :conf, 'pending')
                    """
                ), {"oid": owner_id, "name": r[1], "cat": cat, "conf": conf})
                written += 1
    return written
```

### 4.4 候选落库（人工确认后）

LLM 产出全进 `kb_governance_candidates`（pending）。审核台确认后落 `kb_terms.term_type`：

```sql
-- 审核台点"批准"后，把候选的 category 落到对应 kb_terms.term_type
UPDATE kb_terms t
SET term_type = c.category
FROM kb_governance_candidates c
WHERE c.owner_id = t.owner_id
  AND c.entity_name = t.term
  AND c.excerpt = 'term_llm_classify'
  AND c.audit_status = 'approved'
  AND t.term_type = 'term';

-- category='噪声' 的批准候选 → reject 对应 term
UPDATE kb_terms t
SET status = 'rejected'
FROM kb_governance_candidates c
WHERE c.owner_id = t.owner_id AND c.entity_name = t.term
  AND c.excerpt = 'term_llm_classify' AND c.category = '噪声'
  AND c.audit_status = 'approved';
```

> 高置信度（confidence≥0.85 且 type≠噪声）可开"自动批准"快速通道，走 `governance_service.approve_candidate:54` 批量确认，人工只复核 0.6~0.85 的中间带。

## 5. 搜索改造（search_service.py）

### 5.1 P0 — 堵打分漏洞（不依赖分类，立即做）

rejected 实体（1181 条 / 93134 条关联）在召回被排除，但**打分阶段 entity_stats CTE 没过滤**，虚高 7704 个文档。这是当前最大精度漏洞，先补。

改 `search_service.py:1875-1886` 的 `entity_stats` CTE，加实体状态过滤：

```sql
-- search_service.py:1875 entity_stats CTE，改为：
entity_stats AS (
    SELECT
        ce.document_id,
        count(DISTINCT ce.entity_id) AS entity_count,
        avg(COALESCE(ce.confidence, 0.0)) AS entity_confidence,
        count(DISTINCT ge.id) AS graph_edge_count
    FROM kb_chunk_entities ce
    JOIN kb_entity_dictionary ed
        ON ed.id = ce.entity_id AND ed.owner_id = ce.owner_id
       AND ed.status IN ('candidate', 'confirmed')   -- ★新增：排除 rejected/archived
    LEFT JOIN kb_graph_nodes gn ON gn.owner_id = ce.owner_id AND gn.entity_id = ce.entity_id
    LEFT JOIN kb_graph_edges ge ON ge.owner_id = ce.owner_id AND (ge.source_node_id = gn.id OR ge.target_node_id = gn.id)
    JOIN live ON live.document_id = ce.document_id AND live.doc_owner_id = ce.owner_id
    GROUP BY ce.document_id
)
```

效果：93134 条 rejected chunk_entities 退出打分，7704 个文档虚高分立即归位。改一处 SQL，零副作用。

### 5.2 P1 — term_type 参与召回打分

分类落库后，让 term 通道按类型加权。改 `search_service.py:1517-1534` 的 `term_hits` CTE 和 `1615-1626` 的 score 表达式。

**a) term_hits 里带出加权和（`search_service.py:1522-1523` 附近）：**

```sql
-- term_hits CTE 内，新增按 term_type 的加权列
term_hits AS (
    SELECT
        o.document_id,
        min(o.chunk_id) AS chunk_id,
        min(o.page) AS page,
        count(DISTINCT o.term_id) AS term_count,
        sum(COALESCE(o.weight, 1.0)) AS term_weight,
        -- ★新增：按类型加权（高辨识类型权重高，术语降权）
        sum(COALESCE(o.weight, 1.0) * CASE t.term_type
            WHEN '品牌' THEN 2.2 WHEN '产品' THEN 2.2 WHEN '成分' THEN 2.2
            WHEN '备案批号' THEN 2.2 WHEN '人名' THEN 2.0 WHEN '证书' THEN 2.0
            WHEN '检验标准' THEN 2.0 WHEN '功效' THEN 1.5 WHEN '机构' THEN 1.7
            WHEN '规格' THEN 1.2 WHEN '术语' THEN 0.7 ELSE 1.0 END
        ) AS term_typed_weight,
        left(string_agg(DISTINCT CONCAT(t.term, ': ', COALESCE(o.context, '')), E'\n'), 500) AS evidence_text
    FROM kb_term_occurrences o
    JOIN kb_terms t ON t.id = o.term_id AND t.owner_id = o.owner_id
    JOIN live_docs ld ON ld.document_id = o.document_id AND ld.doc_owner_id = o.owner_id
    LEFT JOIN kb_entity_dictionary ed ON false
    LEFT JOIN kb_entity_aliases ea ON false
    LEFT JOIN kb_fact_candidates fc ON false
    LEFT JOIN kb_causal_candidates cc ON false
    WHERE t.status = 'active'
      AND ({match_clause})
    GROUP BY o.document_id
),
```

> 注意 `term_typed_weight` 需在 merged/最终 SELECT 里透传（`term_hits` 后续 LEFT JOIN 到 merged 时补 `COALESCE(th.term_typed_weight, 0)`）。

**b) 最终 score 用类型加权替换裸 term_weight（`search_service.py:1616-1626`）：**

```sql
-- 原：term_count * 4 + LEAST(term_weight, 10) + ...
-- 改为：
    term_count * 3
    + LEAST(term_typed_weight, 16)   -- ★类型加权替代裸权重，上限提高到 16
    + entity_count * 5
    + fact_count * 6
    + causal_count * 4
    + entity_confidence * 2
    + fact_confidence * 2
    + causal_confidence * 2
```

### 5.3 P2 — 别名查询扩展（激活 3031 条 kb_entity_aliases）

现状：别名只在 entity_hits 反查用（`search_service.py:1546`），没做正向扩展。用户查主名不会召回别名文档。

在 `_load_query_tokenizer_terms`（`search_service.py:501`）之后新增一步：命中实体后查 `kb_entity_aliases`，把别名塞进 needles 再传给 `structured_signal_search`。

```sql
-- 查询扩展：query 命中的实体，取其别名补进 needles
SELECT DISTINCT ea.alias
FROM kb_entity_aliases ea
JOIN kb_entity_dictionary ed ON ed.id = ea.entity_id AND ed.owner_id = ea.owner_id
WHERE ea.owner_id = :oid
  AND ed.status IN ('candidate','confirmed')
  AND (ed.name = ANY(:query_entities) OR ea.alias = ANY(:query_entities))
LIMIT 20;
```
