# 05 知识库召回精度提升

## 阶段一:问题 → 方案 → 影响范围(动手前)

### 发现的问题
召回"命中一堆没用的",精度低。对着代码核实(非假审计),三个根因:

1. **重排没开**:`rerank()` 是真函数(model_services.py:99,真打 bge-reranker-v2-m3),
   但 `hybrid_search` 的 `use_rerank` 默认 False(search_service.py:2224)。
   handler 入口 `use_rerank: bool | None = None` 往下传,从不触发重排。
2. **无阈值过滤**:`hybrid_search` 末尾直接 `results[:top_k]`(search_service.py:2365),
   有多少返回多少,低分噪音全带出来。
3. **别名没接**:16消歧子模块产出了实体别名,但 `plan_query` 检索词扩展没用上,同义词命不中。

### 解决方案
1. handler 入口 `use_rerank` 默认改开,让重排真正跑。rerank 失败已有兜底(非致命,warning)。
2. `hybrid_search` 返回前加相似度阈值过滤:低于阈值砍掉,阈值走动态配置(硬编码兜底)。
   - 量纲分开:rerank 分(bge-reranker输出,0~1)设保守默认 0.3,宁可多留不错杀;
   - 没跑 rerank 时不硬砍融合分(量纲不同),只砍明显为 0 的垃圾。
3. `plan_query` 把消歧别名/同义词并进检索词做查询扩展。

### 影响范围
- `modules/knowledge/backend/handlers/search.py`(入口默认值)
- `modules/knowledge/backend/services/search_service.py`(hybrid_search 阈值过滤 + plan_query 别名扩展)
- 只动知识库模块内,不跨模块。rerank 走已有 model_services.rerank,不新增模型调用路径。

### 回退路径
改动前备份两文件;阈值/开关都做成可配,出问题把开关关掉即恢复原行为。

## 阶段二:实际结果与调用方式(完成后)

### 三处改动(全部落地实测)
1. **重排默认开**:
   - handlers/search.py: `SearchRequest.use_rerank` 默认 False→True
   - router.py:313(Agent能力路径,关键): 硬编码 `use_rerank=False` → 从 params 读默认 True
2. **重排分阈值过滤**:search_service.py 加常量 `RERANK_SCORE_THRESHOLD=0.3`。
   hybrid_search 返回前:只在真跑了重排(reranked_ran)时按 rerank_score≥0.3 过滤,
   量纲明确才砍;没跑重排不硬砍(RRF融合分量纲不同);全砍光则退回原序top_k不返空。
3. **别名扩展**(同义词命中):`_load_entity_idf_table` 追加 kb_entity_aliases 查询,
   别名文本作可匹配词但原名映射回规范名。用户用别名提问→命中→检索词用规范名。

### 验证
- 单元实测:纯别名"娇文诗"(词典无此规范名)→ `_match_query_entities` 命中并返回规范名"Jiaowshi / 娇文诗" ✓
- 整链路:call_capability knowledge.search "娇文诗品牌介绍" → 200,返回结果不崩,重排+阈值+别名全跑通 ✓
- 首次检索 8.2s(含 bge-reranker 冷启动 ensure_model),第二次会快

### 调用方式
- HTTP: POST /api/modules/... knowledge search {query, top_k, use_rerank(默认True), embedding_profile}
- 能力: call_capability module=knowledge action=search params={query,top_k}
- 关阈值/重排:传 use_rerank=false 即恢复原行为(回退路径)

### 遗留观察
- bge-reranker 首次冷启动慢(~8s),常驻后快。若嫌慢可配 watchdog 常驻不休眠。
- 阈值 0.3 是保守默认,可按实际召回质量调(改 RERANK_SCORE_THRESHOLD 常量或做成动态配置)。
