# 产品应用底座、统一内容管线与工作区重构方案

> 日期：2026-07-17
> 状态：已审计回填（2026-07-17，数据库实查+代码核实），可进入目标模式
> 审计补丁：§18.3/§14.3 门禁口径、§19.3-C hash 盐方向、§19.4 状态/origin 映射、§16.1-E adapter 归属、§23 术语对照、§24 本次施工边界（越过 LLM/VLM）。审计明细见 `临时文档/08_方案07可落地性审计_20260717.md`
> 目标：一次性交付到用户可用阶段，不长期保留新旧混合前端
> 核心判断：当前根因不是样式不足，而是“模块、软件、原始文件、统一内容、投影文件、窗口”没有使用同一套正式合同

## 0. 最终结论

上一版方案仍然过度保护现有桌面应用注册表，只在其上增加映射、别名和视觉壳。这会继续积累兼容层，不能从根本上解决问题。

本次应直接重构前端产品底座：

```text
当前错误模型

modules/*/manifest.json
        │
        ├── Parser
        ├── Viewer
        ├── Editor
        ├── Generator
        ├── Service
        └── 业务模块
        │
        ▼
全部同步成 Desktop App
        │
        ▼
Launchpad、Dock、文件关联和窗口都直接面对技术模块


目标模型

Products（用户使用的软件）
        │
        ├── 文件
        ├── 办公
        ├── 图片与媒体
        ├── 文本
        ├── AI 助手
        ├── 知识库
        ├── 消息
        ├── 内容工作台
        └── 设置
        │
        ▼
Product Runtime（窗口、工作区、文档会话、命令）
        │
        ▼
Modules（解析器、编辑器、生成器、服务和业务能力）
        │
        ▼
Backend APIs / Capability Bus（保持现有后端能力）
```

用户只接触 Product。Module 是 Product 内部调用的能力，不再天然等于软件。

内容链路以当前已经落地的 ContentPackage 为基础，不再新造一套平行 Document 数据模型：

```text
拖入桌面或上传
      ↓
SourceFile
原始文件、来源、哈希、归档和重新处理依据
      ↓ 自动异步预处理
Canonical ContentPackage
统一 IR、资源、版本、修改和数据库事实源
      ↓ UI Projection
Product Document Session
Office、文本、图片等产品中的工作会话
      ↓ 显式导出/发布
Projected File
DOCX、XLSX、PPTX、PDF、TXT 等物理输出
```

原始文件默认不作为编辑工作副本。产品读取和修改 ContentPackage；只有导出、发布、下载原件或重新预处理时才访问物理文件。

## 1. 为什么当前底座确实有问题

### 1.1 构建脚本把有前端入口的模块直接变成应用组件

`frontend/scripts/scan-modules.js` 扫描 `modules/*/manifest.json`，只要存在 `frontend/<component_key>` 就生成桌面组件入口。

### 1.2 后端把模块 manifest 直接同步为桌面应用

`backend/app/services/app_service.py` 扫描相同 manifest，并直接读取：

- `name`
- `component_key`
- `show_in_launcher`
- `show_on_desktop`
- `supported_formats`
- `editable_formats`

虽然 manifest 已经有 `module_type`、`module_family` 和 `product_status`，桌面应用合同并没有真正使用这些分类。

### 1.3 一个 appKey 承担了四种职责

当前 `appKey` 同时表示：

1. 用户软件。
2. 前端组件。
3. 文件处理器。
4. 后端能力模块。

因此才会出现：

- Excel 编辑器、PDF 查看器、文档查看器分别出现在 Launchpad。
- Parser、开放接口和生成器需要靠 `show_in_launcher=false` 隐藏。
- 文件关联直接打开技术处理器。
- 窗口会话绑定处理器 key，而不是用户软件。
- 修改一个名称或 key 会牵动文件、窗口、Agent 和后端调用。

这不是 CSS 能修复的问题。

## 2. 用户真正需要感知到什么

macOS 的真实感不是圆角、渐变和毛玻璃的总和，而是以下体验始终一致：

1. **软件是稳定的产品对象**：Dock 中的“办公”永远是同一个软件，不会因为打开 PDF 或 Excel 变成另一个应用。
2. **文件是内容对象**：文件有自己的类型图标、名称、修改状态和权限，不与软件图标混用。
3. **命令具有上下文**：打开 Word 时工具栏是文档工具，打开 Excel 时是表格工具，但仍处于同一个办公软件中。
4. **窗口属于软件**：Dock 按软件分组，窗口会话保存产品状态，文档是窗口内部的工作会话。
5. **首次打开不是空壳**：应用没有载入文件时也有首页、最近内容、新建入口和打开入口。
6. **反馈统一**：加载、保存、错误、只读、弹窗和通知使用同一套语言。
7. **视觉层级安静**：不重复显示应用品牌，不用每个模块自己的蓝色、卡片和大标题抢夺注意力。

因此目标不是 1:1 复制 WPS、Microsoft Office 或 Apple 的品牌资产，而是：

```text
使用 WPS/Office 成熟的软件信息架构
+ 使用当前 macOS 桌面壳的视觉语言
+ 使用项目自己的图标和产品身份
```

## 3. 新的六层对象模型

### 3.1 Product：用户软件

```ts
interface ProductManifest {
  productId: string
  displayName: string
  aliases: string[]
  iconSet: ProductIconSet
  entryComponent: string
  workspaceKind: 'document' | 'conversation' | 'catalog' | 'media' | 'settings'
  singleton: boolean
  supportsTabs: boolean
  requiredCapabilities: string[]
  commands: ProductCommand[]
}
```

Product 才允许进入 Launchpad、Dock、Spotlight 应用结果和窗口管理器。

### 3.2 Module：技术与业务能力

```ts
interface ModuleManifestV3 {
  moduleId: string
  moduleType: 'renderer' | 'editor' | 'parser' | 'generator' | 'service' | 'business'
  moduleFamily: string
  productStatus: 'active' | 'internal' | 'deprecated'
  frontendAdapter?: string
  backendCapabilities: string[]
  supportedFormats?: string[]
  editableFormats?: string[]
}
```

Module 不再自动成为 Desktop App。模块可以有前端组件，但该组件默认是可嵌入适配器，不是独立软件。

### 3.3 SourceFile：原始物理来源

```ts
interface SourceFileDescriptor {
  fileId: number
  fileName: string
  extension: string
  mimeType?: string
  sourceHash: string
  sourceState: 'available' | 'recycled' | 'missing' | 'permanently_deleted'
  permissions: {
    read: boolean
    write: boolean
  }
  lock?: {
    locked: boolean
    owner?: string
  }
}
```

SourceFile 负责来源、原件下载、生命周期和重新预处理，不是前端编辑器的直接工作模型。

### 3.4 CanonicalContent：数据库统一内容

```ts
interface CanonicalContentDescriptor {
  packageId: number
  currentVersionId: number
  packageType: 'document' | 'spreadsheet' | 'presentation' | 'pdf' | 'image' | 'text' | 'media' | 'generic'
  sourceFileId?: number
  sourceExtension?: string
  sourceState: string
  status: 'draft' | 'ready' | 'degraded' | 'failed' | 'stale' | 'archived'
  manifest: Record<string, unknown>
  content: CanonicalContentIR
  capabilities: {
    view: boolean
    edit: boolean
    export: boolean
    publish: boolean
  }
}
```

该对象映射现有：

- `framework_content_packages`
- `framework_content_package_versions`
- `framework_resources`
- `framework_resource_refs`

ContentPackageVersion 及其版本化 Resource 引用共同构成产品读取和修改的统一内容事实。`content_json` 保存 Canonical manifest、语义节点和分片引用；大型格式数据按 Profile 拆入 Resource，不能再由各 Viewer 绕过。

### 3.5 ProjectedFile：统一内容的物理投影

```ts
interface ProjectedFileDescriptor {
  fileId: number
  fileRevisionId: number
  artifactId: number
  artifactVersionId: number
  packageId: number
  versionId: number
  format: string
  projectionType: 'primary' | 'published' | 'exported' | 'preview'
  binarySha256: string
  createdAt: string
}
```

ProjectedFile 是某个 ContentPackageVersion 通过 ArtifactVersion 生成的不可变格式化输出。它可以成为桌面可见文件或下载文件，但不能反向成为编辑事实源；外部修改必须新增 `origin=user_import` 的 FileRevision，再显式进入预处理或导入合并。

### 3.6 WindowSession：软件窗口状态

```ts
interface ProductWindowSession {
  windowId: string
  productId: string
  geometry: WindowGeometry
  activeWorkspaceId?: string
  productState: Record<string, unknown>
}
```

Office 的文档标签、最近文件和当前工具模式属于 `productState`，不再把每个 Viewer 当成独立窗口应用。

不搬动后端模块和数据库模型。新增清晰的产品层与工作区运行时：

```text
products/
├── files/
│   ├── product.json
│   └── frontend/
├── office/
│   ├── product.json
│   └── frontend/
│       ├── index.vue
│       ├── home/
│       ├── workspace/
│       └── adapters/
├── media/
├── text/
├── agent/
├── knowledge/
├── messages/
├── content-studio/
└── settings/

frontend/src/workspace-runtime/
├── contracts/
├── product-registry/
├── content-runtime/
├── command-runtime/
├── session-migrations/
└── ui/

frontend/src/platform-sdk/
├── api/
├── auth/
├── files/
├── content/
├── products/
├── modules/
├── events/
└── generated-contracts/

├── excel-engine/
├── doc-viewer/
├── ppt-viewer/
├── pdf-viewer/
├── docs-open/

```

职责：

- `products/`：面向用户的软件和产品编排。
- `workspace-runtime/`：所有产品共享的前端基础合同。
- `modules/`：可被产品组合的业务与技术能力。
- `desktop/`：只负责操作系统外壳，不再理解具体处理器。

### 4.1 构建与注册调整

`scan-modules.js` 拆成两个扫描结果：

```text
scan-products  → Product Component Map
scan-modules   → Capability/Adapter Component Map
```

新增：

- `/api/desktop/products`：返回用户可见产品。
- `/api/registry/modules`：继续返回内部模块能力。

旧 `/api/desktop/apps` 暂时作为兼容接口，只服务旧客户端和迁移测试，完成切换后不再被新桌面调用。

### 4.2 Platform SDK 收口

当前多个模块复制各自的 `runtime/index.ts`，重复维护：

- API base URL。
- token 和 401 处理。
- platform bridge。
- 文件列表、预览和下载。
- 模块调用与事件。

目标：

- `frontend/src/platform-sdk` 成为产品和模块的唯一平台客户端。
- Pydantic/OpenAPI 合同生成 TypeScript 类型，减少手写字段漂移。
- Product 直接依赖 Platform SDK。
- Module frontend 依赖同一个 SDK；沙箱只注入 base URL、auth 和 mock bridge。
- 旧 `modules/*/runtime/index.ts` 先改成薄 re-export，迁移结束后删除复制实现。
- `check-runtime-drift` 最终应不再维护大量“已知变体”白名单。

这层只统一平台调用，不把模块业务 API 强行合并成一个客户端文件。

## 6. Office 产品必须交付到什么程度

### 6.1 未打开文件时

用户点击 Dock 或 Launchpad 的“办公”，进入可用首页：

```text
办公
├── 最近
├── 我的文件
├── 与我共享
├── 打开文件
└── 新建
    ├── Word 文档
    ├── Excel 表格
    ├── PPT 演示
    └── PDF/转换入口
```

首页不是说明页，也不是空白占位，而是软件真正的启动工作台。

### 6.2 打开文件时

```text
OfficeWindow
├── 文档标签栏
│   ├── 方案.docx
│   ├── 预算.xlsx
│   └── 汇报.pptx
├── 上下文工具栏
├── 当前文档适配器
└── 状态栏
```

- 双击文件会打开或激活“办公”。
- 默认在现有 Office 窗口增加文档标签。
- 支持“在新窗口打开”，但不是默认行为。
- Dock 始终只显示一个“办公”产品图标。
- 标题栏显示办公产品图标，标签显示具体文件类型图标。

### 6.3 新建与保存

- Word、Excel、PPT 新建动作先创建无 SourceFile 的 ContentPackage 和首个版本。
- 需要出现在文件系统时，通过 publish 创建或更新 Projected File，并将其关联到 ContentPackage。
- 普通“保存”创建新的 ContentPackageVersion，不直接写原始物理文件。
- “导出/发布/下载”调用 Projection/Export Adapter 生成 DOCX、XLSX、PPTX、PDF 等物理输出。
- “下载原件”是独立命令，只读取 SourceFile。
- 不支持编辑的格式明确进入只读，不伪造保存能力。

### 6.4 Office Handler 合同

```ts
interface ContentHandler {
  handlerId: string
  productId: 'office'
  packageTypes: string[]
  canCreate: boolean
  resolveMode(context: OpenCanonicalContentContext): Promise<'view' | 'edit'>
  loadAdapter(): Promise<Component>
  create?(request: CreateContentRequest): Promise<CanonicalContentDescriptor>
}
```

现有模块通过 Adapter 实现该合同。Adapter 的输入是 CanonicalContentIR，不允许再次读取原始文件并自行解析。

## 7. 统一内容打开合同

```ts
interface OpenContentIntent {
  intent: 'content.open/v4'
  sourceFileId?: number
  packageId?: number
  preferredProductId?: string
  requestedMode?: 'view' | 'edit'
  openTarget?: 'existing-window' | 'new-window'
  location?: {
    page?: number
    sheet?: string
    slide?: number
  }
}

interface ResolvedContentOpen {
  productId: string
  handlerId: string
  source?: SourceFileDescriptor
  content: CanonicalContentDescriptor
  grantedMode: 'view' | 'edit'
  readonlyReason?: 'permission' | 'legacy-format' | 'locked' | 'unsupported-write' | 'processing' | 'source-unavailable'

```

只有 `ContentRuntime` 可以完成 SourceFile 到 ContentPackage 的解析、权限和锁校验、产品与 Handler 选择。桌面、Finder、Agent、最近文件和业务模块全部发送相同 Intent，禁止各自维护后缀映射或直接调用 Parser。

## 8. 统一内容管线收口

### 8.1 当前已落地能力

以下内容不是重新设计，代码中已经存在，应作为唯一底座继续建设：

| 能力 | 当前实现 |
|---|---|
| 上传后自动通知 | `file.uploaded` 事件 |
| 统一内容包 | `framework_content_packages` |
| 内容版本 | `framework_content_package_versions` |
| 资源统一存储 | `framework_resources`、`framework_resource_refs` |
| 自动解析 | `ContentPipelineService` |
| 内容修改 | block update、append、replace、restore |
| 投影与导出 | `ContentExportService`、`office-gen` adapter |
| 生命周期 | source available/recycled/missing/permanently deleted |
| 知识库关联 | `KbDocument.content_package_id` |
| 下载优先投影 | ContentPackage compile → Excel fallback → original fallback |

这证明“原始文件在底层，统一数据库内容在中层”的主体已经落地。

但当前 `ContentPipelineService.SUPPORTED_EXTENSIONS` 仍是白名单。部分视频、音频和未知二进制会直接返回 `skipped`，因此“所有文件都有统一数据库表示”尚未完全成立。

### 8.2 当前尚未统一的旁路

| 旁路 | 当前行为 | 目标 |
|---|---|---|
| Word/PPT Viewer | 打开时再次调用 Parser 读取原始文件 | 读取 ContentPackageVersion |
| PDF/Image Viewer | 直接下载原始或标准化二进制 | 读取内容包投影与资源，原件只作必要渲染源 |
| Text Editor | 下载并直接保存物理文本 | 修改 ContentPackageVersion，显式保存/发布时投影 |
| Excel Engine | 重新解析物理 XLSX/CSV，维护独立状态 | 使用 Spreadsheet IR Adapter 与统一版本 |
| Knowledge | 优先 ContentPackage，但仍保留自己的 DocumentIr 和解析回退 | 只消费 CanonicalContentIR |
| Parser Map | Content 与 Knowledge 各自维护一份 | 只保留 Content Pipeline 一份 |
| IR Schema | ContentPackageIR、DocumentIR、Knowledge DocumentIr 并存 | 合并为一个版本化 CanonicalContentIR |
| Upload Event | 同步顺序等待 Content 与 Knowledge Handler | 持久化入队后异步执行有序管线 |
| 未知/媒体格式 | 不在白名单时直接 skipped | 创建 generic/media ContentPackage，至少保存元数据、资源和处理状态 |

### 8.3 唯一预处理编排

```text
file.uploaded
   ↓ 同事务持久化 FileRevision / IngestionRun / OutboxEvent
Ingestion Orchestrator
   ├── Stage 1 package_ensure
   ├── Stage 2 canonical_parse
   ├── Stage 3 resource_extract
   ├── Stage 4 derivative_build
   ├── Stage 5 knowledge_register
   └── Knowledge 子 DAG
```

约束：

- 上传登记后必须立即创建或绑定 ContentPackage；所有文件都有统一数据库身份。
- Content Pipeline 必须先完成，Knowledge 只能消费其输出。
- Knowledge 不再重新调用格式 Parser。
- 各阶段幂等，可按 fileId/packageId 重试。
- 上传接口返回 `file_id + ingestion_id/status`，前端显示处理中状态。
- 失败不能让文件消失；允许重试、下载原件或查看降级内容。
- 相同 source hash 不重复解析，复用已有 ContentPackage 或版本。
- 无深度 Parser 的格式进入 `metadata_only`/`generic` 可用状态，不以 skipped 结束整个统一链路。

当前 Event Bus 已有持久化、重试和事件日志，但普通 `emit_module_event` 会同步执行 Handler。上传路径应改用持久化入队与后台 Dispatcher，不再用“await 全部处理器”模拟异步。

### 8.4 CanonicalContentIR 唯一合同

以现有 ContentPackage 存储结构为事实源，将以下模型合并：

- `backend/app/schemas/content_package.py::ContentPackageIR`
- `backend/app/schemas/document_ir.py::DocumentIR`
- `modules/knowledge/backend/ir_models.py::DocumentIr`

目标：

```text
backend/app/contracts/canonical_content_ir.py
├── CanonicalContentIR
├── ContentManifest
├── ContentBlock
├── ResourceRef
├── PatchIR
└── ProjectionIR
```

- Parser 输出先经过 Normalizer，再写入 CanonicalContentIR。
- Office、Agent、Knowledge、Viewer、Editor 只引用该合同。
- 格式特有信息存放在稳定的 `data/style/metadata/source_ref` 中。
- `ir_version` 明确版本，旧版本通过迁移器读取。
- 不再通过中英文 block type 互转维持第二套知识库模型。
- Office 表格、样式、合并单元格、分页、幻灯片坐标和资源引用必须有往返保真测试；未达到保真阈值的格式先只读，不能切换到 IR 编辑。

### 8.5 读取、编辑与投影合同

```text
读取：Product → ContentRuntime → ContentPackageVersion
编辑：Product → Patch/Domain Command → 新 ContentPackageVersion
保存：保存当前数据库版本，不碰 SourceFile
发布：ContentPackageVersion → Projection Adapter → Projected File
原件：显式“下载原件”或“重新预处理”才访问 SourceFile
```

对于 Excel 这类高结构内容，可以保留专用编辑引擎，但引擎状态必须可以从 CanonicalContentIR 初始化并提交回统一版本，不能成为第二个长期事实源。

### 8.6 前端处理中状态

文件上传或首次打开时，产品根据统一状态显示：

- 上传完成，正在预处理。
- 可预览，深度处理仍在进行。
- 已就绪。
- 降级可用。
- 处理失败，可重试或下载原件。
- 原始来源不可用，但数据库内容仍可读。

前端不得因为 ContentPackage 尚未 ready 就重新读取原始文件绕过管线。

## 9. 产品工作区底座

不做一个万能大组件，而是提供共享原语和四种稳定布局：

```text
Workspace Runtime
├── DocumentWorkspace    办公、文本
├── ConversationWorkspace AI、消息
├── CatalogWorkspace     知识库、内容工作台、设置
└── MediaWorkspace       图片与媒体
```

共享原语：

- `ProductChrome`
- `WorkspaceToolbar`
- `WorkspaceSidebar`
- `DocumentTabs`
- `WorkspaceState`
- `WorkspaceDialog`
- `WorkspacePopover`
- `WorkspaceToast`
- `CommandBar`

四种布局共享字体、间距、分隔线、控件、弹层和状态反馈，但允许业务主体保持差异。

### 9.1 用户视觉合同

- 窗口内不重复显示应用品牌大标题。
- 工具栏是上下文命令，不是第二个导航栏。
- 同一产品只允许一个主强调色。
- 默认正文 `13px`，辅助文字 `11/12px`，工作区标题 `16/18px`。
- 工具栏 `40px`，普通控件 `32px`，默认侧栏 `240px`。
- 页面章节不使用整页悬浮大卡片。
- Dialog、Popover 和 Toast 必须属于当前产品窗口层级。
- 空态必须提供下一步操作，而不是只显示“暂无数据”。

## 10. macOS 图标的正确分层

Product Icon   软件身份：办公、文件、AI、内容工作台
Document Icon  内容类型：Word、Excel、PPT、PDF、图片、文本
Command Icon   功能动作：保存、搜索、缩放、导出

## 11. 右键菜单与 Dock 细节

这些属于第一层 UI 的收尾，纳入同一次交付：

### 11.1 右键菜单

- 同一时刻只有一个主高亮。
- 子菜单激活时父项只显示弱路径状态。
- 保留父子菜单之间的指针容错走廊。
- 键盘焦点和指针高亮使用同一状态机。

- 只有直接命中的图标发生变化。
- 删除邻近图标联动放大。

## 12. 一次性交付策略

### 12.1 用户只经历一次切换

不是每完成一个模块就上线一部分。工程内部在隐藏运行时完成七个阶段：

```text
阶段 A：CanonicalContentIR 合并与预处理编排收口
   ↓
阶段 B：Viewer/Editor/Knowledge 旁路迁回 ContentPackage
   ↓
阶段 C：Product/Module/Content/Window 合同与 Product Registry
   ↓
阶段 D：Workspace Runtime + 完整 Office 产品
   ↓
阶段 E：其余产品编排和工作区迁移
   ↓
阶段 F：图标、命名、右键和 Dock 收尾
   ↓
阶段 G：全量回归、数据/会话迁移、一次性替换 /desktop
```

开发期间可以使用隐藏入口或本地 Feature Flag 验证新运行时，但正式 `/desktop` 在完成前继续使用当前稳定版本。

### 12.2 切换条件

只有以下条件全部通过才替换 `/desktop`：

1. 上传自动进入异步预处理，ContentPackage 成为唯一可编辑事实源。
2. Viewer、Editor 和 Knowledge 不再自行重新解析原始文件。
3. “办公”首页、新建、最近、打开和多文档标签可用。
4. Word、Excel、PPT、PDF 真实文件完整打开。
5. 查看、编辑、只读、保存、导出和权限语义正确。
6. 所有目标 Product 可以从 Launchpad 和 Spotlight 启动。
7. 旧内容数据、旧会话、旧 appKey 和旧 OpenFile payload 完成一次性迁移。
8. 主要业务应用完成工作区视觉迁移。
9. 生产构建、自动化和三档视口验收通过。

切换后：

- 新桌面只调用 Product Registry。
- 删除前端旧的“模块即应用”主路径。
- 只保留明确的旧会话与外部调用迁移器。
- 不长期维护两套文件关联和窗口运行时。

## 13. 兼容策略不是长期补丁

兼容只允许存在于入口迁移层：

```text
Legacy appKey / payload / session
              │
              ▼
       One-way Migration
              │
              ▼
ProductId + Content Intent + Product Session
```

- 旧数据读取后立即转换成新结构。
- 新保存的数据只使用新合同。
- 新业务代码不能继续调用旧文件关联。
- 迁移器有删除条件和覆盖测试。

## 14. 验证矩阵

### 14.1 架构合同

- Module 不会因为存在 frontend 目录而自动进入 Launchpad。
- Product 才能创建顶层桌面窗口。
- ContentRuntime 是 SourceFile、ContentPackage、Product 和 Handler 解析的唯一事实源。
- 每个上传文件都存在 ContentPackage，未知格式至少是 generic/metadata-only。
- CanonicalContentIR 是唯一结构化 IR。
- Viewer、Editor 和 Knowledge 不直接调用原始文件 Parser。
- WindowManager 只识别 productId。
- Handler 只能在 Product Workspace 内加载。

### 14.2 Office 用户流

1. 拖入 Word、Excel、PPT、PDF 后立即出现“预处理中”，上传响应不等待深度解析。
2. 预处理完成后 Office 读取 ContentPackage，而不是重新解析原件。
3. 启动办公首页。
4. 新建 Word、Excel、PPT。
5. 打开最近文件。
6. 双击 `.docx/.xlsx/.pptx/.pdf`。
7. 多文档标签切换、关闭和恢复。
8. 查看与编辑权限。
9. 保存产生新 ContentPackageVersion。
10. 另存为、下载和导出由最新版本投影生成。
11. 下载原件明确返回 SourceFile。
12. 旧格式兼容只读。
13. 会话刷新后恢复窗口和标签。

### 14.3 内容管线

- 每个**新上传**文件都有唯一 ContentPackage，包括未知、音频和视频格式（历史散落文件按后台批处理回填，不纳入首次切换硬门禁——见 §18.3 门禁 1 范围口径）。
- 同一 source hash 重复上传不会重复执行昂贵解析。
- Content → Resource → Derivative → Knowledge 阶段顺序正确且可重试。
- Ingestion 处理状态、Package 可读状态、预览状态和 Resolver 的 source-unavailable 结论可分别查询，不混用一套枚举。
- Parser 输出、数据库版本和产品读取内容一致。
- Word、Excel、PPT、PDF 完成 parse → edit → version → export 往返测试。
- 原始文件回收或缺失后，允许读取数据库内容，但禁止依赖原件的重新解析与覆盖发布。
- 并发编辑使用 expected version 或等价乐观锁，冲突不会静默覆盖。

### 14.4 其他产品

- 图片与媒体：打开图片、视频和分析工具。
- 文本：打开文本和 Markdown，按权限编辑。
- AI 助手：会话、工作流和工具调用。
- 知识库：文档、检索、治理和图谱。
- 消息：联系人、会话和通知。
- 内容工作台：抖音内容与公众号写作模块切换。
- 设置：模型路由和系统能力配置。

### 14.5 视觉与交互

- `1440x900`、`1280x800`、`1024x768`。
- Product 首页、工作区、空态、加载、错误、Dialog 和 Toast。
- Dock 默认与直接悬停。
- 右键父菜单与子菜单。
- Product Icon 与 Document Icon 全尺寸矩阵。

## 16. 后端同步改造边界

前端是主要工作量，但后端必须同步修改平台合同。这里不是重写业务后端，而是让后端停止把 Module 直接当成 Product，并让权限、锁和文件模式由后端提供真实结论。

### 16.1 后端必须修改

#### A. Product Catalog

- 新增 Product manifest schema 和扫描服务。
- 新增 `/api/desktop/products`。
- 返回产品名称、权限、入口、命令和所需模块，不再返回技术 Handler 作为用户应用。
- 现有 `/api/desktop/apps` 保留为迁移接口，新桌面不再调用。

主要落点：

- `backend/app/services/app_service.py`
- `backend/app/routers/desktop.py`
- `backend/app/schemas/app.py`
- 新增 `backend/app/services/product_service.py`
- 新增 `backend/app/schemas/product.py`

#### B. Module Registry V3

- 正式使用现有 `module_type`、`module_family`、`product_status`。
- 模块目录返回 renderer/editor/parser/generator/service/business 分类。
- 模块是否有前端组件不再决定它是否为桌面产品。
- 保留现有 capability bus 和模块调用键。

#### C. Content Open Resolver

新增唯一的后端解析入口，例如：

```http
POST /api/desktop/content-open/resolve
```

请求：

```json
{
  "file_id": 123,
  "requested_mode": "edit",
  "preferred_product_id": "office"
}
```

响应：

```json
{
  "source": {
    "file_id": 123,
    "file_name": "预算.xlsx",
    "extension": "xlsx",
    "mime_type": "application/vnd.openxmlformats-officedocument.spreadsheetml.sheet",
    "source_hash": "...",
    "source_state": "available",
    "permissions": { "read": true, "write": true },
    "lock": { "locked": false }
  },
  "content": {
    "package_id": 456,
    "current_version_id": 789,
    "package_type": "spreadsheet",
    "status": "ready"
  },
  "product_id": "office",
  "handler_id": "excel-engine",
  "granted_mode": "edit",
  "readonly_reason": null
}
```

该服务统一处理：

- 文件是否存在和是否删除。
- 对应 ContentPackage 是否 ready、degraded、failed 或 stale。
- 当前用户读写权限。
- 文件锁与版本信息。
- 旧格式只读规则。
- Product 与 Handler 选择。
- requestedMode 到 grantedMode 的降级。

前端不再自行推断写权限或读取原始文件，`docs-open` 也不再维护第二套后缀到 Viewer 的事实源。

#### D. Ingestion Orchestrator

- 上传在同一业务事务内持久化 FileRevision、IngestionRun 和 `file.uploaded` OutboxEvent 并快速返回，不同步等待完整解析。
- Content Pipeline 先生成 ContentPackage，Knowledge 和衍生处理再消费其版本。
- 统一阶段状态、重试、幂等和语义失败。
- 复用现有 Event Log 与任务系统，并按第 19 节增加幂等、事件投递和 Ingestion 账本字段。
- 提供统一 ingestion status 给桌面图标、Office 和知识库使用。

#### E. Canonical Content Runtime

- 将 ContentPackageIR、平台 DocumentIR 和 Knowledge DocumentIr 迁移到“统一信封 + 语义节点 + 格式 Profile + Resource 分片”的 CanonicalContentIRV1，不再直接压扁合并为通用 Block 树。
- Parser 输出通过同一个 Normalizer 写入 ContentPackageVersion。
- Knowledge 删除独立 Parser Map 和原始文件回退。
- Word/PPT/PDF/Text/Image Viewer 改为读取统一内容和资源。
- Excel 保留计算与表格引擎，但输入输出边界改为 CanonicalContentIR/Version。
- **投影/导出适配器归属澄清**：适配器实体在框架侧 `backend/app/services/content/adapter.py::call_export_adapter`，它反向 `call_capability_as_system("office-gen", fmt)` 调用 office-gen；office-gen 是被调用的 capability 提供方，模块内部没有 adapter 文件。凡本方案提到"office-gen adapter / Projection Adapter / Export Adapter"，改动落点都在框架 content 层，不是去 office-gen 模块里找 adapter。
- 所有编辑先产生新版本，物理文件只在显式 Save、export 或 publish 时作为 FileRevision/ArtifactVersion 投影生成，不覆盖原始 Revision。

#### F. Desktop State 版本

- `framework_desktop_states.state_json` 继续使用现有 JSON 字段，不需要数据库迁移。
- 状态版本升级，允许保存 `ProductWindowSession` 与产品内部文档标签。
- 旧状态由单向迁移器转换，后端只需正确保存和返回新 JSON。

#### G. Office 首页所需聚合能力

优先复用现有文件和 Office 能力；缺少时只补薄接口：

- 最近文件。
- 可创建的文档类型。
- 新建文档返回 `package_id/version_id` 与 `file_id=null`；首次显式保存再创建真实 File 与 primary projection。
- 文件锁和保存版本。

这些接口只做聚合和权限检查，不重新实现解析、编辑或生成逻辑。

### 16.2 后端不需要重写

- 用户、角色和认证。
- 文件、文件夹和存储模型。
- Word、Excel、PPT、PDF Parser 的格式解析核心。
- Excel 的公式、单元格、样式和操作引擎；仅重构其输入输出边界。
- PDF OCR、搜索和预览。
- `office-gen` 创建、转换和导出。
- AI、知识库、消息和内容业务 API。
- Capability Bus、任务系统和模块内部服务。
- ContentPackage、Version、Resource 和 ResourceRef 现有模型。

### 16.3 数据库原则

本次优先复用现有数据库表，但不再把“零数据库迁移”作为硬目标：

- Product 定义来自 `products/*/product.json`。
- 产品权限由 Product manifest 与所需 Module 权限组合得出。
- 产品窗口状态继续存现有 desktop state JSON。
- 旧 App 表中的模块记录暂不删除，只停止作为新桌面产品事实源。
- CanonicalContentIR 继续存储在 ContentPackageVersion，不另建平行文档表。
- 现有内容包和知识文档需要数据回填、IR 版本迁移和一致性审计。

数据库修改按第 19 节冻结合同执行：复用现有 Event Log、SystemTaskQueue、Artifact 与 ArtifactVersion，新增它们无法可靠表达的 SourceFileRevision、IngestionRun、EventDelivery、编辑租约、Product 覆盖配置和迁移账本。不能为了避免迁移而保留不可靠的同步事件路径，也不再新建与现有 Artifact、任务队列平行的第二套实现。

### 16.4 后端验收

- `/api/desktop/products` 不包含 Parser、Viewer、Editor 和 Service 技术项。
- `/api/registry/modules` 仍完整暴露内部模块能力。
- Content Resolver 对 SourceFile、ContentPackage、权限、锁、状态和处理器选择返回唯一结论。
- 前端不再维护第二套 grantedMode 或后缀关联。
- 上传响应不等待完整预处理，但后台管线自动、可观测、可重试。
- Knowledge、Office 和 Viewer 读取同一个 CanonicalContentIR 版本。
- 编辑只产生 ContentPackageVersion，导出/发布才生成物理文件。
- 旧 `/api/desktop/apps`、旧模块调用和现有业务 API 回归通过。
- 新旧内容数据、桌面状态和调用合同完成可验证迁移。

## 18. 审计结论与目标模式前置门禁

### 18.1 审计结论

五路只读审计分别检查了内容事实源、产品运行时、后端任务一致性、迁移发布和 macOS 交互。结论一致：

```text
方向正确，蓝图基本完整；但当前仍不是可以直接一次性改代码的最终施工合同。
```

当前不建议直接开启“实现全部重构”的目标模式。原因不是视觉方案不够，而是以下事实还没有在代码和契约层真正成立：

- 所有上传文件都能创建 ContentPackage，包括未知格式、音频、视频和仅元数据文件。
- 上传后的预处理确实是持久化、可重试、幂等的异步管线，而不是请求内顺序等待 Handler。
- Office、Viewer、Editor、Knowledge 和 Excel 不再各自维护另一套内容事实源。
- Product、文件后缀、Handler、权限和旧 App 之间有唯一的服务端解析关系。
- 旧窗口、旧 appKey、旧 payload 和旧内容数据可以迁移、对账、失败重跑并恢复。

“用户一次切换 `/desktop`”仍然保留，但工程内部必须先完成可逆的迁移和影子验证，不能把一次用户切换理解成单向、不可回退的发布。

### 18.2 P0：进入目标模式前必须冻结的合同

#### A. Canonical Content 合同

补充并冻结 `CanonicalContentIRV1` 的字段映射表：

- ContentPackageIR、平台 DocumentIR、Knowledge DocumentIr 的字段、资源、块 ID、元数据和不可逆信息如何映射。
- block/resource ID 的稳定生成规则、内容哈希和 IR 版本兼容规则。
- `ContentPackage -> ContentPackageVersion -> Resource/ResourceRef -> ProjectedFile` 的数据库不变量。
- 每个 SourceFile 至多一个 active ContentPackage；每个 Package 必须有合法 current version；每个 ProjectedFile 必须绑定 package/version/artifact。
- Viewer、Editor、Knowledge、Excel Handler 的输入只能是 package/version 或明确的资源投影；增加架构测试禁止重新读取原始文件 Parser。

物理文件仍是原始来源和发布投影，不是编辑事实源。无法完整编辑的格式必须明确降级为 `read_only`，不能用通用 IR 假设覆盖全部 Office 保真能力。Excel 先以真实样本验证公式、样式、合并、图表、媒体和打印布局；未达到阈值前只读。

#### B. Ingestion 与任务一致性合同

统一唯一枚举 `IngestionStatus` 和状态机，禁止文档与代码并存多套状态：

```text
created -> parsing -> resources -> derivatives -> knowledge -> ready
                         |             |             |
                       degraded      retryable      failed
```

合同至少包含：`ingestion_id`、`source_file_id`、`source_version/hash`、`stage_key`、`idempotency_key`、尝试次数、租约、超时、取消、错误码、重试时间和死信状态。上传业务事务必须和 outbox/event 记录绑定；Handler 按阶段独立记录成功状态，重复投递不得重复副作用，支持人工重放和恢复。

未知格式、音频、视频和媒体文件至少创建 metadata-only ContentPackage，不再以 `skipped` 作为无包终态。前端只依据统一 status/resolver 响应展示“处理中、可读、降级或失败”。

#### C. Product、文件关联与会话合同

冻结 `ProductManifestV1`：

```text
product_id, version, display_name, route, icon, enabled,
permissions, required_capabilities, supported_content_types,
file_associations, create_document_types, window_policy,
deep_links, activation_policy
```

Product Catalog 是用户产品事实源，旧 App 表仅作为兼容配置覆盖层；服务端必须定义 Product 权限、Module capability、启用状态、排序和配置覆盖的合并算法，并对重复 ID、缺失组件、前后端版本漂移、私有模块和扫描失败给出构建失败或明确降级策略。旧 `scan-modules` 不得把私有/能力模块丢失，也不得继续无条件把所有模块同步为桌面 App。

冻结 `ProductWindowSessionV2` 与 `DocumentSessionV1`：

```text
ProductWindowSession: window_id, product_id, geometry, z_order,
  minimized, maximized, active, payload, tabs, schema_version
DocumentSession: tab_id, package_id, version_id, mode, dirty,
  lock_token, expected_version_id, save_state, conflict_state,
  restore_state, undo_redo_boundary
```

必须定义旧 Viewer 窗口到 Office 标签的合并规则、恢复失败降级页、窗口状态 CAS、关闭未保存处理和多窗口/多标签切换规则。`OpenContentIntent` 至少携带 `request_id/idempotency_key`、`expected_version_id`、package/version hash、lock token、resolver contract version 和权限判定结果。

#### D. Office 草稿、保存和发布合同

统一语义：新建立刻创建无 SourceFile 的 `draft ContentPackage + ContentPackageVersion + DocumentSession`，返回 `package_id/version_id`，`file_id=null`。自动保存只新增 ContentPackageVersion；用户首次显式保存时创建桌面可见 File、Artifact 和 primary projection revision，但不得把生成文件反向绑定为原始 `source_file_id`。另存为克隆为新的 Package 并创建新的 primary projection；发布或导出创建新的 ProjectedFile。不得一处定义“新建无 file_id”，另一处又要求新建立即返回真实 file_id。

Office Adapter 必须补齐：

```text
open -> hydrate -> edit -> dirty -> save -> export/publish -> dispose
```

并明确权限复查、版本乐观锁、409 冲突、锁续租/过期、失败重试、关闭前保存和只读降级。Office 首页的 recent/open/create/save 接口也必须具备权限过滤和幂等合同。

#### E. 迁移、切换、回滚与灾备合同

把当前单向迁移细化为可重跑批次：

```text
预检 -> checkpoint 回填 -> 新旧对账 -> 影子读 -> 小范围验证
     -> 冻结窗口状态 -> 切换 /desktop -> 观察期 -> 完成或回滚
```

每批必须有 migration_version、checkpoint、失败隔离、重跑规则、差异报告和回滚阈值。服务端保留 feature flag 和旧 runtime 恢复入口；回滚时明确新产生的版本、任务、会话和文件如何处理。至少完成数据库、原始文件、ContentPackage、Resource、ProjectedFile 的恢复演练。

### 18.3 一次性交付的硬验收门禁

以下任一项未通过，都不能宣称进入“用户可用的一次性切换”：

1. 内容门禁：**新上传文件**与**知识库引用文件**均有 Package；Package/version/resource/projected lineage 完整；所有编辑旁路测试为零。
   - 范围口径（重要，防止误把全量历史文件当门禁分母）：数据库实测存活文件约 2.69 万，有 Package 的仅约 0.9%；知识库文档 1.57 万，无 `content_package_id` 的约 98.9%。这些历史散落文件（资源/派生物/早期上传）**不纳入首次切换的硬门禁**。
   - 首次切换只要求：①切换后所有新上传文件走异步管线必得 Package；②知识库当前引用的文件按批回填 Package（可 metadata-only）。历史散落文件降级为后台批处理，不阻断切换，也不触发昂贵重解析。
2. 任务门禁：重复上传、重复事件、Worker 重启、超时、失败重试、死信重放均不产生重复副作用。
3. 产品门禁：Product Catalog、后缀关联、Handler、权限、旧 App 兼容映射均只有一个解析结论。
4. 会话门禁：旧窗口和 Office 多标签可恢复；保存、冲突、关闭未保存和状态 CAS 有自动化测试。
5. 格式门禁：doc/docx/xls/xlsx/ppt/pptx/pdf/csv/txt/md/image/audio/video/未知二进制均有真实夹具；编辑能力和只读边界逐格式声明。
6. 发布门禁：迁移可重跑、可对账、可回滚，并完成数据库与文件存储恢复演练。
7. 体验门禁：桌面、Dock、菜单、右键子菜单、应用内窗口通过键盘/鼠标/触控、三档视口、减少动效和无障碍验收。

## 19. 冻结施工合同

本节是四路字段级深审后的唯一执行版本。第 18 节负责说明门禁，本节负责消除开放选项；后续实现若与本节冲突，以本节为准。除真实代码审计证明合同无法实现外，施工阶段不得临时再造平行事实源。

### 19.1 不再讨论的架构决定

```text
SourceFileRevision：不可变原始输入和证据
ContentPackage：逻辑内容身份
ContentPackageVersion：唯一可编辑事实源，不可变版本
Resource：内容寻址的不可变二进制资源
Artifact + ArtifactVersion + FileRevision：某内容版本的 ProjectedFile
Knowledge / Excel Runtime / Viewer Cache：可重建派生层
Product：用户可见软件
Module：Product 使用的能力
```

- 所有内容型 Product 和 DocumentSession 只通过 `package_id + version_id` 操作内容。
- File 是用户看到的文件身份；FileRevision 是某一时刻的真实字节。上传、外部替换和投影都只能新增 Revision，不再覆盖历史字节。
- 导入文件的 Package 固定绑定导入时的 SourceFileRevision。编辑保存生成 projection revision，不改变原始证据。
- 由 Projection 生成的 Revision 标记 `origin=projection`，不得再次触发新的 Package，防止导出后循环预处理。
- Module 永不因存在前端入口而自动成为 Product。私有 Module 默认只进 Module Registry，需要 UI 时由 `private-product-host` 承载扩展。
- Office 是一个 Product；Word、Excel、PPT、PDF、文本是内部 Adapter 和文档 Profile，不是桌面软件。
- 旧 Viewer 窗口逐窗口迁移为 Office 窗口/标签，不自动合并窗口，避免丢失位置、尺寸和上下文；Dock 再按 Office Product 分组。
- 不做长期双写。切换前影子读和对账，切换后只有 Canonical 写路径；UI 回退通过兼容 Resolver 读取新版本，不能恢复直接写原件。

### 19.2 CanonicalContentIRV1

Canonical IR 采用“统一信封 + 统一语义节点 + 格式 Profile + 资源分片”，而不是用一棵通用 Block 树压扁所有格式：

```yaml
schema_version: canonical-content-ir/v1
profile: document | spreadsheet | presentation | pdf | media | generic
source:
  file_id: number | null
  revision_id: number | null
  original_name: string
  extension: string
  mime_type: string
  size: number
  sha256: string | null
  imported_at: datetime | null
metadata:
  title: string
  locale: string | null
  author: string | null
  created_at: datetime | null
  modified_at: datetime | null
  properties: object
nodes:
  - id: uuid
    kind: string
    parent_id: uuid | null
    order: number
    text: string
    attrs: object
    style: object
    provenance: object
resource_refs:
  - ref_id: uuid
    resource_id: number
    role: string
    anchor_type: document | node | page | slide | sheet | range
    anchor_id: string | null
    geometry: object | null
    geometry_unit: string | null
    transform: object | null
    alt_text: string | null
profile_data: object
fidelity:
  level: lossless | structural | textual | metadata_only
  editable: boolean
  supported_features: string[]
  unsupported_features: string[]
diagnostics:
  - code: string
    severity: info | warning | error
    path: string | null
    message: string
    recoverable: boolean
extensions: object
```

规则固定如下：

- `nodes` 只保存扁平数组，以 `parent_id + order` 表达树，不再同时存递归 children。
- 新节点使用 UUIDv7；旧数据迁移使用基于旧对象身份的确定性 UUIDv5；节点跨版本保持稳定。
- `extensions` 只允许命名空间键，例如 `legacy.knowledge`、`ooxml.preserved_parts`，未知字段不得静默丢弃。
- `content_sha256` 使用 RFC 8785 规范化 JSON 后计算真实 SHA-256。
- Source hash 必须是原始字节 SHA-256。当前 `SHA256(MD5字符串)` 与再次 MD5 比较的实现必须删除。
- 大型 Spreadsheet/PPT/PDF 不把完整数据塞进一个 `content_json`；Version 保存 manifest 和引用，sheet/range、slide payload、rendered page 等作为版本化 Resource 分片按需加载。

格式 Profile 冻结为：

| Profile | 必填结构 | V1 编辑边界 |
|---|---|---|
| document | sections、page_settings、style_catalog、headers、footers、footnotes | 系统原生文档可编辑；导入文件按 fidelity 判定 |
| spreadsheet | workbook、sheets、sheet_parts、defined_names、calculations、print_settings | 逐文件保真门禁，不按后缀整体放开编辑 |
| presentation | page_size、theme、masters、layouts、slides、elements、notes | 元素保留 geometry、z_order、style 与资源关系 |
| pdf | pages、text_spans、images、annotations、forms、ocr_layers、rendered_pages | 固定版式只读；批注作为新版本派生层 |
| media | streams、duration、dimensions、tracks、transcript、markers、derivatives | 原媒体只读；字幕、标记和描述可版本化 |
| generic | detected_type、preview_refs、metadata、download_capability | metadata-only；必须有 Package，不编辑原始二进制 |

Spreadsheet 单元格至少保存 `address/value_type/value/formula/cached_value/style_ref/number_format/hyperlink/comment`，并覆盖 sheet 顺序、隐藏状态、冻结窗格、合并、行列尺寸、数据验证、条件格式、图表、媒体和打印设置。发现导入文件含未支持特性时，整份文件自动降级只读，禁止“尽量导出”后静默丢内容。

旧 IR 迁移优先级固定为：

```text
按 SourceFileRevision 重新解析
> 现有 ContentPackageIR
> 平台 DocumentIR
> Knowledge DocumentIr
> Excel 旧状态抢救
```

Knowledge chunk、fusion、embedding 都是派生索引，不得反向覆盖 Canonical 内容。Excel 旧表无法证明公式、图表、名称、批注等字段时，只生成 `structural + editable=false` 的恢复版本。

### 19.3 数据库迁移边界

#### A. 复用现有表

- `framework_content_packages`、`framework_content_package_versions`、`framework_resources`、`framework_resource_refs` 继续作为内容主体。
- `framework_system_task_queues` 继续作为唯一 Stage 执行器和 Stage 账本，不新建第二套 Worker/Stage Queue。
- `framework_event_log` 继续作为事务 Outbox 和事件审计。
- `framework_artifacts`、`framework_artifact_versions`、`framework_file_items` 共同承载 ProjectedFile，不另建平行 Artifact 表。
- `framework_desktop_states.state_json` 继续承载窗口会话，但保存接口增加 CAS。

#### B. 新增表

1. `framework_file_revisions`
   `id, file_id, revision_no, storage_path, size, sha256, mime_type, origin, created_by, created_at`；唯一键 `(file_id, revision_no)`。
2. `framework_ingestion_runs`
   `id(UUID), file_id, source_revision_id, owner_id, source_sha256, pipeline_version, generation, package_id, package_version_id, status, trigger, requested_by, replay_of_id, cancel fields, error fields, lock_version, timestamps`；唯一键 `(source_revision_id, pipeline_version, generation)`。
3. `framework_event_deliveries`
   `event_id, handler_key, handler_version, status, attempt_count, max_attempts, lease fields, retry_at, last_error, timestamps`；唯一键 `(event_id, handler_key, handler_version)`。
4. `framework_content_edit_leases`
   `package_id, base_version_id, token_hash, holder_id, expires_at, revision, timestamps`；每个 Package 最多一个有效编辑租约。
5. `framework_product_overrides`
   `product_id, scope_type(global|user), scope_id, enabled, sort_order, visibility_json, config_json, revision, updated_by, timestamps`；唯一键 `(scope_type, scope_id, product_id)`。
6. `framework_migration_batches` 与 `framework_migration_items`
   保存 migration_version、scope、checkpoint、before/after hash、目标 ID、attempt、status、error 和报告，用于中断续跑、隔离和对账。
7. `framework_resource_analyses`
   `resource_id, analysis_type, model, model_version, input_sha256, output_json, status, timestamps`；分析结果按输入和模型版本不可变，替代 Resource 上原地更新的 OCR/VLM/ASR 字段。

#### C. 现有表新增字段/约束

- `framework_file_items`：新增 `current_revision_id`、`sha256_hash`；现有 `storage_path/size/md5_hash` 在兼容期作为当前 Revision 镜像，不再是血缘事实源。
- `framework_content_packages`：新增 `profile`、`schema_version`、`source_revision_id`、`active_ingestion_id`；`source_file_id WHERE deleted=false` 建部分唯一索引。
- `framework_content_package_versions`：新增 `parent_version_id`、`schema_version`、`profile`、`content_sha256`、`source_sha256`、`fidelity_level`、`retention_state(active|superseded|gc_pending)`；内容、Profile、hash 和 parent 创建后不可 UPDATE，只有独立 retention_state 允许按状态机 CAS 迁移。
- `framework_resources`：`hash` 统一保存真实字节 SHA-256，删除全局 unique，改为 `(owner_id, hash)` 唯一；迁移先新增/回填真实 hash 字段，再切约束。**注意甄别方向与直觉相反**：代码实际是默认 `sha256(纯字节)` 无盐（本就是真 hash），仅当同内容被别的 owner 占用、撞全局唯一约束时才降级为 `sha256("{owner}:{content_hash}")`（`resource_service.py:40-43`）。因此绝大多数存量值就是真字节 hash，可直接保留；迁移只需挑出带 `{owner}:` 前缀的少数碰撞降级项按原字节重算，不要把普遍的纯字节 hash 当成需要清洗的加盐值。
- `framework_resource_refs`：`version_id` 改为必填，新增稳定 `ref_key`；删除当前 `(package_id, resource_id)` 唯一索引，改为 `(version_id, ref_key)`，允许同一资源跨版本、多位置引用。
- `framework_system_task_queues`：新增 `idempotency_key` 唯一索引、`ingestion_id`、`stage_version`、`handler_version`、`input_hash`、`output_hash`、取消字段；继续复用现有 lease、heartbeat、retry、dependency 和 attempt。
- `framework_event_log`：新增 `event_version`、`aggregate_type/id`、`correlation_id`、`causation_id`、`available_at`。
- `framework_artifact_versions`：新增 `source_content_version_id`、`projection_kind`、`target_format`、`adapter_version`、`options_sha256`、`fidelity_level`；一个 projection 的不可变身份是 `artifact_version_id + binary_hash`。
- `kb_documents`：新增 `content_version_id`、`indexed_content_version_id`、`active_index_generation`，与 `content_package_id` 成对写入；Knowledge 只索引确定版本，换版完成后 CAS 活跃索引代次。
- Excel workbook/session 表新增 `package_id/base_version_id/session_id/runtime_revision/dirty/expires_at`，切换后只作为可丢弃运行缓存。
- Desktop State 保存请求新增 `expected_version`；数据库 CAS 失败返回 409，禁止多个窗口静默覆盖。

数据库不变量：

- 每个非删除 File 必须有且只有一个 current Revision，Revision 必须属于该 File。
- 每个非删除 source File 至多一个 active Package。
- `ready/degraded/stale` Package 必须有属于自己的 current Version；使用组合外键保证 Package/Version 对应。
- Version 的 `(package_id, version_no)` 唯一；parent 必须同 Package 且版本号更小。
- Resource 保存真实字节 SHA-256；ResourceRef 必须属于确定 Version。
- 任何保存都必须携带 `expected_version_id`；不匹配返回 409，不能最后写入覆盖。
- Knowledge、Excel Runtime、搜索索引和预览可删除重建，不参与真值判定。

### 19.4 Ingestion Orchestrator 合同

唯一执行链：

```text
不可变 Blob
-> 单一事务提交 File + FileRevision + IngestionRun + OutboxEvent
-> Outbox Delivery
-> Ingestion Orchestrator
-> SystemTaskQueue Stage DAG
-> ContentPackage/Version/Resource
-> Knowledge 派生 DAG
```

- 上传服务不得在事件之前内部 commit；数据库事务失败产生的孤儿 Blob 由 hash 巡检清理。
- 复用 `append_event_in_transaction`，提供至少一次投递，不承诺全局 exactly-once。
- Content 和 Knowledge 不再分别直接监听 `file.uploaded` 做解析；事件只唤醒 Orchestrator。
- Stage 结果、Stage 终态、后继任务和 `task.settled` 必须同事务提交。
- 所有 Worker 提交带 `task_id + lease_token` fencing，过期 Worker 无权覆盖新 Worker。
- 终态 Run 不原地重开；人工 replay 创建 `generation + 1` 并保留原账本。

幂等键固定为：

```text
outbox     = file.uploaded:v1:{file_id}:{source_sha256}
ingestion  = ingest:v1:{source_revision_id}:{pipeline_version}:{generation}
stage      = stage:v1:{ingestion_id}:{stage_key}:{stage_version}:{input_hash}
task       = ingestion-stage:{stage_idempotency_key}
delivery   = event:{event_id}:{handler_key}:{handler_version}
projection = projection:{version_id}:{kind}:{format}:{options_sha256}
```

状态枚举只保留以下一套：

```text
Package:    draft | ready | degraded | failed | stale | archived
Ingestion:  queued | running | waiting_retry | paused | cancelling |
            completed | degraded | failed | dead_letter | cancelled
Stage/Task: blocked | pending | running | retry_wait | succeeded |
            degraded | skipped | failed | dead_letter | cancelled
Projection: queued | building | ready | failed | superseded
Migration:  planned | running | paused | validated | failed |
            rolled_back | completed
```

存量状态映射（施工必须先执行，否则门禁 0 通过）：数据库 `framework_content_packages.status` 现有值只有 `parsed / archived / degraded / stale`，**没有 `ready / draft / failed`**。迁移按下表统一：

```text
parsed   -> ready       （已完成解析、有 current_version 的存量包）
archived -> archived    （保持）
degraded -> degraded    （保持）
stale    -> stale       （保持）
draft    仅由 Office 新建草稿产生（新逻辑）
failed   仅由 Ingestion 终态失败写入（新逻辑）
```

`origin_type` 现有值只有 `uploaded / generated`；FileRevision.origin 值域固定为 `user_import | projection | external_replace`，迁移映射 `uploaded->user_import`、`generated->projection`；packages 旧 `origin_type` 停用为兼容镜像，血缘以 FileRevision.origin 为准。

`skipped` 只能表示某个 Stage 不适用；未知格式仍必须经过 `package_ensure` 并得到 generic metadata-only Package。重新预处理时旧 current Version 继续可读，处理状态留在 Ingestion，不把 Package 临时改成不可读。

Stage DAG 固定为：

```text
package_ensure
-> canonical_parse
-> resource_extract
-> derivative_build
-> knowledge_register
-> Knowledge 子 DAG
```

API 至少包括：

- 上传响应：`202 + file_id + source_revision_id + ingestion_id + status=queued`。
- `GET /api/ingestions/{id}` 与 `GET /api/files/{id}/ingestion`。
- `POST /api/ingestions/{id}/cancel|resume|replay`。
- `GET /api/ingestions/{id}/events`，支持 SSE 或带 cursor 的增量轮询。

### 19.5 Product、Resolver 与会话合同

`ProductManifestV1` 固定字段：

```text
schemaVersion, productId, version, displayName, aliases, description,
category, iconSet, entryComponentKey, workspaceKind, visibility,
permissionPolicy(allOf/anyOf), requiredCapabilities(module/action/minVersion),
fileAssociations, createDocumentTypes, windowPolicy, activationPolicy,
deepLinks, commands, frameworkCompatibility
```

FileAssociation 必须声明 `associationId/contentTypes/extensions/mimeTypes/modes/adapterId/priority/readOnlyFormats`。数据库覆盖层只能修改 enabled、sort order、visibility 和允许的 config，不能覆盖 Product 身份、入口组件或 required capabilities。Catalog 合并顺序：

```text
ProductManifest -> global override -> user override
-> permission policy -> capability availability -> effective Product
```

无权限 Product 不下发前端。重复 Product ID、缺失组件、能力版本不满足、Manifest hash 漂移直接使构建或 Catalog 校验失败，不允许静默跳过。

`OpenContentIntentV1`：

```text
resolverVersion, requestId,
source(fileId | packageId/versionId | deepLink),
requestedMode(view|edit), preferredProductId,
activation(reuse-tab|new-tab|new-window),
expectedVersionId, lockToken, origin(productId/windowId)
```

Resolver 响应包含 `resolutionId/outcome/productId/adapterId/grantedMode/readonlyReason`，以及 file/source revision/projection/package/version/content type/format/mime/title、permissions、ingestion、lock、session reuse、deep link 和 Catalog revision。解析顺序固定为：

```text
定位 File/Revision/Artifact Projection
-> 权限
-> Package/Version
-> Ingestion/readable state
-> Product 权限
-> FileAssociation
-> Adapter/fidelity
-> 模式降级
-> 编辑租约
```

前端删除后缀到 Viewer 的自主选择，只消费 Resolver 结果。

`ProductWindowSessionV2` 保存 `windowId/productId/instanceKey/geometry/windowState/zOrder/active/tabOrder/activeDocumentSessionId/productState/restoreState/revision/timestamps`。

`DocumentSessionV1` 保存 `sessionId/windowId/productId/packageId/versionId/fileId?/artifactId?/title/contentType/format/adapterId/requestedMode/grantedMode/lifecycle/dirty/saveState/expectedVersionId/lock/conflict/viewState/undoRedoBoundary/restoreState`。Desktop State 只存引用和视图状态，不存完整文档正文。

### 19.6 OfficeDocumentAdapterV1 与保存语义

Adapter 生命周期固定为：

```text
descriptor
createDraft
open
hydrate(page/slice)
edit
getDirtyState
save(expectedVersionId, idempotencyKey, lockToken)
saveAs
export
publish/materializeProjection
canClose
renewLock
dispose
```

- Adapter 只能调用 Content Runtime，不能接收物理路径或自行调用 Parser。
- `createDraft` 原子创建 source-less Package 和空白 Version，返回 `file_id=null`。
- 自动保存只新增 Canonical Version，不产生桌面 File。
- 首次显式 Save 选择名称/位置，创建 File + Artifact + primary ArtifactVersion/FileRevision；Package 仍无 source_file_id。
- 导入文件保存时，SourceFileRevision 永久保留；保存创建新的 Canonical Version，并把 primary projection 物化为该 File 的新 `origin=projection` Revision，不覆盖导入 Revision。
- Save As 克隆为新 Package，再创建新 File 与 primary projection。
- Export 创建一次性 ArtifactVersion；Publish 创建持久 projection。每次物化都必须绑定唯一 ContentPackageVersion。
- 保存时重新校验权限、lock token 与 expected_version_id；冲突返回当前版本，不自动覆盖。
- PDF、图片、音视频和 generic 原内容只读；允许批注、字幕、标记、描述等派生层版本化。

稳定错误响应结构：`code/message/request_id/retryable/details/expected/current`。至少固定以下错误码：

```text
PRODUCT_NOT_FOUND / PRODUCT_DISABLED / PRODUCT_FORBIDDEN
PRODUCT_RUNTIME_UNAVAILABLE / ADAPTER_UNAVAILABLE
CONTENT_NOT_FOUND / CONTENT_SOURCE_UNAVAILABLE
CONTENT_IR_SCHEMA_UNSUPPORTED / CONTENT_IR_PROFILE_INVALID
CONTENT_VERSION_CONFLICT / CONTENT_LOCKED / CONTENT_LOCK_LOST
CONTENT_FORMAT_READ_ONLY / CONTENT_FIDELITY_UNSUPPORTED
CONTENT_RESOURCE_MISSING / CONTENT_PROJECTION_FAILED
INGEST_SOURCE_MISSING / INGEST_SOURCE_CHANGED / INGEST_LEASE_LOST
INGEST_STAGE_TIMEOUT / INGEST_RETRY_EXHAUSTED / INGEST_DEAD_LETTER
DESKTOP_STATE_CONFLICT / IDEMPOTENCY_CONFLICT
MIGRATION_ITEM_UNMAPPABLE / MIGRATION_ITEM_QUARANTINED
```

处理中和只读降级是正常 Resolver outcome，返回 200；异步任务受理返回 202；版本/幂等冲突 409；租约问题 423；权限 403；不支持内容 422；运行时不可用 503。

### 19.7 迁移、切换与回滚

迁移不硬编码当前数据规模。M0 预检必须实时生成数据库大小、存储大小、表计数、最大 ID、文件 SHA-256 清单、无 Package 文件、无 Version Package、Knowledge 未绑定和旧会话统计，并将其作为后续对账基线。

执行批次固定为：

1. M0：冻结 schema；完成数据库与文件存储同 checkpoint 快照；生成基线报告。
2. M1：仅增加表、字段、索引和服务端 Feature Flag，不切换读取。
3. M2：为现有 File 回填 FileRevision 和真实 SHA-256，先去重/隔离重复 Package。
4. M3：先回填 Knowledge 引用文件，再回填其余文件；每批不超过 500 项或 2GB，以 `(owner_id,file_id)` checkpoint 续跑。
5. M4：迁移三套 IR；无法无损迁移的条目进入 quarantine，并按 SourceRevision 重解析。
6. M5：Knowledge 绑定确定 `package_id + version_id`；旧 chunk/fusion 保留为派生历史，不再读原件。
7. M6：Excel 旧状态转换为 Spreadsheet Profile；保真不足的版本只读，旧 Excel 表转只读缓存。
8. M7：旧 appKey/payload/Desktop State 迁移；失败窗口生成恢复失败标签，禁止静默丢弃。
9. M8：Artifact/File 回填 projection lineage；完成全量不变量巡检与至少 72 小时影子读对账。
10. M9：短暂写冻结，排空任务，增量补齐，再次对账，切换 `content_write_v2` 和 `/desktop`。

服务端开关固定为 `content_ingestion_v2`、`content_read_v2`、`content_write_v2`、`desktop_runtime_v2`、`migration_freeze`，每次变更写审计日志，不能只放在前端 localStorage。

回滚规则：

- UI 异常先关闭 `desktop_runtime_v2`，目标 15 分钟内恢复旧 UI；旧 UI 必须通过兼容 Resolver 读 Canonical Version。
- `content_write_v2` 不因 UI 回滚自动关闭。内容写链异常时开启 `migration_freeze`，保留所有新 Version，修复后按账本重放。
- 已接受的 ContentPackageVersion、FileRevision、ArtifactVersion 永不覆盖或删除；禁止把新内容反写到原始 Revision。
- 灾备目标：数据库 WAL/PITR 与文件存储增量快照使用同一 checkpoint，建议 RPO 不超过 15 分钟、完整恢复 RTO 不超过 4 小时、UI 回退 RTO 不超过 15 分钟。

旧键迁移至少固定：

```text
desktop -> files
recycle -> recycle
doc-viewer/pdf-viewer/ppt-viewer/excel-engine/docs-open -> office
text-editor -> text
image-viewer/media-intelligence -> media
agent -> ai

写作模块 -> content-studio

```

旧 `{fileId,fileName,format,mode,page}` 先转换成 OpenContentIntent，再由 Resolver 生成 DocumentSession；`page` 进入 viewState。

## 21. 开工前生命周期闭环

本节来自对现有 File、Recycle、Share、Resource、Artifact、Knowledge 和 Office 服务的逐操作预演。它补齐第 19、20 节没有覆盖的对象生命周期；后续不得由各 Router 自行决定复制、删除、权限和下载语义。

### 21.1 File、Revision、Package 的操作矩阵

| 用户操作 | File/FileRevision | ContentPackageVersion | Ingestion | Knowledge/Projection |
|---|---|---|---|---|
| 上传 | 创建 File + `origin=user_import` Revision | 创建 Package | 新 Run | 后续消费确定 Version |
| 重命名/移动 | 只改 File 元数据 | 不变 | 不触发 | Artifact/Knowledge 引用不变 |
| 复制 | 新 File + 独立 Revision 身份，可复用同一 Blob | Fork 新 Package/Version，保留 `copied_from` lineage | 可直接复用已验证 Version；否则新 Run | 新副本后续独立编辑 |
| 自动保存 | File 不变 | hash 变化时新增 autosave Version | 不触发 Parser | Knowledge 可标 stale，不立即重建深层索引 |
| 显式保存 | 新增 Canonical Version；primary projection 完成后新增 `origin=projection` Revision | 原子切 current Version | Projection Task | Knowledge 增量换版 |
| 移入废纸篓 | File 标记 recycled，Revision/Package 均保留 | 继续可恢复 | 不重跑 | 普通入口隐藏；已开会话降为只读 |
| 恢复 | File 恢复 active | 原 Version 继续使用 | hash 未变不重跑 | 恢复原入口和权限 |
| 永久删除 File | File tombstone；撤销普通下载 | Package 根据引用图决定保留或归档 | 取消未开始任务 | Knowledge/Artifact 独立引用不盲删；无引用对象进入 GC |
| 彻底清除内容 | 删除 Package 可见性与派生引用，保留最小审计 tombstone | 终止会话和租约 | 取消/死信隔离 | Knowledge、Projection、Resource 引用按图清理 |

复制不能继续只复制 `framework_file_items` 一行。目标语义：

1. 新 File 拥有自己的 Revision ID，但底层不可变 Blob 可以内容寻址复用。
2. 已 ready/degraded 的 Package 通过 `fork_package(source_package_id, source_version_id)` 创建独立 Package 和 Version，复用不可变 Resource。
3. 两个 Package 后续保存互不影响；不能让两个桌面文件指向同一个可编辑 Package。
4. 复制时源内容未 ready、hash 不一致或 Profile 迁移失败，则为副本创建独立 IngestionRun。

文件夹复制必须补齐为 durable copy job：先创建目录结构，再按稳定 source item ID 顺序逐文件复制/分叉 Package；使用 batch checkpoint 和 item idempotency，支持取消、失败续跑和进度展示。大文件夹不使用一个超长数据库事务，但任一 item 内的 File、Revision、Package fork 和 Outbox 必须原子提交。

文件夹批量移动、回收和恢复只生成一个 batch/correlation ID，再为每个 File 写幂等 lifecycle item；不能在 Router 循环中逐个 commit 后再同步发事件。

### 21.2 回收、永久删除与引用图

“永久删除文件”与“彻底删除所有衍生内容”不是同一动作：类似文件被导入独立知识库后，删除 Finder 文件不应静默摧毁知识库资产。

- 移入废纸篓：File、Revision、Package、Version、Resource、Knowledge 全部保留；普通 Files/Office 最近入口隐藏。
- 已打开会话发现 File recycled 后，允许读取已加载/已提交 Version，但禁止继续覆盖 primary projection；用户可选择恢复或 Save As。
- 永久删除 File：删除 File 的普通访问与 source download；若 Package 仍被 Knowledge、Artifact、共享或其他正式引用使用，则保留 Package 并标记 `source_unavailable` Resolver 状态。
- 无正式引用的 Package、Artifact、Revision 和 Resource 进入 `gc_pending`，经过可配置宽限期后物理清除。
- “彻底清除内容和衍生数据”必须先返回 impact preview，列出 Package、Knowledge Document、Projection、Share 和 Resource 数量，再要求显式 confirm token。
- 共享对象、已发布 Artifact 或被其他用户引用的内容不能由单个 File 删除动作越权清除；只撤销当前用户可控制的引用。

GC 不依赖可漂移的手工 `ref_count`。引用事实来自 File.current_revision、PackageVersion ResourceRef、ArtifactVersion、Knowledge 绑定和有效 Share；`ref_count` 只作缓存，可由巡检重算。

### 21.3 统一 ContentAccessPolicy

新建 `ContentAccessPolicyService`，Resolver、Content Runtime、Office、Export、Knowledge 和 Resource 下载全部调用同一服务，禁止继续散落 `owner_id == current_user` 判断。

权限来源：

```text
owner
-> FileShare(read/edit/comment/publish/reshare + scope + expiry)
-> Product permission policy
-> Package/Artifact/Knowledge object scope
-> requested action
```

规则固定为：

- 导入 Package 默认继承 Source File 的有效权限；source-less draft 由 Package owner/ACL 控制。
- `read` 允许读取 Canonical 内容和授权 Resource；`edit` 还需编辑租约；`publish` 必须显式授权，不能由 edit 推导。
- Save 时重新校验 Package edit 权限；物化到目标 File 时再校验目标 File write 权限。
- Share scope 明确区分 `file/source_revision/content_package/artifact/knowledge`，不能因拿到 Resource ID 直接下载字节。
- Resource API 通过 PackageVersion/ArtifactVersion 引用做授权，不提供只凭 resource_id 绕过上下文的公开读取。
- 权限或 Share 到期后立即阻止新请求；已打开编辑会话在下一次 autosave/renew lock 时收到 `PERMISSION_CHANGED` 并转只读。

### 21.4 Version、自动保存与原子可见性

每次保存采用两阶段构建、单事务发布：

```text
构建/复用 Resource 分片
-> 校验 CanonicalContentIR/Profile/fidelity
-> 计算 content_sha256
-> 事务内 INSERT Version + ResourceRef + current_version CAS + OutboxEvent
-> commit 后 Version 才对 Resolver 可见
```

- 构建阶段失败只留下可 GC 的孤儿 Blob，不产生半成品 current Version。
- 内容 hash 与 current Version 相同，返回现有 Version，不制造空版本。
- autosave 使用稳定 idempotency key 并在短窗口内合并；manual save、publish、restore 永不被合并。
- Version operation type 固定为 `import/create/autosave/manual_save/save_as/restore/system_migration`。
- unchanged Resource 分片跨版本复用；Spreadsheet 只重写变化 range/sheet part，避免每次 autosave 复制整个工作簿。
- V1 永久保留 manual/published/restored/migration Version。Autosave 保留最近 100 个以及所有被会话、Projection、Knowledge 引用的版本；更老的未引用 autosave 先汇总为每日 checkpoint，再由后台 compactor 清理 payload。
- compactor 不能修改已有 Version 内容；只新增 checkpoint Version，并把可清理旧 Version 标记 `superseded` 后交给引用图 GC。

### 21.5 Resource 不可变与分析结果版本化

当前 Resource 服务存在三个必须在 WP1 修复的问题：内部 commit、跨 owner 时用 owner+hash 派生假 hash、OCR/description 原地更新。

目标合同：

- Resource 二进制和核心元数据创建后不可变；`sha256` 永远是真实字节 SHA-256。
- Resource 唯一键改为 `(owner_id, sha256)`；不同 owner 不通过公开模型共享 Resource 身份，底层 Blob 是否去重由存储层内部决定。
- `create_resource/add_ref` 只 flush，由上层 Unit of Work commit。
- OCR、VLM 描述、ASR、缩略图诊断不再修改 Resource；新增 `framework_resource_analyses`：`resource_id, analysis_type, model, model_version, input_sha256, output_json, status, timestamps`，唯一键覆盖输入和模型版本。
- ResourceRef 必须绑定 Version 和稳定 ref_key；同一 Resource 可在同一 Version 多位置出现。
- 删除 Version/Artifact/FileRevision 时只删除引用；真实 Blob 在所有引用消失且超过 GC 宽限期后清理。
- 每日巡检对比数据库 sha256、物理 Blob、引用图和缓存 ref_count；任何不一致进入 quarantine，不能直接删文件。

### 21.6 Knowledge 换版与可查询一致性

Knowledge 只能索引确定 `package_id + content_version_id`，换版采用新索引构建完成后原子切换：

```text
current PackageVersion N
-> enqueue knowledge_sync(N)
-> 构建 chunks/embeddings/fusion/profile/graph，全部记录 N
-> search_ready 后 CAS active_index_version=N
-> 旧索引延迟 GC
```

- 搜索结果必须返回 `indexed_content_version_id` 和 `is_stale`。
- Canonical Version 已到 N+1、Knowledge 仍在 N 时，旧索引继续可搜索但明确 stale，不出现搜索空窗。
- 普通 autosave 只触发轻量 debounce；manual save/publish 才立即触发完整 Knowledge 更新。
- Knowledge 深层阶段失败不回滚 Canonical Version；`content_ready/search_ready/deep_ready` 三种 readiness 分别展示。
- 删除或取消 Knowledge Document 只移除 Knowledge 派生层和引用，不反向删除用户 File/Package；“彻底清除”除外。

### 21.7 Projection、下载与用户可见保存状态

Canonical 保存与物理投影是两个可观察步骤：

```text
content_state: saving -> saved | conflict | failed
projection_state: idle -> queued -> building -> ready | failed | stale
```

- Canonical commit 成功即显示“内容已保存”；primary projection 未完成时同时显示“文件同步中”。
- Projection 失败不丢内容、不回滚 Version；提供重试，下载当前版本时可重新物化。
- primary projection 必须携带 source content version；若小于 Package current version，Resolver 返回 `projection_state=stale`。
- 不再使用“编译失败后静默返回原件”的 fallback。无法保真投影时返回明确错误或只读结论。
- 默认“下载”下载当前用户可见 Version 的最新 ready projection；无 ready projection 时先创建 export job。
- “下载原件”是单独命令，只读取 Package.source_revision_id，要求 source read 权限，并在原件已永久删除时不可用。
- “导出为”总是从指定 ContentPackageVersion 生成新 ArtifactVersion；不能依据 File 当前物理路径猜测内容版本。

### 21.8 新增生命周期验收

以下用例补入 Gate 1、3、5、7：

1. 复制同一文件后两个 Package 独立保存，任一副本修改不影响另一副本。
2. 重命名/移动不创建 Version，不触发 Parser，不改变 Knowledge 引用。
3. 回收、恢复、永久删除、彻底清除分别验证 File/Package/Knowledge/Artifact/Resource 状态。
4. 共享 read/edit/publish、权限过期、保存时权限变化和 Resource 越权下载测试。
5. 100 次相同 autosave 只产生一个有效 Version；并发 manual save 只有一个 CAS 成功。
6. 保存中崩溃不产生半成品 current Version；孤儿 Blob 可被巡检识别并安全 GC。
7. Resource OCR/VLM 换模型生成新 Analysis，不改变历史 Version 的可复现结果。
8. Knowledge N 到 N+1 重建期间持续可搜索 N，完成后原子切 N+1，并正确返回 stale 标记。
9. Canonical 保存成功但 Projection 失败时，刷新后内容仍在，可重试并最终下载当前版本。
10. 默认下载、下载原件、导出为三条路径返回各自声明的 Version/Revision/hash。

完成本节后，用户常见对象操作已经覆盖完整生命周期。后续若实现阶段发现未列出的业务操作，必须先归入上述 File、Version、Projection、Knowledge 或权限合同之一，不能再新增旁路语义。

## 22. 在线迁移与容量护栏

本节防止合同正确、迁移方式却把现有生产数据拖垮。2026-07-17 本地只读基线为：PostgreSQL 约 31GB，`data/uploads` 约 148GB/54,883 个物理文件，当前磁盘可用约 2.5TiB。该数字只用于评估量级，M0 必须重新测量并写入 migration batch，不能硬编码为验收结果。

### 22.1 不复制 148GB 原件

- FileRevision 回填只引用现有 content-addressed `storage_path`，计算并记录真实 SHA-256，不复制、不重命名现有 Blob。
- 只有新上传、外部替换和新 Projection 才写新 Blob；相同 owner + sha256 可复用存储层内容。
- 旧 MD5 路径在兼容期保持可读；迁移完成后也不要求批量改名为 SHA-256 路径。
- SHA-256 回填按字节预算限流，默认最多 2 个并发 reader；支持按吞吐、系统负载和业务时段动态降速。
- 每个文件 hash 完成立即提交 migration item checkpoint，进程退出不重扫已验证文件。
- 遇到物理缺失、权限错误、hash 变化或读取失败进入 quarantine，不生成伪 Revision。

### 22.2 PostgreSQL 在线 DDL

- Additive migration 只添加 nullable 字段、无重写默认值的列和新表。
- 大表索引使用 `CREATE INDEX CONCURRENTLY`；Alembic 通过 autocommit block 执行，失败后可检测并清理 invalid index。
- 外键和 CHECK 先 `NOT VALID` 添加，回填对账后 `VALIDATE CONSTRAINT`，最后再提升 NOT NULL。
- active Package 等唯一约束先运行重复检测与 quarantine，再创建并发唯一索引，禁止直接在脏数据上尝试。
- DDL 设置短 `lock_timeout`，拿不到锁立即失败并重排窗口，不等待到阻塞业务请求。
- Data backfill 不进入 Alembic transaction；每批最多 500 项或 2GB，并按实际耗时自动下调。
- 禁止在应用启动、Model import、Worker bootstrap 中执行本轮 DDL，避免多进程争锁和不可审计的半迁移。

### 22.3 磁盘、备份与临时文件水位

- M0 同时记录数据库、上传目录、Resource、临时导出目录和 WAL 的容量与 inode。
- 开工最低水位：可用空间不少于总盘 20%，且不少于 `max(100GB, 两倍最大临时导出需求 + 一次数据库物理备份)`；低于任一条件自动暂停。
- 数据库备份和文件快照不能只存同一块物理盘。至少一份 checkpoint 快照位于独立卷或远端存储，并完成实际恢复。
- Parser、Office Export 和 Projection 的临时文件统一进入受配额目录；任务结束、取消、超时和 Worker 崩溃后由 janitor 清理。
- 禁止为影子读同时保存第二份全量 Resource；影子结果只保存 hash、差异摘要和必要的 quarantine 证据。

### 22.4 迁移运行时限流

迁移 Worker 使用独立 lane 和资源预算，不与在线上传、Office 保存和 Knowledge 搜索抢同一优先级：

```text
online save/upload   highest
interactive preview high
normal ingestion    normal
knowledge deep DAG  low
migration backfill  lowest
```

自动暂停条件至少包括：

- 数据库连接池等待或锁等待超过阈值。
- API p95 延迟连续超标。
- 磁盘可用空间低于水位。
- Outbox/Task 正常业务积压持续增长。
- Migration item 失败率、hash mismatch 或 quarantine 比例超阈值。
- Worker 内存、临时目录或 WAL 增长异常。

恢复时从 checkpoint 继续，不提高并发“追进度”。迁移吞吐、ETA 和暂停原因必须在管理接口可见。

### 22.5 Cutover 前容量验证

切换前必须使用真实最大样本验证：

1. 最大 DOCX/XLSX/PPTX/PDF 的 parse、hydrate、save 和 Projection 峰值内存。
2. 大 Spreadsheet 分片加载不会把完整 workbook JSON 拉到浏览器或单次 API 响应。
3. 1,000 个并发小文件上传与少量大文件混合时，Outbox 和 Stage 能公平调度。
4. 迁移、正常 Ingestion、Knowledge 搜索和 Office 保存同时运行时，交互请求不被饿死。
5. Worker 强杀后临时文件、lease、任务和 migration checkpoint 均可恢复。
6. 完整数据库恢复后，FileRevision hash、ContentVersion hash 和物理 Blob 100% 对账。

### 22.6 监控与停止线

必须提供并在 72 小时影子期观察：

- 上传事务成功率、Outbox 延迟、Stage queue age、重试/死信数量。
- Package/Version/ResourceRef 不变量巡检结果。
- Knowledge indexed version lag、Projection stale/failed 数量。
- Desktop State CAS 冲突率、Office save conflict/lock lost 比例。
- 数据库 p95、锁等待、WAL 增长；磁盘使用、临时文件和孤儿 Blob。
- 新旧 Resolver 结果差异，按声明差异与异常差异分别统计。
