> 审计方式：只读。数据库表结构/约束实查 + 4 路子代理核实代码层落点（内容管线/桌面注册/Viewer 旁路/三套 IR）

## 0. 总体结论

**方案诊断方向正确，蓝图基本完整，绝大多数点名的文件路径与断言都被代码证实。** 但有 4 个"合同层"的错误/不精确点会直接让目标模式验收翻车，必须先修方案再开工；另有 1 个规模现实会让 §18.3 的"所有文件都有 Package"这条硬门禁在当前口径下不可能通过。

一句话：**方案能落地，但不能照现在的字面开目标模式。** 目标模式按目标验收，而方案里有几个目标的枚举值、字段名、依赖方向和代码实际对不上，AI 会照方案里写死的错值去建表/改代码，然后自己验自己、假绿收工。

## 2. 会让目标模式翻车的问题（P0，开工前必须先改方案）

### 问题 A：Package 状态枚举，方案写死 `ready`，数据库实际没有 `ready`

- 方案 §19.4 冻结 `Package: draft | ready | degraded | failed | stale | archived`。
- 数据库实查：`framework_content_packages.status` 实际值只有 `parsed(221) / archived(1152) / degraded(80) / stale(79)`，**没有一条 `ready`，也没有 `draft/failed`**。
- 后果：目标模式会按方案建"ready 门禁"，而所有存量 Package 都是 `parsed`。要么迁移脚本把 `parsed→ready` 全量改写（方案没写这个映射），要么门禁永远 0 通过。
- **需要方案补一张状态映射表**：`parsed→ready`、旧无`draft/failed`如何产生、迁移方向。

### 问题 B：origin_type 枚举对不上

- 方案 §19.1 用 `origin=projection` / `origin=user_import` 区分投影/导入。
- 数据库实查：`framework_content_packages.origin_type` 实际值是 `uploaded(1442) / generated(90)`。
- 后果：方案的"投影生成的 Revision 标 origin=projection、不再触发预处理"这条防循环逻辑，落到现有字段上没有对应值。§19.3 新增的 `framework_file_revisions.origin` 是新表新字段可以自定义，但方案没说清"新表 origin 值域"和"旧 packages.origin_type 值域"是两套还是要统一。
- **需要方案明确**：FileRevision.origin 的合法值集合 + 与旧 origin_type 的关系。

### 问题 C：office-gen adapter 方向写反了

- 方案 §3(3.1 表)、§8.1、§16.1-E 反复写"office-gen adapter"作为投影/导出适配器。
- 代码实际：适配器在**框架侧** `backend/app/services/content/adapter.py:16 call_export_adapter`，它反过来 `call_capability_as_system("office-gen", fmt)` 去调 office-gen。office-gen 是**被调用方**，模块内部没有 adapter 文件。
- 后果：方案 §16.2 把"office-gen 创建/转换/导出"列入"不需要重写"，这没问题；但 §6.4/§16.1-E 说"现有模块通过 Adapter 实现 ContentHandler 合同"会误导 AI 去 office-gen 模块里找/改 adapter，实际要改的是框架 content 层。
- **需要方案澄清**：投影适配器归框架 `content/adapter.py`，office-gen 只提供 capability。

### 问题 D：framework_resources hash 的"owner 盐"是碰撞降级，不是常态

- 方案 §19.3-C 说"删除全局 unique，改为 `(owner_id, hash)` 唯一；迁移先回填真实 hash，**不能把现有 owner-salted 值误当内容 hash**"。
- 代码实际：`resource_service.py:33` 默认 `hash=sha256(纯字节)` **无盐**；只有当同内容已被别的 owner 占用、撞全局唯一约束时，才降级成 `sha256("{owner}:{content_hash}")`（`:40-43`）。`hash_algorithm` 字段恒为 "sha256"。
- 后果：方案假设"现有值大多是 owner-salted 需甄别"，实际相反——**绝大多数是纯字节 sha256（本就是真 hash），只有极少数碰撞降级项是加盐的**。迁移脚本的甄别逻辑要反过来写：默认认为是真 hash，只挑出 `{owner}:` 前缀的少数项重算。
- **需要方案修正**：owner-salted 是少数碰撞项，不是普遍状态。

## 3. 会让"硬门禁"不可能通过的规模现实（P0，需重定义门禁口径）

### 问题 E：99% 的文件当前没有 ContentPackage

数据库实查（deleted=false）：

| 指标 | 数值 |
|---|---|
| 存活文件 framework_file_items | 26,909 |
| 存活 ContentPackage | 1,453 |
| **有 Package 的文件占比** | **约 235 / 26,909 ≈ 0.9%** |
| 没有 Package 的文件 | 26,674 |
| kb_documents 存活 | 15,696 |
| kb_documents 无 content_package_id | 15,526（98.9%） |
| Package 无 current_version | 49 |
| Package 无 source_file（草稿/生成类） | 1,218 |
| 一个 source_file 对多个 Package | 0（现状干净，但无 DB 约束保证） |

- 方案 §14.3 / §18.3 门禁 1 写"每个上传文件都有唯一 ContentPackage，所有真实样本均有 Package"。
- 现实：26,909 个文件里绝大多数是**资源文件/派生物/历史散落文件**，不是都要建 Package 的用户文档。但方案没有定义"哪些 file 必须有 Package"的范围口径。
- 后果：AI 若照字面理解"所有文件都要有 Package"，要么给 2.6 万文件全量回填（§19.7 M3 每批 500 项/2GB，估算要跑几十上百批，且华哥当前云端额度紧张——见记忆"准备期-额度恢复即跑清单"），要么门禁永远不过。
- **需要方案明确**：门禁口径改为"**新上传文件**必须有 Package"+"**知识库引用文件**必须回填"，把 2.6 万历史散落文件排除或降级为 metadata-only 批处理，不阻断首次切换。
