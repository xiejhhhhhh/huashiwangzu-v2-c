# 桌面应用图标、Office 与应用内视觉统一收口计划

> 日期：2026-07-17
> 状态：已被《07_前端产品应用底座与统一工作区重构方案》替代，不实施
> 本地基线：`http://127.0.0.1:5173/desktop`
> 前置成果：桌面 MenuBar、Dock、WindowFrame、Launchpad、Spotlight 和桌面交互已经完成 macOS 化
> 核心原则：只重构产品呈现、文件打开合同和应用视觉壳，不重写现有业务逻辑与后端能力

## 0. 一页简述

### 0.1 目标流程

```text
当前桌面壳已经基本完成
   │
   ├── 修正右键子菜单双高亮
   ├── 移除 Dock 邻近图标联动放大
   ├── 重做 macOS 风格主应用图标和小尺寸图标
   └── 统一面向用户的应用名称
   │
   ▼
建立产品层应用模型
   │
   ├── 用户只看到：文件、办公、图片、文本、AI 助手等产品
   ├── Parser、开放接口、生成器、ASR 等继续作为内部能力
   └── 旧 appKey、后端模块键和历史会话保持兼容
   │
   ▼
建立统一 OfficeWorkspace
   │
   ├── Word ─→ 现有 doc-viewer 能力
   ├── Excel ─→ 现有 excel-engine 能力
   ├── PPT ───→ 现有 ppt-viewer 能力
   └── PDF ───→ 现有 pdf-viewer 能力
   │
   ▼
建立薄层 WorkspaceShell
   │
   ├── 统一工具栏、侧栏、背景、字体、间距
   ├── 统一空态、加载、错误、弹窗、Toast
   └── 保留聊天、表格、图谱、写作等业务主体
   │
   ▼
逐组迁移可见应用并完成三档视口回归
```

### 0.2 本次要解决的六件事

1. 主应用图标从“Lucide 线稿套渐变方块”升级为真正具有对象感和材质层次的 macOS 风格图标。
2. 右键菜单同一时刻只允许一个主高亮，子菜单打开时父项只能保留弱路径提示。
3. Dock 只响应指针直接命中的图标，不再让相邻图标跟随放大。
4. 面向用户的名称改为通俗名称，技术模块名不再直接出现在 Launchpad。
5. Word、Excel、PPT、PDF 统一为一个“办公”应用，查看与编辑变成文件模式而不是应用名称。
6. 打开窗口后的业务页面使用同一套 WorkspaceShell，消除每个模块像独立后台系统的问题。

### 0.3 明确不做

1. 不直接删除或重命名 `doc-viewer`、`excel-engine`、`ppt-viewer`、`pdf-viewer` 等内部模块键。
2. 不修改现有文件解析、保存、OCR、导出和 Office 生成业务实现。
3. 不把所有业务页面塞进一个巨型通用组件。
4. 不要求聊天、表格、知识图谱和内容运营页面长得完全相同。
5. 不复制 Apple、Microsoft、微信或抖音的商标与不可再分发资产。
6. 不为了视觉统一而改变权限、文件数据、任务状态或对外接口语义。

## 1. 审计结论

桌面外壳已经收口，但应用层仍存在五套以上视觉语言：

- Finder 风格文件工作区。
- AI/消息聊天工作区。
- Element Plus 管理后台。
- ViewerShell 文档查看器。
- Excel 旧式工具栏和表格工作区。
- Media Intelligence 英文接口调试页。
- 知识库普通管理页与 3D 沉浸页混合。

问题不在业务逻辑，而在产品边界和视觉合同缺失：

1. `appKey` 同时承担模块标识、产品标识和窗口标识。
2. 文件关联直接指向具体处理器，用户因此看到多个“查看器/编辑器”。
3. 各模块独立维护颜色、字体、工具栏、侧栏、空态和弹层。
4. 应用大图标和标题栏小图标使用同一套复杂结构，缩小后无法辨认。
5. 用户可见名称直接取自 manifest，技术能力被误当成产品展示。

正确方向不是重写模块，而是在现有模块上增加两层薄合同：

```text
产品层：用户看到什么应用、名称和图标
呈现层：应用打开后使用什么工作区骨架
能力层：现有模块、后端接口和处理器保持原样
```

## 2. 当前问题与根因

### 2.1 右键子菜单双高亮

当前父项打开子菜单后获得 `.is-open`，而 `.is-open`、`:hover`、`:focus-visible` 使用同一种蓝色背景。指针进入子菜单时：

```text
父项 .is-open  ─→ 主高亮
子项 :hover    ─→ 主高亮
                   ↓
             同时出现两个蓝色高亮
```

目标合同：

- 指针在父菜单时，当前父项使用主高亮。
- 指针进入子菜单时，只有当前子项使用主高亮。
- 父项只保留浅背景、箭头变色或细路径提示。
- 键盘导航可以保留父级路径，但主焦点仍只有一个。
- 父菜单与子菜单之间的 `260ms` 容错走廊保留，避免斜向移动闪退。

主要落点：

- `frontend/src/desktop/context-menu/context-menu.vue`
- `frontend/src/desktop/context-menu/use-context-menu.ts`
- `frontend/tests/desktop-macos-shell.spec.mjs`

### 2.2 Dock 非命中图标发生变化

当前实现会给悬停图标相邻两侧增加 `.is-neighbor`，执行 `scale(1.09)` 和上移 `4px`。这是代码定义的行为，不是动画残留。

目标合同：

- 只有指针直接进入某个图标按钮时，该图标才允许放大或提亮。
- 相邻图标始终保持 `scale(1)` 和 `translateY(0)`。
- 指针进入分隔线、Dock 空白区或右键菜单，全部图标立即恢复静止。
- Dock 菜单打开时暂停放大，避免菜单与图标共同位移。
- 保留当前图标的轻量反馈，建议最大不超过 `scale(1.10)`、上移不超过 `4px`。

主要落点：

- `frontend/src/desktop/taskbar/desktop-taskbar.vue`
- `frontend/tests/desktop-macos-shell.spec.mjs`

### 2.3 当前图标统一但不像 macOS 软件图标

当前图标统一使用：圆角方块、渐变、顶部高光、阴影和中心 Lucide 线稿。它解决了混用 PNG/SVG 的问题，但仍像“统一图标模板”，缺少独立对象和应用个性。

主要问题：

- PDF 与 Word 等应用复用相近线稿，只靠颜色区分。
- 所有应用共用同一横向高光，更像旧 Aqua/iOS 模板。
- `58px` Launchpad 与 `16px` 标题栏共用同一结构。
- 标题栏内实际中心符号约 `8px`，只剩彩色小方块。
- 应用图标和文件类型图标的职责没有完全分开。

### 2.4 应用命名不面向普通用户

当前同时存在：

- `Excel 编辑器`
- `演示文稿查看器`
- `文档查看器`
- `PDF 查看器`
- `Media Intelligence`
- `文档开放接口`
- 多个 Parser、Generator、ASR 技术模块

“查看器”和“编辑器”是实现方式，不应成为产品名称。标准拼写为 `Excel`，不是 `execl`。

### 2.5 Office 文件处理被拆成多个产品

当前桌面文件关联会直接选择：

| 后缀 | 当前处理器 | 当前用户看到的产品 |
|---|---|---|
| `doc/docx` | `doc-viewer` | 文档查看器 |
| `xls/xlsx/csv` | `excel-engine` | Excel 编辑器 |
| `ppt/pptx` | `ppt-viewer` | 演示文稿查看器 |
| `pdf` | `pdf-viewer` | PDF 查看器 |

同时 `docs-open` 后端还维护另一套后缀到处理器的映射，文件关联存在两个事实源。

`mode: edit` 目前也只根据 manifest 的 `editable_formats` 推断，没有完整结合用户权限、文件可写状态、旧格式兼容性和处理器真实能力。

### 2.6 应用内部视觉没有统一合同

主要混乱：

- 主强调色包括 `#0a84ff`、`#0086a8`、`#2395bc`、`#409eff`、`#2563eb` 等多套蓝色。
- 侧栏宽度存在 `160/240/260/280px` 多种标准。
- 圆角存在 `3/4/6/8/10/12/14px` 多种标准。
- 字号从 `9px` 到 `30px` 任意分布。
- 窗口标题栏已显示应用名，内部页面又重复显示一次品牌标题。
- Element Plus Dialog、自制 fixed 遮罩、动态 DOM Toast 同时存在。
- Emoji、内联 SVG、Lucide、Element Plus 图标仍混用。
- 空态、加载、错误和只读状态没有共享组件。

## 3. 目标产品模型

### 3.1 产品应用与内部处理器分离

新增两个概念：

```ts
type ProductAppKey = 'office' | 'files' | 'images' | 'text' | 'agent' | 'knowledge' | string
type HandlerAppKey = 'excel-engine' | 'ppt-viewer' | 'pdf-viewer' | 'doc-viewer' | string
```

文件关联不再只返回一个 `appKey`，而是返回：

```ts
interface FileAssociationV2 {
  productAppKey: string
  handlerAppKey: string
  format: string
  requestedMode: 'view' | 'edit'
  grantedMode: 'view' | 'edit'
  readonlyReason?: 'permission' | 'legacy-format' | 'unsupported-write' | 'locked'
}
```

这样用户只看到“办公”，内部仍可调用现有模块。

### 3.2 统一文件打开协议

定义 `desktop.file.open/v2`：

```ts
interface DesktopFileOpenV2 {
  protocol: 'desktop.file.open/v2'
  fileId: number
  fileName: string
  format: string
  productAppKey: string
  handlerAppKey: string
  requestedMode: 'view' | 'edit'
  grantedMode: 'view' | 'edit'
  page?: number
  readonlyReason?: string
}
```

兼容要求：

- 继续接受旧 `{ fileId, fileName, format, mode, page? }`。
- 旧 payload 在 Office 入口归一化为 V2。
- 后端模块调用键保持原样。
- 历史窗口中的旧 `appKey` 在恢复时迁移到 `office + handlerAppKey`。
- Agent 工具和跨应用调用继续支持旧 key。

### 3.3 OfficeWorkspace

```text
OfficeWorkspace
├── OfficeChrome
│   ├── 文件名
│   ├── Word / Excel / PPT / PDF 类型标识
│   ├── 查看/编辑/只读状态
│   └── 保存、下载、缩放、搜索等上下文工具
├── DocumentAdapter      ─→ doc-viewer
├── SpreadsheetAdapter   ─→ excel-engine
├── PresentationAdapter  ─→ ppt-viewer
├── PdfAdapter           ─→ pdf-viewer
└── OfficeStatusBar
```

普通启动“办公”时显示最近文件、创建入口和 Word/Excel/PPT/PDF 类型入口；双击真实文件时直接进入对应适配器。

## 5. macOS 风格图标系统 V2

### 5.1 推荐采用混合方案

| 使用位置 | 方案 | 原因 |
|---|---|---|
| Launchpad、Dock、桌面主应用 | 生成或精制的位图母版 | 需要对象感、材质和层次 |
| Spotlight 32/34px | 简化 PNG 或 SVG 变体 | 兼顾识别和清晰度 |
| 窗口标题栏 16/20px | 独立简化 SVG | 不能直接压缩复杂大图 |
| 工具栏、菜单、命令 | Lucide | 适合作为功能符号，不作为主应用品牌图标 |
| 桌面真实文件 | 文件类型 SVG | 应表达后缀，不表达应用品牌 |

### 5.2 主应用图标规范

- 母版 `1024x1024`，导出 `256/128/64/48/32px`。
- 项目自有圆角轮廓，保留 macOS 式视觉节奏，但不复制系统原图。
- 每个应用一个明确核心对象，最多三层材质和三种主色。
- 光源统一来自左上，阴影方向和强度统一。
- 禁止“同一个线稿只换颜色”。
- 禁止全局强制顶部高光，材质与阴影写入单个资产。
- 灰度下仍应能依靠轮廓区分主要应用。
- 保存母版、导出文件、提示词和资产来源清单。

### 5.3 主应用图标对象建议

| 应用 | 核心对象 |
|---|---|
| 文件 | 立体文件夹与分页层次 |
| 办公 | 组合纸张/工作台，不使用 Office 商标 |
| 图片 | 相片、取景框或画布 |
| 文本 | 纸张与编辑笔 |
| AI 助手 | 对话核心与智能光点 |
| 知识库 | 打开的书与索引层 |
| 消息 | 双层对话气泡 |
| 抖音内容 | 内容画布与播放/时间线语义，不使用官方 Logo |
| 公众号写作 | 稿纸与书写工具，不使用微信 Logo |
| 媒体分析 | 画面、声波和分析取景框 |
| 模型路由 | 节点与路径调度 |
| 回收站 | 金属或磨砂废纸篓 |

### 5.4 文件类型图标

文件图标与 Office 应用图标分离：

- Word 文档
- Excel 表格
- PPT 演示
- PDF 文档
- 文本
- Markdown
- 图片
- 音频
- 视频
- 通用未知文件

真实文件始终根据后缀显示文件类型图标；打开后标题栏显示 Office 小图标与当前文件类型标识。

## 6. 应用内 WorkspaceShell

### 6.1 只做薄壳，不接管业务

```text
DesktopWindow
└── WorkspaceShell
    ├── WorkspaceSidebar（可选）
    ├── AppChrome
    │   ├── ContextTitle / FileName
    │   ├── Toolbar
    │   └── Tabs / SegmentedControl
    ├── WorkspaceContent
    ├── WorkspaceState（空态/加载/错误/只读）
    └── WorkspaceOverlay（Dialog/Popover/Toast）
```

WorkspaceShell 只统一：

- 字体与文本等级。
- 背景、分隔线和选中态。
- 侧栏和工具栏尺寸。
- 页面内间距与控件圆角。
- 空态、加载、错误和权限提示。
- 弹窗、Popover、Toast 的层级与按钮顺序。

业务模块继续拥有：

- AI 对话流和工具调用。
- Excel 表格和单元格交互。
- PDF/PPT/Word 渲染内容。
- 知识库图谱与治理能力。
- 抖音内容、计划和素材管理。
- 公众号写作流程。

### 6.2 建议视觉合同

| 项目 | 目标规格 |
|---|---|
| 正文 | `13px` |
| 辅助文字 | `11/12px` |
| 工作区标题 | `16/18px` |
| 工具栏 | `40px` |
| 普通控件 | `32px` |
| 主操作 | `36px` |
| 默认侧栏 | `240px` |
| 聊天侧栏 | 最多 `260px` |
| 控件圆角 | `6px` |
| 面板圆角 | `8px` |
| 间距等级 | `4/8/12/16/24px` |
| 强调色 | `#0a84ff`，业务可有一个辅助色 |
| 状态色 | 只表达成功、警告、错误和处理中 |

页面章节保持无边框布局；只有独立重复数据项、弹窗和真正工具面板使用卡片，避免窗口里再套整页白色大卡片。

## 7. 模块迁移优先级

### P0：先建立合同

1. 右键菜单单主高亮。
2. Dock 只直接悬停反馈。
3. Workspace Tokens、WorkspaceState、统一 Dialog/Toast。
4. 文件关联 V2 与 Office 产品/处理器分层。
5. 面向用户名称和 Launcher 可见性收口。

### P1：Office 与基础工作区

1. 文件管理作为工作区视觉基线。
2. PDF、PPT、Word 从 ViewerShell 迁入 OfficeWorkspace。
3. Excel 保留表格主体，替换旧 Windows 灰色渐变工具栏。
4. 文本和图片复用相同文件工具栏与状态组件。

### P2：聊天与运营工作区

1. AI 与消息共享侧栏、输入区、空态和弹层规格。
2. 抖音内容减少顶部重复标题和四层导航。
3. 模型路由移除整页大卡片和重复管理器标题。
4. 公众号写作替换 Element Plus 默认蓝和动态 DOM Toast。

### P3：复杂与待产品化模块

1. 知识库普通工作区迁移共享壳；3D 图谱保留独立沉浸模式。
2. 媒体分析从英文接口调试页改为正式中文工作台。
3. 清理剩余 Emoji、内联 SVG、硬编码颜色和重复弹层。

## 9. 兼容与风险

### 9.1 不能直接改 appKey

旧 key 被以下能力依赖：

- 文件关联和分类。
- 窗口会话恢复。
- Agent 工具和跨应用调用。
- Playwright 契约。
- `office-gen`、`docs-open` 和 Excel 导出链路。

因此使用别名和产品层映射，不做破坏性重命名。

### 9.2 查看与编辑权限必须显式区分

- Viewer 用户不能显示保存和编辑控件。
- `xls/doc/ppt` 等旧格式默认标记“兼容只读”。
- 文件锁定或后端不可写时明确显示只读原因。
- 前端模式只负责呈现，后端权限继续作为最终事实源。

### 9.3 弹层不能逃出所属窗口

应用内部 Dialog/Popover 默认相对当前窗口管理层定位，不允许普通模块使用覆盖整个桌面的任意 `position: fixed`。

### 9.4 图标资产必须可追溯

- 项目保存源文件和提示词。
- 不使用系统原图或品牌 Logo。
- 每个生成资产都需人工检查文本、边缘、透明通道和小尺寸效果。
