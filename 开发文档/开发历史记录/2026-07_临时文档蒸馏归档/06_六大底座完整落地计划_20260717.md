## 一、先回答你三个纠结

### 纠结1:要跑多少次 LLM / VLM

剩这批收尾(~350份活文档),三阶段调用机制**跟chunk数无关**(80个chunk的文档≠80次):

| 阶段 | 机制 | 次数 | 说明 |
|------|------|------|------|
| 融合 fusion | 每页1次 | ≈10,200次 | |
| 画像 profile | 每文档1次 | ≈340次 | |
| 图谱 graph | 每融合页1次 | **≈14,600次** | ★大头,占一多半 |
| **合计** | | **≈25,200次** | 约6500万token |
| VLM 看图 | — | **0次** | 这批全是文字文档,不需要看图 |

- **模型**:全用 deepseek-v4-flash(opencode go 订阅),订阅额度制,边际成本≈0。
- **耗时**:并发30下,预计 **10~14小时**(图谱单阶段重、超时3600s,是瓶颈)。
- **中转站GPT**:留给VLM看图用,这批没VLM,不动它。
- ⚠️ **要你确认**:opencode go 订阅实时余额从代码看不到,你去后台看下够不够跑2.5万次。

### 纠结2:强制还是可选

**可选补课,不是强制闸门。** 管线是每份文档独立跑的任务图,没有"全库必须跑完某阶段才开下一能力"的硬闸门。这350份是**最近10天的新导入**(7-07/7-09),只跑到解析层,不是老数据缺件。

不跑完的后果(只影响这350份自己):

| 功能 | 影响 |
|------|------|
| 向量/全文搜索能否命中 | ✅ 能(基础解析层照常召回) |
| 召回精度 | ⬇️ 用粗解析原文,非融合纠错正文 |
| 文档画像召回+加权 | ❌ 进不去 |
| 结构化事实/因果召回 | ❌ 进不去 |
| 知识图谱/实体检索/Agent查上下文 | ❌ 缺这批实体节点 |
| 跨文档关系关联 | ❌ 不建边 |
| 其他文档/库级功能 | ✅ 零影响 |

### 纠结3:落盘的是不是我们要的

**是好东西,值得跑。** 已跑完的14788份整体可信(实测抽样):
- 融合:自动提炼原文没有的标题+摘要,表格读成人话,图片连屏幕小字都读出来。
- 图谱:实体准(公司名/产品/成分),**每条关系都带原文证据**,技术噪音实体只占0.7%。
- 唯一瑕疵:约16%的**照片**因VLM认不出退化成像素垃圾——但那批照片已经跑完沉库了。

**剩这350份恰恰全是全库正文最厚的PDF/Word/PPT**(安全评估报告13万字/份、企业介绍、创业历程),几乎没图片,那个"照片认不出"的风险这里不存在。跑完是纯知识增量,不是重复劳动。

### 底座④ 模型网关收口 ★重点未做★
**现状**:施工方案写了"砍知识库那层多余熔断",但 `model_routing.py` 代码一行没删。`record_model_rate_limit`(数429)、`pause_model_stage_queue`(写paused_stages)、`model_auto_pause` 全在。今天那次卡死+要人工跑清熔断脚本,就是它——**下次上游一抖(429)数到30,又把整组阶段永久暂停,无TTL无自愈。** 网关自己有retry+fallback,知识库外面又套一层pause,两套逻辑对同一个429各判各的=打架。
**怎么做**:1)砍 model_routing.py 熔断逻辑(降级成纯记账,绝不写paused_stages),让网关当唯一决策者;保留选profile和并发限流。2)给paused_stages加TTL自愈。3)收编 generate_image(低优先)。4)embedding/rerank散调(model_services.py:65,122)只登记不强收。
**验收**:制造一次429,验证管线不再永久暂停、抖动恢复后自动继续、不用人工救火。
**依赖**:④是③的前置(④不修,③尾巴反复卡)。④无前置,可立即做。

### 底座⑤ PgBouncer 连接池
**现状**:进程真在跑(6432),配置全,但后端没连它——5432直连81条、6432走0条。`.env`没设DB_PORT(默认5432),且DB_POOL_SIZE=120还是旧大池。空转摆设,差最后改三行。
**怎么做**:1).env加 DB_PORT=6432。2)DB_POOL_SIZE/MAX_OVERFLOW砍小(10+5,真池子在pgbouncer)。3)executor子进程小池也指6432。4)★asyncpg+transaction模式坑:必须设 statement_cache_size=0,不然切了炸。5)重启复验。
**验收**:6432有连接、5432归零、24并发不再too many clients、asyncpg无报错。
**依赖**:无前置。中风险,低峰期切,留回退(改回DB_PORT=5432即恢复)。

### 底座① 三协议流式(95%)
差:ant工具流式的 input_json_delta 参数累积没确认上层认不认;4个历史红测试没清。做:确认tool_call_accumulator吃不吃ant这条→真调deepseek-anthropic+工具验证。

## 三、推荐执行顺序(一次只推一个到100%,别并开)

原则:先修正在流血的(④卡管线)→再补假装完成的(⑤空转)→最后收独立小尾巴。

1. **④ 模型网关收口**(半天~1天)★先做:唯一正在造成损害的,砍熔断层让网关当唯一决策者+暂停名单加自愈。④修好③自动受益,不用再救火。中风险,改前备份留回退。
2. **③ 语义层补尾巴**(半天,④修完顺手):888 pending补投+6硬错误看日志。④修好后基本自动化。← **就是这批2.5万次LLM**。
3. **⑤ PgBouncer接上**(半天,单独低峰窗口):改.env三行+验statement_cache_size=0+重启复验。中风险留回退。
4. **① 三协议流式收尾**(1~2小时):补ant工具流式+真调验证。低风险独立。
5. **② 召回阈值调优**(半天):等③数据齐,真实query标注反推阈值。
6. **⑥ 资源管理器前端**(等视觉方案,前端做):后端就绪,不阻塞。

**为什么不并开**:④⑤都动核心底座(检索管线/DB连接),同改出问题分不清谁引起、回退乱。各留回退,一个验收100%再动下一个。

## 四、等华哥拍板的决策点

1. **350份补课跑不跑、什么时候跑**:2.5万次LLM、10~14小时、deepseek边际成本≈0但要你确认opencode额度够不够。可选非强制,但值(全库最厚的PDF硬货)。
   - 可选加速:图谱占14600次(一多半),若嫌慢可先只跑融合+画像(约1万次,快很多),图谱单独排后面。
2. **执行顺序认不认**:建议 ④→③→⑤→①→②→⑥。
3. **④砍熔断层**:动核心检索管线,中风险,要你点头才动。
4. **⑤ embedding/rerank散调**:只登记不强收(它们没降级需求),认不认这个取舍。
