# 五大底座收口施工方案(2026-07-16)

> 病根:资源(模型/DB连接/系统资源/后台进程)没有统一收口,各模块各搞各的,底座千疮百孔,后续开发反复踩杂七杂八的问题。华哥定性:体力活为主,精细活不多。
> 施工原则:一个个收口落盘,每个改完 `kill -9 $(lsof -i :33000 -t)` 重启 + curl /api/health 验 + 真调验。改核心底座前备份、留回退。

## 共性根因
一次性/分散的进程各自抓全局资源:模型各降各的、连接各开各的、资源各读各的、后台进程放出去不登记。缺"统一底座 + 大家读/打同一个入口"。

---

## 底座① 系统资源(5秒缓存,毫秒读)【最先,已备料】

**现状(证据)**:
- `task_dispatcher.py:405 _resource_snapshot()` 每 tick 现调 psutil,GPU 直接填 `None/unknown`(没读)
- `model_watchdog/launcher.py` 11处 psutil(多是查单个模型进程,非整机)
- `system_status_service.py` 4处
- 三处各读各的,零缓存;GPU powermetrics 要 sudo 又秒级

**收口设计**:
- 新建 `backend/app/services/resource_monitor.py`:后台任务每 5 秒采样一次,落盘 `/tmp/华世_资源缓存.json`(跨进程共享)。CPU/内存用 psutil,GPU 用 **macmon**(已装 `~/.cargo/bin/macmon`,无需 sudo,毫秒级,`macmon pipe -s 1` 出 JSON)。
- 采样器挂在后端 lifespan(常驻)。dispatcher/status 读缓存文件(毫秒),不再现调 psutil/powermetrics。
- 缓存字段:cpu_percent / memory_percent / memory_available_mb / gpu_percent / gpu_memory_percent / 采样时间戳。读取方判断时间戳超 15 秒算过期降级。

**要改**:新建 resource_monitor.py;改 task_dispatcher.py `_resource_snapshot` 读缓存;main.py lifespan 起采样器。
**验证**:macmon 出 GPU 数;缓存文件 5 秒刷新;dispatcher 读到非 None 的 GPU。

## 底座② 模型全局网关(全局都走网关,不只知识库)

**现状(证据,子代理核实)**:
- 知识库各阶段其实已走 `gateway_router.chat`/`describe_image_detailed`,**无直连模型端口**(比预想好)
- 但知识库在网关外**又套一层熔断/限流/暂停**(`model_routing.py`):自建 rate_limit_auto_pause / model_auto_pause / pause_model_stage_queue
- **两个致命 bug**:①`entity_service.py:580`(重复在:227)对 str 调 .get 抛 `'str' object has no attribute 'get'`,被熔断层误判成"模型降级链耗尽"→暂停整条管线 ②暂停后**零恢复**(dispatcher 读暂停名单无 TTL/自愈)→ 一次抖动永久卡死(今天 219 卡死的元凶)
- 全局其他模块是否有直连:待另一子代理(直连普查)回来补

**华哥意图**:全局所有模型调用都走网关;调用方只报标识符(用途),网关按标识符决定 优先模型→降级→本地兜底;配置全集中网关,别处不管底层模型。

**收口设计**:
- 网关做**唯一**降级/容错决策者。砍掉知识库这层多余熔断/暂停(model_auto_pause/pause_model_stage_queue)。
- 修 `entity_service.py:580/227` 的 str.get bug(加类型防御:拿到 str 不当 dict)。
- 全仓直连点(待普查清单)收编进网关。
- 标识符路由:调用方传"用途标识"(如 knowledge_text/knowledge_vision),网关查 models.json 的 fallback_policies 决定链路。已禁 GPT 期间:文本→deepseek、视觉→本地 qwen3-vl。

**要改**:model_routing.py 砍熔断层;entity_service.py 修 bug;直连点收编(体力活,子代理批量)。
**风险**:改核心检索/管线,子代理出方案主控审再改。

## 底座③ DB连接池网关(快连快断+按类型均衡+打满300)

**现状(证据)**:
- `database.py` 模块级全局 engine,`DB_POOL_SIZE=120 + MAX_OVERFLOW=60`=每进程上限 180
- executor 是一次性子进程,每个 import database.py **各建 120 大池**,24并发爆 PG `max_connections=300`(too many clients,今天炸过)
- 治标已做:executor 检测 `--executor-task-id` 用小池(2+2);lane_limits llm 24→8、vision 40→12

**华哥意图**:像轮询器一样可无限堆进去,只是如何放行——按来源类型均衡,尽量打满 300;查询快连快断(<0.5s);出现长连接/长查询=SQL问题或知识库没总结好的信号。

**收口设计(治本)**:
- 上 **PgBouncer**(连接池网关):所有进程(后端/dispatcher/executor/脚本)连 PgBouncer,transaction 模式复用,对 PG 只保持少量真实连接(如 50)。彻底根治 too many clients。
- 按 lane 类型配额均衡(local_preprocess/vision/llm/relation 各给一档),总放行接近 300 上限但不超。
- 配套:审计长查询(>0.5s 的 SQL 揪出来,大概率缺索引或知识库召回没优化)。

**要改**:装 PgBouncer + 配 ini;DATABASE_URL 指向 PgBouncer 端口;executor 保持小池(经 PgBouncer 更稳)。
**风险**:要装东西,单独排期。治标已止血,不急。

**要改**:logging_middleware.py 加来源字段(一行);给个聚合脚本。**风险极低**。

## 底座⑤ 后台进程注册表(全盘进程管理器)

**现状(证据)**:只有队列 executor 记 `executor_pid`(任务表)。其他全黑户:模型服务(screen)、LibreOffice转换×3、沙盒、ASR/媒体/github子进程×5、watchdog screen、nohup脚本。**全仓无统一进程注册表**。找bug靠 ps grep 瞎翻。

**华哥意图**:放后台进程/线程要有地方记 PID/标识,方便落盘;类似网站内部全盘资源管理器。

**收口设计**:
- 新表 `framework_background_processes`:id / 标识符(语义名如"模型-qwen3vl"/"回填-fusion") / pid / 类型(queue/model/convert/sandbox/script/screen) / 命令 / 日志路径 / 启动时间 / 心跳 / 状态(running/exited/killed) / 退出码 / 备注
- 极简登记函数 `注册进程(标识符,pid,类型,日志路径,命令)`,每个 spawn 点加一行调用
- 注销:进程退出标 exited;或巡检对账(pid 没了标死)
- 查询:一个接口/页面列所有活进程+日志路径,找bug查一张表
- 和资源底座配对:资源底座=机器剩多少力,进程注册表=力被谁占了、日志在哪

**要改**:建表;写登记helper;各 spawn 点(约8处)加登记(体力活,子代理批量);加查询接口。
