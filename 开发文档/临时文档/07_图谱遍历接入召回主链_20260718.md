# 07_图谱遍历接入召回主链_20260718

## 阶段一：问题 → 方案 → 影响范围

### 问题（三审计交叉验证的核爆结论）
用户问"华世王镞有什么品牌"这类关系/列举型问题，当前架构**从设计上答不出**：
- 图谱建了 10.3万节点 / 17.2万边，但 `hybrid_search` 全链路**一行遍历代码都没有**。图谱边只在 `verify_and_score_results:2069` 被 count 成 `graph_edge_count` 当排序加分（0.35权重），那条"华世王镞→拥有→娇薇诗品牌"的边从没被走过。
- 三条召回路（keyword/vector/structured）全是文档/块粒度，命中主体后不摸下级。
- `get_graph_context`（entity_service.py:897）有边遍历能力，但只挂 HTTP 端点+agent 工具，不在主召回链。

实测（owner=4）：从"华世王镞"节点沿边一跳能摸到真下级——娇薇诗品牌/蔻诺品牌（下级品牌✓）、眼霜/防晒喷雾/舒颜莹润啫喱（产品✓）、云南华世王镞生物科技公司✓。数据是通的，缺的是"走边"的代码。

### 方案（增量加第五路，不推翻现有四路）
新增图谱扩展召回路，作为 RRF 融合的第五路。命中主体节点→沿边摸一跳下级实体→回填 document_id→进融合。

**新增/改动清单：**
1. **新函数 `graph_expansion_search(db, query_plan, owner_id, top_k)`**（search_service.py）
   - 取 `query_plan["core_entities"]` 的 name（高权前几个）
   - 按 `KbGraphNode.label IN (names)` + owner_id 查主体节点（复用建节点时 label=实体名）
   - 沿 `kb_graph_edges`（source/target 双向）摸邻居节点，取 relation 是关系型（拥有/属于/产生/包含）的边
   - 下级实体 entity_id 反查 `kb_chunk_entities` 回填 document_id/chunk_id（否则进不了 verify_and_score）
   - 结果 dict 对齐现有结构：`document_id/text/score/rank/source="graph_expansion"/index_layer="graph_expansion"/block_type="图谱扩展"`，text 用"主体 --关系--> 下级"拼串，keywords 放下级实体名
   - score 用边 weight 归一，带 owner_id 过滤，节点查询 limit 防爆

2. **新门控 `_should_graph_expand(query_plan)`**：有非空 core_entities（能锚主体）AND（intent 关系/列举型 OR answer_shape in {list,table,catalog} OR need_document_level_results）。

3. **hybrid_search 注入**（2311行后、keyword 前）：
   ```python
   graph_results = []
   if not fast_local_plan and _should_graph_expand(query_plan):
       graph_results = await graph_expansion_search(db, query_plan, owner_id, top_k=top_k*4)
   ```
   加 diagnostics.stage 观测。

4. **rrf_fusion 加第五路**：新增 `graph_results=None` 参数，result_groups 加 `("graph_expansion", graph_results or [])`，放在 structured 之后 keyword 之前。2335 调用处传参。

### 影响范围
- 只改 `search_service.py`（加函数+注入+融合参数），不动 entity_service/建图/管线。
- **纯增量**：门控不命中时（普通语义检索）走原路，零影响。只有关系/列举型问题多走一路。
- 零算力（纯 SQL 遍历，索引齐全：source_node_id/target_node_id/entity_id 都有 btree）。
- owner_id 全程过滤。节点/边查询 limit 防慢。

### 回退路径
- 改动集中在新增函数+一处注入+一个融合参数。回退=删 graph_expansion_search、删注入 3 行、删 rrf_fusion 的 graph 参数。
- 门控默认保守（只列举/关系型触发），出问题可把 `_should_graph_expand` 直接 return False 关掉整条路。

### 已知数据层缺陷（本轮先不动，记录待后续）
- 主体碎片：华世王镞205碎片/娇薇诗449碎片，规范节点仅1个。本轮遍历只查规范 label 命中的节点（够用），碎片归并是下一步 P0。
- 谓词乱：拥有/属于/产生混用同一种归属关系。本轮遍历时把这几种都纳入"关系型边"一起摸，规避。
- 苏蜜雅与华世王镞零边：数据本身缺，遍历摸不到，属数据补齐范畴，非本轮。
- 下级节点 type_id 多为空：本轮不按 type_id 硬过滤（会漏），按 relation + category 软过滤。

### 验收
- "华世王镞有什么品牌"能召回下级品牌节点（娇薇诗/蔻诺等）。
- 普通语义查询（如"烟酰胺的功效"）走原路，结果不变。
- 遍历路单次 <200ms（索引命中）。

## 阶段三：暴露的下一步问题（华哥2026-07-18追问）

### 问题A：按用户问的"类型"过滤返回（关键缺失）
现状：图谱遍历把主体的**所有关系边**都摸出来（商标/产品/技术/事件混一起）。用户问"有什么品牌"却返回第35类商标、CB技术——答非所问。
正解：问句要解析出**目标类型**（品牌/产品/商标…），遍历结果按目标类型过滤。
病根（实测）：华世王镞下级实体类型分布=其他231/事件170/产品名112/术语97/**品牌仅2**。娇薇诗(是品牌)被标"产品名"、蔻诺(是品牌)标"产品名"。**按类型过滤的算法能写，但语料的类型标注是错的/缺的**→这正是"定类"的真正用途：按类型召回的前置条件。

### 问题B：有必要向量原始数据吗 → 华哥定原则"过度冗余=污染，每节点尽量干净"
华哥决策：用不上的向量备份到外部→生产库删除。兜底本身就是污染源。

实测16141文档分布：
- 两层都有 2890篇：原始层向量=**纯冗余，可删**（融合层是干净源，删原始层召回只升不降）
- 只有融合层 10644篇：已无原始层，无需处理（大部分文档已干净单层）
- 只有原始层·fusion_status=done却无融合块 2203篇：**不能删（删了瞎）=状态造假欠债**（状态说融合好了实际没产出融合块，与"僵尸债/漏投"同类真bug）
- 只有原始层·pending/running 400篇：不能删（融合没跑完）

执行分两步（不能一刀切删）：
1. 能删先删：2890篇"两层都有"的原始层向量→备份外部→生产库删。
2. 不能删先补：2607篇"只有原始层"的先补跑融合（尤其2203篇状态造假），补完变"两层都有"再删原始层。
终态：全库只留融合层向量，原始层向量整个从生产库消失、备份在外。
⚠️ 删向量是破坏性操作，须华哥拍板后执行。
