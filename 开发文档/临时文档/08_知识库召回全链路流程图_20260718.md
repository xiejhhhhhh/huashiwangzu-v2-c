# 知识库召回全链路流程图（2026-07-18）

华哥版·全中文。基于三轮子代理审计 + 实测数据画的，不是印象。
目的：把整条链路看清楚，哪里脏、改后怎么走、每个节点干什么，缺什么补什么，别原地打圈。

---

## 一、总览：两条大链

```
【入库链】文件 → 一堆加工节点 → 存进各种表         （离线，慢工，一次做好）
                                    ↓
【召回链】用户提问 → 拆词 → 多路召回 → 融合排序 → 返回  （在线，要快、要准、要干净）
```

核心原则（华哥定）：
- 1-3 层原料（原始文本 / 图片VLM / 融合）是大头，不可再生，**保留**。
- 后面的产物（实体 / 图谱 / 词表）都是算法加工出来的，**脏了就删掉、改算法重跑**，不为保生产数据埋坑。
- 每个节点尽量干净，**过度冗余 = 污染**，是污染源就干掉。
- 用户问什么类型，就返回什么类型（问品牌返品牌，问商标返商标）。

## 二、入库链（11 个阶段，当前真实顺序）

```
① 源校验(source_validate)      检查文件能不能处理
        ↓
② 解析建索(parse_index)        把文件拆成文本块 → 存 kb_chunks(base_parse层)  ★原料·干净
        ↓
③ 原始取文(raw_text)           抽纯文本 → 存 kb_raw_data                      ★原料
        ↓
④ 页面渲染(page_render)        把页/图渲染成图片
        ↓
⑤ 图像OCR(raw_ocr)             图片识字 → kb_raw_data(round2)                ★原料
        ↓
⑥ 图像理解(raw_vision·VLM)     图片内容描述 → kb_raw_data(round3)            ★原料
        │   ⚠️ 这里产出整句"本页为XX品牌培训宣传海报"——描述句本身对(给检索用)，
        │      但成了下游抽实体的污染源
        ↓
⑦ 融合(fusion·LLM)             多来源合成一段干净正文 → kb_page_fusions       ★原料·干净·检索主力
        ↓
⑧ 画像(profile·LLM)            文档级摘要/标签 → 文档画像
        ↓
⑨ 认知索引(cognitive_index)    分词器切词 → kb_terms / kb_term_edges          ⚠️见下"死数据"
        ↓
⑩ 图谱抽取(graph·LLM) ★病根    从融合正文抽实体+关系 → kb_entity_dictionary / kb_graph_nodes / kb_graph_edges / kb_chunk_entities
        │   ❌ 提示词只说"全称优先"，没说"剥离修饰语只取主体"
        │      → LLM 把"华世王镞培训宣传海报"整句当实体名
        │      → 华世王镞被切成 205 个碎片，娇薇诗 449 个碎片
        │   ❌ 写库只精确同名匹配，无主体归并 → 碎片各自成节点
        │   ❌ 抽取时给的 category 不准 → 娇薇诗被标"产品名"而非"品牌"
        ↓
⑪ 关系构建(relations)          跨文档关系 → kb_file_relations / kb_doc_relations
```

**打齐层（semantic_align）**：挂在⑩里（抽完就地调），但设计上**只纠 OCR 单字错字**，不管整句碎片。碎片归并 = 无人负责。

**归并层（canonical/merge）**：❌ 根本没接进这 11 个阶段（STAGE_SEQUENCE 里没有）。canonical_id 使用率 0.1%。形同虚设。

## 三、召回链（当前·华哥问的"华世王镞有什么品牌"走不通）

```
用户提问「华世王镞有什么品牌」
        ↓
【拆词】plan_query / _match_query_entities
        ⚠️ 用 IDF(词频)算权重，对高频核心主体是反的：
           华世王镞(出现447篇)权重最低，碎片"华世王""王镞"(出现2篇)权重反高
        ↓
【四路召回】（全是"文档/文本块"粒度，命中主体后不摸下级）
   ├─ 关键词(keyword)        文本里 LIKE 匹配
   ├─ 向量(vector)           语义相似的文本块
   ├─ 文档画像(document)     报告/名单类
   └─ 结构化信号(structured) 证据锚定
        ↓
【融合】rrf_fusion 合并排序
        ↓
【验证打分】verify_and_score
        ❌ 图谱的 17 万条边，只在这里被"数了几条"当加分项(0.35权重)
           那条"华世王镞→拥有→娇薇诗"的边，从来没被真正走过
        ↓
【重排+阈值】rerank 精修
        ↓
返回：一堆"提到华世王镞的文本块"，答不出"有哪 5 个品牌"
```

**断点**：命中"华世王镞"后，没有任何代码去顺着图谱边摸出下级品牌。图谱建了不查 = 华哥说的"一坨"。

## 四、目标链路（改后·干净高效版）

### 入库链改动（治本，从源头产出干净数据）

```
⑩ 图谱抽取（改提示词 + 换 grok-4.5）
   ✅ 提示词加"实体名只保留主体，剥离'海报/宣传/培训/照片'等修饰语"
      → "华世王镞培训宣传海报" 抽成：实体=华世王镞(品牌) + 事件=培训
   ✅ 抽取直接用 18 类语义体系分类（成分/原料/功效/品类/产品/品牌/系列/规格/
      肤质/人物/组织/地点/事件/时间/技术标准/视觉素材/营销内容/噪音）
      → 娇薇诗直接标"品牌"，抽取+分类一次完成，省掉二次定类 9 万条
   ✅ 写库做主体归并（不只精确匹配）：华世王镞的变体合回一个规范节点
        ↓
   新增【归并阶段】接进管线（补 canonical/merge）
   ✅ 存量碎片跑一遍归并，205 碎片 → 1 个规范主体（这里才用 LLM 判"哪些是同一主体"）
```

### 召回链改动（把图谱接进来 + 按类型过滤）

```
用户提问「华世王镞有什么品牌」
        ↓
【拆词】改进主体选择：滤掉疑问词/通名(品牌/产品)，专名优先，图谱节点表当过滤器
        ↓ 拆出：主体=华世王镞、目标类型=品牌
        ↓
【五路召回】原四路 + ✅新增图谱扩展路(graph_expansion)
   └─ 图谱扩展：命中"华世王镞"节点 → 沿边摸一跳下级 → ✅按目标类型过滤(只留品牌)
        ↓
【融合】图谱结果进 RRF，关系型问题排前
        ↓
返回：娇薇诗、蔻诺、俏小喵…（干净的品牌列表）
```

**关键新增**：问句解析出"目标类型" → 图谱遍历结果按类型筛。
问"有什么品牌"只返品牌，问"有什么商标"只返商标 —— **用户问题决定返回类型**。

## 五、要清理的污染源（华哥"过度冗余=污染"原则）

| 对象 | 判定 | 处置 |
|---|---|---|
| kb_raw_data / base_parse / 融合层 | 原料·大头·不可再生 | **保留** |
| kb_entity_dictionary（含碎片） | 脏产物·可重生 | 删，改算法重跑 |
| kb_graph_nodes/edges（205碎片·一实体14节点·谓词乱） | 脏产物·可重生 | 删，改算法重跑 |
| kb_chunk_entities（挂在脏实体上） | 脏产物 | 删，跟着重跑 |
| kb_terms 那套（570万行·主检索链根本不查） | 死数据·纯冗余 | 待拍板：删 |
| "两层都有"的2890篇 base_parse 向量 | 融合层已覆盖·冗余兜底 | 待拍板：备份+删 |

## 七、执行顺序（先验证再放量，别闷头烧）

```
1. 改抽取提示词（剥离修饰语+18类分类+从描述句抠主体）

5. 全库重跑抽取（≈2.15万次，1.5-3小时）
6. 新增归并阶段，存量碎片归并
7. 召回链：图谱遍历已接入（今天做完），补"按目标类型过滤"
8. 清冗余：原始层向量、kb_terms（拍板后）
9. 全链路验证："华世王镞有什么品牌"能干净返回品牌列表
```
