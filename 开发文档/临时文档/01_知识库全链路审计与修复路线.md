## 四、知识库可投用性 + 剩余节点(问题2)

### 能用的部分
- 图片(jpg/png/jpeg)/表格(xls/xlsx)/doc/txt 基本 100% 洗到语义层
- 真正影响召回的实体(被 chunk 引用+含汉字≥2)已 100% align done
- 检索"快"的底子好:向量走 HNSW 索引(qwen3-8b,子向量2000维粗召回+完整4096维精排),全文走 trgm GIN 索引,库侧召回不暴力全表
- 数据喂得饱:chunks 55万+、向量覆盖 99.95%、kb_evidence 26万、kb_graph_edges 16.8万、kb_fact_candidates 5.9万、kb_causal_candidates 8146,全满

### 硬伤(现在还不算"干净可投用")
**① 数据没洗干净(核心)**
- 缺语义层文档:pptx 101/107(94%)、ppt 52/69(75%)、pdf 424/1140(37%)、docx 175/883(20%),合计约 760 文档只有原始文本层无实体/图谱
- 元凶:`kb_pipeline_runs` running=9038 僵尸记录 + degraded=6011,文档卡在"运行中"再没推进(远超并发上限)
- 断链位置:`pipeline_service.py:557` `_ready_for_fusion` 门槛 + fusion→graph 边

**② 召回不精(华哥痛点"枚举一堆分不清有用没用")**
- 全链路无相关性分数阈值,恒定凑满 top_k=10 返回(低分噪音照返)——search_service.py:1415/1927/2365
- rerank 精排硬编码关闭(router.py:318 use_rerank=False),装了 bge-reranker-v2-m3 却不用
- 昨天打齐的别名表下游检索没读:`_match_query_entities` 只查正名,keyword/vector 通道零感知,只 structured_signal 一处用;别名覆盖率仅 2.8%(2926/103490)→ 打齐成果没转化成召回增益
- 普通查询不按文档去重(rrf dedupe_by_document 默认 False),同文档多块刷屏 top_k
- 返回块 text 截 500 字,无相邻块拼接(仅 page_fusion 提供整页上下文);溯源链完整(文件名/页码/section)

**③ agent 工具调用慢(不缺数据,是这三个)**
- Qwen3-Embedding-8B 冷启动:闲置被 reap 后首次 search 拉起 8B 权重几十秒(最痛)
- search 不内联关系卡:问"关系/依据"要额外调 get_graph_context/get_evidence_detail,2-3 次往返
- agent 元工具调用占 54.8%(skill_use/list/describe,找工具而非干活)——dp4 查到

**④ 其他债**
- 422 失败任务债(340 kb_enterprise_import_file + 57 kb_pipeline_stage)
- profile_evolve 持续失败(dict 塞 text 列,profile_evolve.py:312),用户画像演进从没成功,6条全空——dp4 查到
- kb_document_profile_vectors 仅 28% 覆盖(4328/15628),影响跨文档关系算全
- 治理候选 kb_governance_candidates pending 21.2万;实体 confirmed 仅 2242 vs candidate 8.98万,确认闭环没跑
- diagnose 工具库名误报(huashiwangzu_v2 vs 华世王镞_v2);model-router README 缺失(docs_audit BLOCKER)
