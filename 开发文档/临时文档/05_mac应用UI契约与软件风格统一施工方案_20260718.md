# mac 应用 UI 契约与软件风格统一施工方案

> 日期：2026-07-18
> 状态：**切片 1–6 全部完成（用法已蒸馏到主文档；Win11 完整壳仍不动）**
> 基线：`http://127.0.0.1:5173/desktop`
> 范围：只做 **macOS 默认皮肤**；Win11 完整壳另立阶段
> 不做：锁屏 / 开机动画
> 原则：
> 1. **后端只负责逻辑与接口**
> 2. **前端只走统一 UI 契约底座**
> 3. **快捷键有就好，不强制抢浏览器/系统快捷键**
> 4. 新增软件必须接契约，禁止各写一套后台皮肤

---

## 0. 一页结论

### 0.1 问题

桌面壳已经像 OS，但打开软件后常常像“不同后台系统拼在一起”：

- 访达（文件管理）仍偏管理系统
- Office / Agent / 知识库入口擦了玻璃，内部布局语言不统一
- `AppWindowFrame` 已有，但**几乎没被业务强制使用**
- 产品有 `product.json`，缺 **UI 布局契约** 字段与验收
- 菜单上写了 ⌘W / ⌘Space，实现层却 `preventDefault` 抢键，Web 场景不合适

### 0.2 目标

```text
桌面壳（已完成的 mac chrome）
  + 应用 UI 契约（HIG-lite）
  + 首批主软件按契约改造（访达优先）
  + 快捷键：可选、可关、默认不抢系统键
  = 后续无论新增多少软件，都长得像同一套 mac 软件
```

### 0.3 前后端边界（必须干净）

| 层 | 负责 | 不负责 |
|---|---|---|
| **后端 / modules backend** | API、权限、数据、任务、capability | 颜色、布局、组件皮肤、Element 主题 |
| **Product manifest** | 产品身份、打开策略、文件关联、窗口策略、**声明 UI 布局类型** | 具体 CSS |
| **前端 desktop kit** | 窗口框、工具条、侧栏、列表、表单、空态、反馈 | 业务数据规则 |
| **业务 frontend** | 组合 kit + 调 API | 私自发明第三套灰白后台皮肤 |

一句话：
**后端出能力，前端出脸；脸必须来自 kit。**

## 1. 现状审计（执行依据）

### 1.1 已有、可复用

| 能力 | 位置 | 说明 |
|---|---|---|
| 桌面壳 mac | `frontend/src/desktop/**` | MenuBar/Dock/LP/Spotlight/CC/通知/App Switcher |
| 皮肤底座 | `desktop/skins/*` | mac 默认完成；win11 仅插槽 |
| 窗口外框 | `desktop-window-frame.vue` | 交通灯/拖拽/最小化 |
| 应用内框 | `components/app-window-frame.vue` | toolbar/sidebar/content/statusbar slots |
| 工具条/状态栏 | `app-toolbar.vue` / `app-status-bar.vue` | 已有 glass 方向 |
| 桌面反馈 | `desktop-feedback.ts` + toast/dialog host | 系统级提示 |
| Product 清单 | `products/*/product.json` | 有 workspaceKind，但未驱动 UI |

### 1.2 关键缺口

1. **无强制应用 UI 契约**
   业务可直接裸写页面，kit 可选不选。
2. **访达未产品级 HIG**
   `platform/components/apps/desktop/**` 仍是文件管理器后台味。
3. **主应用未统一接入 AppWindowFrame**
   搜索显示业务侧几乎不引用 `AppWindowFrame`。
4. **反馈双轨**
   桌面路径已 desktopMessage，Office 等仍 Element Message。
5. **快捷键策略错误**
   全局 capture + preventDefault，会抢浏览器键；Web 产品应默认不抢。
6. **Settings 产品壳弱**
   有 product，缺完整 mac 设置外观与桌面偏好页。
7. **新增软件无门禁**
   无“必须用 kit / 禁止私有后台皮肤”的检查清单与测试。

## 2. 目标架构

### 2.1 分层

```text
┌─────────────────────────────────────────────┐
│  Desktop Shell (mac skin)                   │  已完成主体
│  MenuBar / Dock / Window chrome / overlays  │
└─────────────────────────────────────────────┘
┌─────────────────────────────────────────────┐
│  App UI Kit (HIG-lite contract)             │  本方案核心新增
│  AppShell / Toolbar / Sidebar / List / ...  │
└─────────────────────────────────────────────┘
┌─────────────────────────────────────────────┐
│  Product Frontends                          │  只组合 kit + API
│  files / office / knowledge / ai / settings │
└─────────────────────────────────────────────┘
┌─────────────────────────────────────────────┐
│  Backend / modules                          │  纯逻辑与接口
│  routers / services / capabilities          │
└─────────────────────────────────────────────┘
```

### 2.2 契约对象

#### A. Product UI 声明（写在 product.json，可演进）

在现有 `ProductManifestV1` 上**增量**（不破坏旧字段）：

```json
"uiContract": {
  "kit": "mac-app-v1",
  "layout": "finder | document | chat | settings | dashboard | utility",
  "shell": {
    "useAppWindowFrame": true,
    "sidebar": "optional | required | none",
    "toolbar": "required | optional | none",
    "statusbar": "optional | none"
  },
  "feedback": "desktop-kit",
  "density": "comfortable"
}
```

规则：

- 新 Product **必填** `uiContract.kit = mac-app-v1`
- `useAppWindowFrame=true` 的产品，入口组件必须使用 App UI Kit 壳
- 后端 **不解析样式**，只存/透传 manifest；前端 loader/扫描器读它做校验与默认布局

#### B. 前端 App UI Kit（实现合同）

建议目录（新建，不打散现有 desktop）：

```text
frontend/src/desktop/app-kit/
  README.md                 # 给业务的使用说明
  types.ts                  # layout / density / feedback 枚举
  mac-app-shell.vue         # 统一应用壳（包装 AppWindowFrame）
  mac-toolbar.vue           # 标准工具条（左导航/中标题或路径/右操作）
  mac-sidebar.vue           # 标准侧栏
  mac-list.vue              # 列表/表格基础态
  mac-empty-state.vue
  mac-inline-banner.vue
  mac-form-section.vue
  tokens-app.css            # 应用内 token（基于 shell tokens）
  use-app-feedback.ts       # 统一 desktopMessage，禁止业务直引 ElMessage
```

现有 `app-window-frame` / `app-toolbar` / `app-status-bar` **升级并入 kit**（可 re-export，避免一次大搬家）。

#### C. 后端契约边界

后端 / module backend **允许改**：

- capability 参数与返回结构（若 UI 需要更干净的 DTO）
- product catalog 透传 `uiContract`
- 文档与 manifest schema 校验

后端 **禁止**：

- 下发颜色/CSS/组件名当业务逻辑
- 为某个前端皮肤写分支逻辑

模块 `manifest.json` 继续描述 API/权限；**脸在 Product UI + App Kit**。

## 3. mac 应用 HIG-lite（统一长相）

不抄苹果私有资源，只统一“系统软件味道”。

### 3.1 布局原型

| layout | 用于 | 结构 |
|---|---|---|
| `finder` | 文件/资源浏览 | 顶工具条 + 左源列表 + 主列表/网格 + 底状态栏 |
| `document` | Office/文本/内容 | 顶工具条 + 可选侧栏 + 文档画布 |
| `chat` | AI | 左会话列表 + 顶栏 + 消息区 + 输入坞 |
| `settings` | 设置 | 左分类 + 右表单分组 |
| `dashboard` | 概览/运营 | 顶栏 + 卡片栅格 |
| `utility` | 轻工具 | 顶栏 + 单栏内容 |

### 3.2 视觉铁律

1. **材质**：工具条/侧栏只吃 `--desktop-lg-*` / app-kit token，禁止业务硬编码 `#fff/#e5e7eb` 当系统灰
2. **圆角**：控件 6–8，卡片 10–12，面板跟随 shell
3. **字号**：标题 13–15 semibold，正文 13，说明 11–12 次级色
4. **间距**：8 的倍数；工具条高度 40–44
5. **侧栏**：浅分隔 + 选中用系统蓝透明底，不用厚重后台选中条
6. **反馈**：只用 `desktopMessage` / app-kit banner；**禁止**业务页 `ElMessage`（迁移名单见切片）
7. **图标**：Lucide + AppIcon；同尺寸同描边节奏
8. **空态/加载/错误**：必须用 kit 组件，文案可业务化

### 3.3 访达专项（第一优先软件）

把 `apps/desktop`（files）收成 mac Finder 气质，而不是“网页文件管理器”：

| 区域 | 目标 |
|---|---|
| 工具条 | 返回/前进/上移为紧凑按钮组；路径段 pill；搜索框右侧 |
| 侧栏 | 源列表风格（个人/位置/废纸篓），选中浅蓝 |
| 列表 | 行高舒适、列头轻、选中态统一 |
| 状态栏 | 计数 + 视图切换，轻材质 |
| 属性 | 系统小面板，不走 Element 重对话框味 |

后端仍用现有 files API；**只改前端组合与样式合同**。若列表字段不够展示，再开后端 DTO 增量（可选，不阻塞 UI）。

## 4. 快捷键策略（Web 友好）

### 4.1 原则

1. **默认不抢浏览器/系统快捷键**
2. 菜单可显示建议键位，但实现必须可关
3. 只在“桌面壳聚焦且输入框未聚焦”时处理
4. 与浏览器冲突的组合：**默认不 `preventDefault`**
5. 需要增强时，用户在设置里显式打开“启用桌面快捷键”

### 4.2 分级

| 级别 | 示例 | 默认 |
|---|---|---|
| **Safe** | 仅壳内按钮、菜单点击、应用内自定义（不含 ⌘/Ctrl 系统常用） | 开 |
| **Opt-in Desktop** | ⌘/Ctrl+S 保存（仅文档 app 内）、应用内 `Ctrl+K` 命令板 | 可开 |
| **Never capture by default** | ⌘/Ctrl+W、⌘/Ctrl+T、⌘/Ctrl+N、⌘/Ctrl+R、⌘/Ctrl+Tab、⌘/Ctrl+Space、⌘Q 等 | **禁止默认抢占** |

### 4.3 当前代码改造点

- `shell/index.vue`：`handleGlobalShortcut` 去掉默认 `preventDefault` 抢占
- 现有 App Switcher：保留 **公开 API** + 按钮入口；快捷键改为 opt-in 或改用 **不冲突组合**（如 ``Ctrl+Shift+` `` 仅在设置开启时）
- Spotlight：菜单/Dock/状态栏入口为主；⌘Space **默认不拦截**
- 菜单上的 `<kbd>`：改为“建议”或设置开启后才显示可用

配置建议（进 desktopConfig）：

```ts
enableDesktopHotkeys: false  // 默认关
```

## 6. 后端/模块若必须动，怎么动

仅以下情况动后端：

| 场景 | 动法 |
|---|---|
| catalog 要带 uiContract | product_catalog 透传 JSON 字段 |
| 列表缺显示字段 | 只读 DTO 增量，不改业务语义 |
| 设置项持久化 | 走现有 desktop state / user prefs，不新建皮肤后端 |

模块开发规范补充一句：

> 模块 backend 禁止依赖前端皮肤；模块 frontend 必须使用 `mac-app-v1` kit（桌面内运行时）。

## 7. 新软件准入清单（mac-app-v1）

新 Product / 桌面应用合并前必须满足：

- [ ] `product.json` 含 `uiContract.kit = mac-app-v1`
- [ ] 入口使用 `MacAppShell`（或官方 layout 包装）
- [ ] 无业务私有整页灰白后台皮肤
- [ ] 反馈走 `useAppFeedback` / desktopMessage
- [ ] 不默认 `preventDefault` 浏览器快捷键
- [ ] 空态/加载/错误用 kit
- [ ] 图标走 AppIcon catalog
- [ ] 不在 backend 写 UI 样式分支

## 8. 风险与默认决策

| 点 | 默认 |
|---|---|
| 是否保留 ⌘Space 开 Spotlight | **默认否**（仅按钮/菜单）；设置可开 |
| App Switcher 快捷键 | **默认否**；保留 API + 可选 Ctrl+Shift+` |
| Element Plus | 业务复杂表格可继续用 **EP 组件**，但 **容器/反馈/布局必须 kit**；禁止 EP Message 当系统反馈 |
| 一次改所有模块 | 否；先 files → office → knowledge → agent → settings |
| Win11 | 不动完整壳；kit 设计时避免写死仅 mac 的 DOM 假设（layout 名中性） |
