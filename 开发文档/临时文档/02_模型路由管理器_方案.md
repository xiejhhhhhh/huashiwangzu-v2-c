# 模型路由管理器 - 实现计划

## 概述

新增一个独立模块 `model-router`，在桌面有独立图标，管理员打开后可以：
1. 查看所有"模型调用节点"当前配置
2. 修改每个节点的模型/提供商/参数/降级链
3. 管理提供商（新增/编辑/删除/测试连通性）
4. 保存后热重载网关配置（不需重启后端）

## 架构

```
前端（Vue 独立模块页面）
    ↓ HTTP API
后端 /api/model-router/*（新 router）
    ↓ 读写
backend/data/config/models.json（单一数据源）
    ↓ 热重载
gateway/config.py 重新加载缓存
```

核心原则：**models.json 仍是唯一数据源**，不引入新 DB 表。后端 API 只是对这个文件做结构化读写 + 校验 + 热重载。

## 调用节点定义

每个"节点"是一个在系统中实际消耗模型的调用点。从 models.json 提取出来的节点清单：

| 节点 ID | 节点名称 | 对应配置路径 |
|---------|---------|------------|
| `agent_chat` | Agent 对话 | model_types.llm.profiles[deepseek-v4-flash] |
| `agent_planning` | Agent 规划（结构化输出） | 同上 |
| `knowledge_text` | 知识库-文本分析 | module_routing.knowledge.default_profile → profiles[X] |
| `knowledge_vision` | 知识库-视觉分析 | module_routing.knowledge.default_vision_profile → profiles[X] |
| `knowledge_ocr` | 知识库-OCR | module_routing.knowledge.stages.raw_ocr |
| `knowledge_fusion` | 知识库-融合 | module_routing.knowledge.stages.fusion |
| `knowledge_profile` | 知识库-摘要画像 | module_routing.knowledge.stages.profile |
| `knowledge_graph` | 知识库-图谱 | module_routing.knowledge.stages.graph（暂无，同 default） |
| `image_gen` | 生图 | model_types.image_gen.profiles[primary] |
| `vision` | 通用视觉 | model_types.vision.profiles[primary] |
| `embedding` | 向量嵌入 | model_types.embedding.profiles[primary] |
| `rerank` | 重排序 | model_types.rerank.profiles[primary] |

## 后端 API 设计

### GET /api/model-router/nodes
返回所有节点当前状态（含实时健康状态）

### PUT /api/model-router/nodes/{node_id}
修改某节点的模型配置（profile_key、温度、预算、降级链等）

### GET /api/model-router/providers
返回所有提供商列表

### POST /api/model-router/providers
新增提供商

### PUT /api/model-router/providers/{provider_key}
修改提供商（URL、密钥环境变量、超时等）

### DELETE /api/model-router/providers/{provider_key}
删除提供商（校验无节点引用）

### POST /api/model-router/providers/{provider_key}/test
测试提供商连通性（发一个 ping 请求）

### GET /api/model-router/profiles
返回所有 profile 列表（按 model_type 分组）

### POST /api/model-router/profiles
新增 profile

### PUT /api/model-router/profiles/{profile_key}
修改 profile

### POST /api/model-router/reload
热重载网关配置（清除 _CONFIG 缓存，重新读 models.json）

### GET /api/model-router/usage-summary
返回各节点最近的调用量/费用/错误率摘要

## 前端页面结构

```
模型路由管理器
├── 顶部状态栏（网关状态 + 最近错误数 + 重载按钮）
├── 标签页切换
│   ├── 调用节点（主页，按分组展示卡片）
│   │   ├── Agent 组
│   │   ├── 知识库组
│   │   └── 工具组（生图/嵌入/重排/视觉）
│   ├── 提供商管理
│   │   ├── 列表（名称/类型/URL/状态灯）
│   │   └── 新增/编辑弹窗
│   └── 模型档案（profiles 全量列表）
└── 底部操作栏（保存 + 重载 + 测试全部连通）
```

## 每个节点卡片包含的字段

- **当前 profile**（下拉选择，只显示同 model_type 的 profiles）
- **模型名**（只读，跟随 profile）
- **提供商**（只读，跟随 profile）
- **温度**（可编辑滑块）
- **最大输出 token**（可编辑）
- **上下文预算**（可编辑）
- **降级链**（可排序列表，可添加/移除 profile）
- **状态灯**（绿/黄/红，来自 /health 探针）
- **最近 24h 调用量**（来自 usage 表）

## 热重载机制

`gateway/config.py` 里的 `_CONFIG` 是模块级全局变量。添加一个 `reload_config()` 函数：
1. 将 `_CONFIG` 设为 None
2. 重新调用 `_load_models_config()`
3. 刷新 `MODEL_PROFILES` 和 `DEFAULT_MODEL`
4. 返回新配置摘要

这样无需重启进程，管理员点"保存"后自动热刷新。

## 安全设计

- 所有写操作要求 admin 角色
- 密钥不明文返回前端（只返回 env 变量名，如 GPTSTORE_API_KEY）
- 保存前校验 JSON 结构合法性
- 保存时自动备份当前 models.json 到 backend/backups/

## 开发分批

- **批 1**：后端 router + 热重载机制 + 读接口
- **批 2**：前端主页面 + 节点卡片 + 保存
- **批 3**：提供商管理 + 测试连通
- **批 4**：用量统计 + profile 管理
