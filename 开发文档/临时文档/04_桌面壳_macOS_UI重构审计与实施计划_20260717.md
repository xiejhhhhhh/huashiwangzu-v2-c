# 桌面壳 macOS UI 重构审计与实施计划

> 日期：2026-07-17  
> 状态：已实施并通过验收（2026-07-17）
> 对标参考：[macOS 27 Demo](https://macos27.kimi.page/)  
> 本地基线：`http://127.0.0.1:5173/desktop`  
> 核心原则：保留正式项目的业务底座，替换 OS Chrome（操作系统外壳），不迁移 Vue 技术栈，不复制 Demo 业务逻辑。

## 0. 一页简述

### 0.1 这次到底改什么

准确地说，这不是把网站接入“原生 macOS 系统”，而是在现有 Vue Web 应用里统一实现 macOS 的外壳、操作语义和视觉规格：

```text
当前状态

正式业务底座
  + Windows 开始菜单/任务栏/托盘/贴靠
  + macOS 菜单栏/交通灯/玻璃外形
  + Web 管理系统按钮/Emoji/Element Plus 弹层
                         │
                         ▼
                  视觉和操作语义混乱


目标状态

保留现有正式业务底座
  ├── 应用注册、权限、文件关联
  ├── 窗口状态、会话恢复、多实例
  ├── 文件操作、拖放、剪贴板
  ├── 通知数据、命令搜索
  └── 对外调用合同
                         │
                         ▼
建立唯一 macOS Shell Contract
  ├── DesktopInsets：统一菜单栏、Dock、窗口安全区
  ├── Design Tokens：统一字体、材质、圆角、阴影、动效
  ├── MacWindowFrame：交通灯、标题栏、最小化/恢复/缩放
  ├── MacMenuBar：应用菜单、系统状态、通知、账户、时间
  ├── MacDock：纯应用图标、运行状态、徽标、进度
  ├── Spotlight + Launchpad：替换 Windows 开始菜单
  ├── MacMenu + Popover：替换零散右键菜单和系统弹层
  └── macOS Desktop Icons：右上排列、选择、拖拽反馈
                         │
                         ▼
                统一的纯 macOS 桌面外壳
```

### 0.2 落点是否完整

方案落点已经完成，包含：菜单栏、Dock、窗口、桌面图标、Spotlight、Launchpad、右键菜单、通知中心、系统状态、账户入口、加载态、响应式降级、可访问性、动画、统一应用图标和视觉回归。

当前 `/desktop` 已统一为 macOS Web 桌面壳；现有 Vue 业务底座、应用注册、权限、文件操作、窗口状态和公开接口保持不变。原 Windows Start Menu、窗口长条 Taskbar、Tray、显示桌面细条和重复系统入口已从主交互路径移除。

### 0.3 能否一次性改成参考例子

可以按一个完整版本一次性交付到接近参考 Demo 的一致程度，但代码内部不能一次性无边界重写。推荐方式是“一个总交付，五个连续切片”：

```text
切片 A：测试基线 + DesktopInsets + 状态问题修复
   ↓
切片 B：MacWindowFrame + 统一 Design Tokens
   ↓
切片 C：MacMenuBar + MacDock，删除 Windows Taskbar 语义
   ↓
切片 D：Spotlight + Launchpad + Menu + Notification Center
   ↓
切片 E：桌面图标/拖拽收口 + 像素校准 + 全量回归
   ↓
一次性作为“macOS 桌面壳重构版本”验收和交付
```

这样用户最终看到的是一次完整换壳，工程内部则每一步都能验证，不会把现有窗口、文件和权限业务一起推倒。

### 0.4 最终界面职责

```text
MenuBar       只负责：当前应用菜单、系统状态、通知、账户、时间
Dock          只负责：启动应用、激活应用、运行状态、徽标、进度
WindowFrame   只负责：窗口控制、标题、工具区和内容承载
Spotlight     只负责：搜索应用、文件和命令
Launchpad     只负责：浏览和启动全部应用
Desktop       只负责：文件/应用图标、选择、右键和拖放
```

同一个状态只在一个区域表达：底部不再重复时钟和通知，Dock 不再显示窗口长标题，菜单栏不再放没有行为的假菜单。

## 1. 结论

当前桌面壳的主要问题不是功能不可用，也不是玻璃效果不够，而是同时存在三套界面语义：

1. Windows：开始菜单、任务栏窗口长条、托盘时钟、显示桌面细条、窗口预览、边缘贴靠。
2. macOS：顶部菜单栏、交通灯窗口按钮、悬浮玻璃 Dock 外形。
3. Web 管理系统：Element Plus 按钮、Emoji 图标、卡片式弹层、深色侧栏。

这些元素单独看都能工作，组合后却没有统一的空间、材质、尺寸、状态和动效合同，因此视觉越加效果越显凌乱。

本次重构不应推倒现有窗口管理和业务能力。正确路线是：

- 保留应用注册、权限、窗口状态、会话恢复、文件操作、拖放、剪贴板、通知数据和命令搜索。
- 重构菜单栏、Dock、窗口框架、系统菜单、Spotlight、Launchpad、通知入口和桌面图标表现。
- 先统一桌面几何与状态语义，再做像素级视觉复刻。

## 2. 审计范围与证据

本次审计覆盖：

- 参考 Demo 的锁屏、桌面、菜单栏、Dock、Finder 窗口、桌面图标和状态反馈。
- 本地桌面的空桌面、文件管理窗口、启动器、右键菜单、任务栏、图标选中态。
- CodeGraph 中的桌面入口、窗口管理、图标网格、任务栏、启动器、菜单、通知和持久化调用关系。
- 现有 Playwright 契约和测试选择器。

在统一 `1440 x 900` 视口下观察到的关键规格：

| 项目 | 参考 Demo | 当前本地 | 结论 |
|---|---:|---:|---|
| 菜单栏高度 | `24px` | `28px` | 本地偏厚，且菜单为静态文字 |
| 系统字体 | `13px / 19.5px` | 多数继承 `16px`，内部再零散覆盖 | 缺少壳层字体等级 |
| Dock 外框 | 高 `62px`、圆角 `24px`、内边距 `7px` | 高 `58px`、宽 `1180px`、圆角 `24px` | 外形接近，但信息架构仍是任务栏 |
| Dock 图标 | `48px`、间距 `4px`、仅图标 | 图标 + 窗口标题 + 托盘 + 双行时钟 | 本地密度和职责失控 |
| 窗口圆角 | `16px` | `16px` | 数值接近，整体材质仍不一致 |
| 标题栏高度 | `44px` | `36px` | 本地偏挤，工具和标题层级不清 |
| 交通灯 | `12px`，左侧固定顺序 | `12px` 外观，但 DOM/Tab 顺序与视觉顺序冲突 | 需要重做结构而非只改 CSS |
| 启动入口 | Dock Apps + Spotlight | 左下角 Windows Start Menu | 必须替换信息架构 |
| 系统状态 | 顶部菜单栏唯一表达 | 顶部状态 + 底部托盘/时钟重复 | 应收口到菜单栏 |

参考 Demo 在窄视口也存在 Dock 溢出和窗口越界，因此“1:1”只复刻其成熟的桌面视觉与交互语言，不复制 Demo 的响应式缺陷和假数据业务。

## 3. 当前混乱点

### P0：实施前必须先收口

#### 3.1 桌面安全区没有唯一来源

当前同时存在 `44px`、`48px`、`58px` 等任务栏高度，实际 Dock 还带 `10px` 底部间距；顶部菜单栏新增了 `28px`，但最大化和吸附仍从 `y = 0` 计算。

直接后果：

- 最大化窗口可能进入菜单栏下方。
- 窗口底部、吸附预览和恢复坐标可能落入 Dock 区域。
- 图标层、窗口层和持久化恢复使用不同可用区域。

相关位置：

- `frontend/src/desktop/config/desktop-preferences.ts`
- `frontend/src/desktop/design-system/desktop-design-tokens.css`
- `frontend/src/desktop/window-manager/window-manager.ts`
- `frontend/src/desktop/window-manager/use-window-interaction.ts`
- `frontend/src/desktop/window-manager/desktop-session-restore.ts`

#### 3.2 底部仍是 Windows Taskbar

`desktop-taskbar.vue` 同时承担：开始按钮、按窗口切换、标题显示、进度、窗口预览、通知、托盘、时钟和显示桌面。

macOS Dock 应只围绕“应用”组织：

- 未运行：点击启动应用。
- 已运行：点击激活该应用最前窗口。
- 最小化：窗口回到所属应用或 Dock 右侧最小化区。
- 多窗口：通过应用菜单或右键菜单选择，不在 Dock 常驻长标题。

#### 3.3 启动器是 Windows Start Menu

当前“已固定 / 全部应用 / 推荐 / 最近打开 / 刷新 / 退出”的左下角面板，即使换成透明材质，仍然不是 macOS 交互。

目标应拆成：

- Spotlight：居中搜索，复用现有 `commandRegistry`。
- Launchpad：全屏应用网格，负责浏览全部应用。
- 用户与退出：迁移到菜单栏账户菜单。
- 最近文件：进入 Finder 或 Spotlight 结果，不在启动面板堆叠。

#### 3.4 菜单栏是视觉贴图，不是菜单系统

顶部“文件 / 编辑 / 查看 / 窗口 / 帮助”目前是 `span`，没有菜单、命令、pressed 状态、键盘导航或激活应用上下文。绿色在线点也是固定装饰。

原则：不可点击的菜单不应伪装成可点击；第一阶段要么实现真实菜单，要么只保留已接通的菜单项。

#### 3.5 材质与图标没有单一来源

当前同时出现：浅色窗口玻璃、近黑启动器、白色右键菜单、深蓝侧栏、Element Plus 图标、Emoji、字符图标、自定义 SVG 和图片图标。

特别需要清理：

- `🔍 👤 ⚡ 🚪 🏠 🗑 📌 ⬆ 📁 ↻` 等 Emoji/字符图标。
- Windows 蓝、Element Plus 蓝、Sky 蓝、Tailwind Blue 多套强调色。
- 组件内部散落的颜色、阴影、圆角和模糊值。

### P1：影响操作流畅度和可信度

#### 3.6 部分控件看起来可用但没有行为

- 菜单栏项目没有行为。
- 部分桌面右键菜单项没有对应执行器。
- 启动器最近项目不是完整可操作入口。
- 右侧快捷栏是 iframe 面板，与 macOS 外壳无明确归属。

#### 3.7 窗口状态存在已知错误

- “显示桌面 / 最小化全部 / 恢复全部”通过逐个 toggle 实现，混合状态时会反向恢复窗口。
- 最小化动画查询不到对应 Dock 目标，可能飞向窗口自身或默认底部中心。
- `preMaximizeState` 没有真实传入，最大化后拖回会跳到兜底尺寸。
- 交通灯 DOM 顺序与视觉顺序不一致，键盘焦点顺序错误。

#### 3.8 选择、右键和拖拽状态不统一

- 常规空白区域点击/框选可能被壁纸子层阻断。
- `Cmd/Ctrl` 点击不能稳定切换选择。
- 右键图标前没有统一同步选中项。
- 新旧两套拖拽状态仍同时存在。
- 窗口接收拖放的 hover 状态没有完整视觉反馈。

#### 3.9 自动化契约已漂移

- 当前启动器类名与部分测试中的 `.desktop-launcher-panel` 不一致。
- 窗口测试把 Windows 左右吸附和固定 `48px` 底部预留写成硬契约。
- 视觉重构前必须先确认哪些测试仍真实执行、哪些已经失效。

### P2：视觉噪声

- 顶部和底部重复显示时钟。
- 无窗口时 Dock 显示“没有打开的窗口”，不是 Dock 应承担的信息。
- Dock 图标、通知、时钟各自上浮，动效互相抢注意力。
- 圆角从 `3px` 到 `999px` 无语义分级。
- 阴影强度没有层级规则。
- 首屏骨架与加载后的桌面材质完全不同，存在闪断。
- 只支持降低动态效果，没有“降低透明度”降级策略。

## 4. 重构边界

### 4.1 完整保留

| 能力 | 保留内容 |
|---|---|
| 应用注册 | `app-loader`、`app-registry`、`app-opener`、权限与可见性规则 |
| 窗口业务状态 | 打开、关闭、激活、单例、多实例、payload、窗口类型 |
| 会话能力 | 窗口和图标状态持久化、恢复入口 |
| 文件能力 | 文件列表、打开方式、上传、移动、回收站、剪贴板 |
| 桌面交互 | 图标网格、选择、拖放的业务结果和事件合同 |
| 通知数据 | `useNotifications`、`NotifyPanel` 的数据与动作 |
| 搜索能力 | `commandRegistry` 及应用/文件/命令搜索 |
| 对外合同 | `window.__HSWZ_WINDOW_MANAGER__` 等已有公开调用签名 |

### 4.2 重构视图和交互投影

| 当前组件 | 目标组件 | 处理方式 |
|---|---|---|
| `desktop-taskbar.vue` | `MacDock` | 重写模板和投影逻辑，保留进度/徽标数据 |
| `desktop-launcher.vue` | `Spotlight` + `Launchpad` | 拆分，复用搜索和应用列表 |
| shell 内静态菜单栏 | `MacMenuBar` + menu registry | 独立组件，接入激活应用和系统状态 |
| `desktop-window-frame.vue` | `MacWindowFrame` | 保留窗口事件合同，替换 Chrome 和动画 |
| `context-menu.vue` | `MacMenu` | 保留 action key，重写渲染和键盘交互 |
| `taskbar-notifications.vue` | 菜单栏通知入口/通知中心 | 保留通知数据，迁移入口 |
| `desktop-right-sidebar.vue` | 普通窗口或控制中心 Popover | 移除 Windows 式 iframe 侧栏 |
| 桌面图标视图 | macOS 桌面图标 | 保留文件/应用 key，改排列、选中和标签 |

### 4.3 删除或默认关闭的 Windows 语义

- “开始”按钮和开始菜单。
- 按窗口显示的底部长标题。
- 右下角托盘时钟。
- 显示桌面细条。
- Windows 风格窗口悬浮缩略预览。
- 默认左右边缘贴靠预览。
- 左上到右下的 Windows 自动排列方向。

边缘贴靠能力先保留在内部，不作为默认 UI 暴露；后续如需要，可改造成绿色按钮长按/悬停后的 macOS 窗口平铺菜单。

## 5. 目标 macOS 外壳合同

### 5.1 结构

```text
DesktopShell
├── Wallpaper
├── MenuBar
│   ├── AppMenu
│   └── SystemStatus / Account / NotificationCenter
├── DesktopIconLayer
├── WindowLayer
│   └── MacWindowFrame[]
├── OverlayLayer
│   ├── MacMenu
│   ├── Spotlight
│   ├── Launchpad
│   ├── Popover
│   └── NotificationCenter
└── Dock
```

壳层组件只负责呈现和触发命令；文件上传、移动、回收站和权限判断继续留在现有控制层。

### 5.2 桌面几何

新增唯一的 `DesktopChromeMetrics`/`DesktopInsets`：

```ts
type DesktopInsets = {
  menuBarHeight: number
  dockHeight: number
  dockGap: number
  windowTop: number
  windowBottom: number
  workArea: { x: number; y: number; width: number; height: number }
}
```

所有新窗口、最大化、缩放、拖拽边界、会话恢复、图标区域和弹层定位必须使用该对象，禁止再写 `44/48/58` 等独立常量。

建议桌面规格基线：

| Token | 基线值 |
|---|---:|
| MenuBar | `24px` |
| Dock 图标 | `48px` |
| Dock 内边距 | `7px` |
| Dock 图标间距 | `4px` |
| Dock 外框高度 | `62px` |
| Dock 圆角 | `24px` |
| Window 圆角 | `16px` |
| Window Titlebar | `44px` |
| Traffic Light | `12px` |
| 系统控件圆角 | `6px` |
| 菜单圆角 | `10px` |
| Popover 圆角 | `12px` |

这些值以参考 Demo 为视觉基线，最终允许针对中文标签和正式业务密度做小幅校准，但不可在组件内自行漂移。

### 5.3 材质

只保留四级 OS 材质：

1. MenuBar：轻透明、低阴影、壁纸自适应文字。
2. Dock/Menu/Popover：中等模糊、统一边线和高光。
3. Window Chrome：浅色实体/半透明混合，强调内容可读性。
4. Content Surface：业务内容面，不强制玻璃，不让壁纸干扰阅读。

每级材质统一定义：背景、模糊、饱和、边线、高光、阴影、深浅色降级。组件不得自行声明另一套玻璃。

### 5.4 字体和图标

- 壳层字体固定为系统栈：`-apple-system, BlinkMacSystemFont, "SF Pro Text", "PingFang SC", sans-serif`。
- 只保留 Menu、Title、Body、Caption 四级，不再零散使用 500/600/700/800。
- 应用图标保持全彩品牌图标。
- 系统操作图标统一使用一个矢量源，优先 `lucide-vue-next`；禁止 Emoji 和字符图标混用。
- OS Chrome 内不直接使用 Element Plus 图标；Element Plus 继续服务业务应用内部。

### 5.5 交互状态

所有可交互控件必须覆盖：

`rest / hover / pressed / selected / focus-visible / disabled / busy`

状态变化不得改变控件布局尺寸。所有假按钮必须移除或接通真实行为。

### 5.6 动效

仅保留三档：

| 类型 | 时长 | 用途 |
|---|---:|---|
| Fast | `100-140ms` | hover、pressed、菜单项 |
| Standard | `180-220ms` | 菜单、Popover、Spotlight |
| Window | `240-320ms` | 打开、关闭、最小化、恢复 |

统一曲线，不在每个组件中重新定义。必须支持：

- `prefers-reduced-motion`
- 降低透明度模式
- 不支持 `backdrop-filter` 时的实体背景降级

## 6. 分阶段实施计划

### 阶段 0：冻结基线与修复测试合同

目标：确认现有底座真实可用，防止视觉重构掩盖状态错误。

任务：

1. 为 `/desktop` 建立稳定 mock 登录与固定数据场景。
2. 修正已漂移的 Launcher 测试选择器。
3. 将现有窗口打开、关闭、最小化、恢复、拖动、缩放、会话恢复测试改为行为断言，不绑定 Windows UI。
4. 建立空桌面、单窗口、多窗口、启动器、右键菜单的截图基线。
5. 记录现有公开 DOM 合同：`.desktop-window`、`data-window-id`、`data-accepts-drop` 等。

涉及：

- `frontend/tests/ui-e2e/desktop.mjs`
- `frontend/tests/desktop-launcher-fileops.spec.mjs`
- `frontend/tests/window-snap-preview.spec.mjs`
- 新增桌面视觉回归 spec

退出条件：所有底座行为在不依赖当前任务栏外观的情况下可重复通过。

### 阶段 1：统一设计 Token 与桌面几何

目标：建立后续所有组件的唯一尺寸和材质来源。

任务：

1. 新增 `DesktopInsets/ChromeMetrics` 计算模块。
2. 菜单栏、Dock、窗口、图标层、弹层共用 work area。
3. 统一颜色、圆角、阴影、字体、z-index 和动效 token。
4. 为旧会话几何增加版本迁移和 clamp，避免窗口恢复到遮挡区。
5. 修复 `preMaximizeState`、显示桌面 toggle、最小化目标定位。

### 阶段 2：MacWindowFrame

目标：先让所有业务窗口共享稳定的 macOS Chrome。

任务：

1. 按“关闭 / 最小化 / 缩放”顺序重建交通灯 DOM 和键盘顺序。
2. 标题栏统一为 `44px`，标题、应用图标和工具栏建立清晰层级。
3. 激活/非激活窗口只改变阴影、标题和交通灯饱和度，不整体变灰或透明。
4. 修复打开、关闭、最小化到 Dock、恢复和缩放动画。
5. 保留 `.desktop-window`、drop attributes 和现有 emit 合同。
6. 默认关闭 Windows 边缘贴靠视觉；保留内部几何能力。

涉及：

- `frontend/src/desktop/window-manager/desktop-window-frame.vue`
- `frontend/src/styles/desktop-shell.css`
- `frontend/src/desktop/window-manager/use-window-interaction.ts`

退出条件：任一现有业务应用无需改业务代码即可获得一致窗口框架。

### 阶段 3：MacMenuBar + MacDock

目标：彻底移除 Windows Taskbar 语义。

MenuBar 任务：

1. 菜单随激活应用更新标题和命令。
2. 实现真实菜单的 hover、pressed、键盘和外部点击关闭。
3. 系统状态、通知、账户和时间只在顶部表达。
4. 没有命令来源的菜单项不显示。

Dock 任务：

1. 将窗口列表投影为按 `appKey` 分组的应用列表。
2. 区分固定应用、运行中应用、分隔区和废纸篓。
3. 仅显示图标、运行指示点、徽标和进度；标签只在 tooltip 出现。
4. 点击遵循应用语义，不再切换单个窗口标题按钮。
5. 多窗口入口通过右键菜单显示窗口列表。
6. Dock 放大效果使用邻近图标插值，保持外框稳定，不造成布局跳动。

涉及：

- `frontend/src/desktop/shell/index.vue`

4. 通知入口迁移到菜单栏，通知数据和业务动作保持不变。
5. 右键菜单替换 Emoji，补齐方向键、Enter、Escape、焦点回收和子菜单延迟。

任务：

4. 合并新旧拖拽状态，补齐文件夹和窗口接收态。
5. 保持现有持久化 key，必要时迁移网格坐标而不丢用户位置。

## 7. 验收清单

### 7.1 视觉

- [ ] MenuBar、Dock、Window、Menu、Popover 使用同一套材质合同。
- [ ] 桌面首屏没有 Windows 开始菜单或任务栏视觉。
- [ ] Dock 只显示应用图标、指示点、徽标和进度。
- [ ] 交通灯尺寸、顺序、间距和焦点顺序一致。
- [ ] 系统图标不再混用 Emoji、字符和 Element Plus。
- [ ] 强调色只有一套系统蓝，红黄绿只表达状态和交通灯。
- [ ] 顶部和底部没有重复时钟或系统状态。

### 7.2 交互

- [ ] Dock 点击、右键、多窗口和最小化语义正确。
- [ ] 菜单栏只显示真实可执行命令。
- [ ] Spotlight、Launchpad、通知和菜单支持键盘关闭与焦点回收。
- [ ] 打开、关闭、激活、最小化、恢复、缩放、拖动和会话恢复无回归。
- [ ] 文件关联、权限、上传、移动、回收站、剪贴板和拖放无回归。
- [ ] 混合最小化状态下“显示桌面/恢复”不会反向切换窗口。

### 7.3 布局与性能

- [ ] `1440x900`、`1280x800`、`1024x768` 无窗口/菜单/Dock 重叠。
- [ ] 动态内容不会改变 Dock 和工具栏固定尺寸。
- [ ] 拖窗、Dock 放大和窗口动画保持流畅，无持续 layout thrashing。
- [ ] 截图基线覆盖空桌面、单窗口、多窗口和主要系统弹层。
- [ ] 降低动态效果、降低透明度和无 blur 环境可用。

## 8. 实施约束

1. 不迁移到 React、Tailwind 或 Framer Motion；参考站技术栈不是舒适度来源。
2. 不复制参考 Demo 的 Finder/Safari 等假业务，只复制 OS Chrome 的视觉和交互语言。
3. 不直接改动业务模块页面，第一轮只建立壳层合同；业务应用内部的 macOS 化另立计划。
4. 不使用 Apple 专有二进制图标或不可再分发资产；采用自有应用图标和统一开源系统 glyph。
5. 每一阶段必须可独立回归，禁止一次性同时重写窗口状态、文件业务和视觉。
6. 当前工作区已有未提交改动，实施时按文件边界小批次提交，不覆盖无关修改。

## 9. 推荐首批实施切片

第一批只做以下内容，避免一开始铺得过大：

1. 修复有效测试基线。
2. 建立 `DesktopInsets` 唯一安全区。
3. 修复窗口最大化/恢复/显示桌面三个状态问题。
4. 完成 `MacWindowFrame`。
5. 将现有 Taskbar 改为纯应用 Dock，暂不做 Launchpad 和通知中心。

这批完成后，桌面壳会先从“Windows + macOS 混合”收口成可信的 macOS 基础框架；Spotlight、Launchpad、菜单和通知中心再作为第二批补齐。

## 10. 最终判断

参考 Demo 值得学习的是边界清楚和系统级一致性，不是 React 组件数量或 Liquid Glass 特效本身。我们的正式项目已经拥有比 Demo 更完整的业务底座，因此重构重点应是：

**保留能力，统一几何，替换外壳，删除重复入口，最后做像素校准。**

按本计划推进，可以在不牺牲现有业务能力的前提下，把桌面壳收口为纯 macOS 视觉和交互系统。

1. 菜单栏、Dock 和窗口安全区使用统一几何合同，最大化、拖动、恢复和会话坐标不会落入系统 Chrome。

6. 应用图标改为统一圆角、光泽、内描边和阴影的 macOS 风格图标系统，并按 `appKey` 区分文档、表格、PDF、演示、AI、知识库、模型路由等应用，避免复用同一缩略图。
