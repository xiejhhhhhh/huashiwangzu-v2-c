# 10_WP2 内容摄取异步编排器施工方案（20260717）

## 一、本文件作用

冻结 WP2「上传→异步编排→阶段 DAG」的全部接口契约，让子代理按契约并行开发、主线串行整合，杜绝各建各的整合账单。

## 三、阶段 DAG（冻结）

单一 task_type：`content_ingest_stage`。一个任务跑一个阶段，settlement 在 fenced 事务里发后继阶段。链路顺序：

```
package_ensure → canonical_parse → resource_extract → derivative_build → knowledge_register
```

| 阶段 | 干什么 | 本轮是否需模型 | 本轮动作 |
|------|--------|----------------|----------|
| package_ensure | get_or_create ContentPackage，绑定 source_file_id/owner | 否 | 正常跑 |
| canonical_parse | 调解析器出 CanonicalIR，写 ContentPackageVersion + 真源 sha256 | 看格式 | 结构类正常跑；需 VLM/LLM 的格式(纯图/扫描 PDF)降级 metadata_only，标 deferred |
| resource_extract | 抽图片/表格/媒体为 Resource + ResourceRef | OCR/VLM=是 | 结构抽取正常；OCR/VLM 描述 skipped(model_budget_deferred) |
| derivative_build | 生成预览图/标准图等派生物 | 否(纯结构) | 正常跑 |
| knowledge_register | 注册进知识库(分块+embedding) | embedding=是 | 整阶段 skipped(model_budget_deferred)，账本标 deferred |

阶段状态取值(§19.4)：`done | degraded | skipped | failed`。`skipped(reason=model_budget_deferred)` 不算失败、不阻塞后继、可在额度恢复后重入(replay generation+1)。

## 四、IngestionRun 账本状态机（冻结）

`framework_ingestion_runs.status` 取值：
```
queued → running → (completed | degraded | failed | dead_letter | cancelled | paused)
```
- 终态不原地重开。replay = 新建一条 generation+1 的 run，`replay_of_id` 指向原 run。
- 唯一键 (source_revision_id, pipeline_version, generation) 保证同一字节版本同一代不重复建 run（幂等触发）。
- `cancel_requested=true` 时，各阶段 handler 开头检查，主动落 `cancelling→cancelled`。
- `lock_version` 乐观锁：状态跃迁 `UPDATE ... WHERE id=:id AND lock_version=:v`，rowcount=0 视为并发，跳过。

阶段账本：**复用 SystemTaskQueue 行本身**（stage_key 标阶段，status/result 记结果），不新建 stage_run 表——与 §19.3-B 冻结的 5 表一致（无 stage_run 表）。IngestionRun 做跨阶段聚合。

## 六、幂等键（冻结，每阶段 handler 开头自检）

dispatcher 重试 = 同一队列行重跑 handler，故每阶段必须幂等：

| 阶段 | 幂等判据（已完成则直接返回既有结果，不重做） |
|------|------|
| package_ensure | ContentPackage 已存在且 source_file_id 匹配 → 返回 |
| canonical_parse | IngestionRun.package_version_id 已置 **且** run.source_sha256==当前 sha256 → 返回既有 version |
| resource_extract | 该 version 的 ResourceRef 已存在(count>0) → 跳过重抽 |
| derivative_build | FileDerivative 该 kind 已存在(唯一键 file_id+kind) → 跳过 |
| knowledge_register | 本轮恒 skipped(model_budget_deferred)，天然幂等 |

### 关键设计落点
- **上传即返回**：`file.uploaded` handler 只建 run + 发 `package_ensure` 任务就返回，不再内联解析。解析由 Dispatcher 异步跑阶段 DAG。
- **DAG 推进**：handler 跑一阶段并 commit；Dispatcher 在 fenced settlement tx 里调 `_settle_ingest_task` 发后继阶段——发后继与标结果同事务，要么都成要么都不成。
- **§24 诚实降级**：`knowledge_register` 本轮恒 `skipped(model_budget_deferred)`；需 VLM 的图片格式在 `canonical_parse` 落 metadata_only 版本、package 状态 `degraded`（不假绿 ready）；run 终态落 `degraded`+`error_code=model_budget_deferred`，额度恢复后 `replay_run` 建 generation+1 重跑。
- **幂等**：同字节版本（source_revision_id+pipeline_version+generation）只有一条 run；每阶段 handler 开头自检（package 已存在/version 已写且 sha 未变/ResourceRef 已有/派生物已有）→ 已完成直接返回，重试不产生重复。
- **package_id/version_id 回填**：`package_ensure` 建的 package id 只在 result 里，settlement 按 result>run>params 优先级解析并乐观锁回填到 run，供后继阶段与重放用。

### E2E 验证（已跑过）
上传 md → kickoff 建 run(queued) → 驱动 DAG：package_ensure(done)→canonical_parse(done)→resource_extract(done)→derivative_build(done)→knowledge_register(skipped/model_budget_deferred) → run 终态 `degraded`，pkg_ver 正确回填，**幂等再解析不产生第 2 个 version**。巡检 PASS=5/FAIL=0，后端启动 68 路由，handler/settlement/reconciler 均注册。

### 调用方式
- **自动**：上传文件 → `file.uploaded` 事件 → 编排器自动 kickoff，无需手动调。
- **重放（额度恢复后重跑 deferred）**：`await replay_run(origin_run_id, caller)` → 建 generation+1 新 run 跑全 DAG。
- **查询/取消/重放 HTTP API**：见 `content_ingestion` 路由（子代理-1 落地，API 契约以其交付为准）。
