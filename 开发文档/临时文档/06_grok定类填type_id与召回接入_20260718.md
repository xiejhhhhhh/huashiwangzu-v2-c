# 06_grok定类填type_id与召回接入_20260718

## 阶段一：问题 → 方案 → 影响范围

### 问题（三个断链，都验过库）

1. **type_id 挂载=0**：18类语义体系表建了、deepseek 也判过，但旧脚本 `批_实体归类.py` 只 UPDATE 了 `category`（中文类名），从来没写 `type_id` 外键。之前烧的算力等于白烧。
   - 待定类（type_id IS NULL）：93855 条
   - 垃圾桶（术语/其他/通用）：34287 条
   - 确认率仅 2.1%（10.5万实体只确认2242）

2. **旧脚本没喂证据**：光看实体名判类，精度上不去。取证工具组（查词频/查上下文/差异字）现成，没被定类复用。

3. **召回层完全不读 type_id**：search_service.py 全链路只读 `category`（当元数据带出，不加权）+ `status`（唯一噪音剔除杠杆）。定类只写 type_id = 悬空成果。

### 方案（数据层 + 代码层并行，否则又白烧）

**数据层（grok-4.5 定类，烧算力）：**
- 新脚本 `批_grok定类_20260718.py`，同时写 `category` + `type_id`。
- 喂证据：实体名 + 当前类目 + df（文档频率）+ 2~3 条真实上下文（JOIN kb_chunk_entities→kb_chunks 拿原文，比 LIKE 准）。
- 噪音纯规则预筛：纯数字/分辨率(如5168×3448px)/条形码/13位数字串 直接判噪音，不烧 grok。
- 噪音判定红线：grok 判噪音的**第一波只写 type_id=18 + 落隔离表 kb_entity_noise_review**，不物理删、不改 status（保持可召回）。精度验够（≥95%）再批量归档。
- 正向归类：即时写 category + type_id（收益直接拿，幂等：WHERE type_id IS NULL 重跑同命中）。
- 高频优先：按 df 降序，先定影响召回最大的实体。

**代码层（召回读 type_id，零算力，后续 P1）：**
- search_service.py `_load_entity_idf_table` 带出 type_id / is_noise。
- 噪音类实体（type_id=18）召回时降权或排除。
- 按查询意图做类型加权（问成分→成分类加权）。
- 此项影响面大，先出方案、备份、确认回退再动，本轮先把数据层跑通。

### 影响范围
- 只改 `kb_entity_dictionary.category` + `type_id`（正向）/ `kb_entity_noise_review`（噪音落表）。
- 不删实体、不改 status、不动 chunk/向量/图谱。
- owner_id=4。幂等可断点续跑。

### 执行步骤
1. 清成坨僵尸并入定类（≥13字 9749 条，规则预筛噪音 + 真实体归类）。
2. grok 定类脚本，dry-run 50 条人工核精度。
3. 精度过 → 放量 84106 条，后台 agent 每15分钟抽样监控。
4. 精度掉立即 pkill + 回滚。

### 验收
- type_id 挂载数从 0 → 覆盖率＞90%。
- 垃圾桶（术语/其他/通用）大幅下降。
- 噪音隔离表可复审，无误删。
