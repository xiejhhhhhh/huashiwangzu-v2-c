## 2. 审计发现的确定问题

### 问题 1：Knowledge 文档半孤儿

当前审计 SQL 证据：

```sql
SELECT COUNT(*) active_docs,
       COUNT(*) FILTER (WHERE fi.deleted=false) visible_source_docs,
       COUNT(*) FILTER (WHERE fi.deleted=true) hidden_source_deleted_docs
FROM kb_documents d JOIN framework_file_items fi ON fi.id=d.file_id
WHERE d.deleted=false;
```

结果：

```text
active_docs = 165
visible_source_docs = 2
hidden_source_deleted_docs = 163
```

含义：Knowledge 里大多数 active 文档的源文件已经被删除或进回收站。用户看 Knowledge 时会看到大量 source unavailable / failed，而不是一个清楚的治理状态。

### 问题 2：ContentPackage 半孤儿

审计 SQL：

```sql
SELECT COUNT(*) packages,
       COUNT(*) FILTER (WHERE deleted=false) active_packages,
       COUNT(*) FILTER (WHERE source_file_id IS NOT NULL AND EXISTS (
         SELECT 1 FROM framework_file_items fi
         WHERE fi.id=framework_content_packages.source_file_id AND fi.deleted=true
       )) source_file_deleted
FROM framework_content_packages;
```

结果：

```text
packages = 204
active_packages = 204
source_file_deleted = 198
```

含义：ContentPackage 大量仍 active，但它们的 source_file 已删除。Content IR 写入后没有跟文件生命周期同步，后续 publish / preview / export 可能指向不可用源。

### 问题 3：测试文件污染用户文件域

审计 SQL：

```sql
SELECT COUNT(*) total_files,
       COUNT(*) FILTER (WHERE deleted=false) active_files,
       COUNT(*) FILTER (WHERE deleted=true) deleted_files
FROM framework_file_items;
```

结果：

```text
total_files = 303
active_files = 6
deleted_files = 297
```

大量 `smoke-*`、`e2e-*`、`recycle-*` 等测试文件进入回收站，但没有真正从用户数据域、Knowledge 派生数据、ContentPackage 派生数据里清干净。

### 问题 4：文件生命周期事件只被 Knowledge 部分处理，ContentPackage 未闭环

当前代码事实：

- `backend/app/routers/files.py` 会发出 `file.deleted` / `file.restored` 等事件。
- `modules/knowledge/backend/services/file_lifecycle_service.py` 会对 `file.deleted` 标记 `source_file_deleted`。
- ContentPackage / Content IR 链路没有等价生命周期处理。

### 问题 5：release gate 还没有拦住这类债务

当前 `/api/health` 和 task audit 是绿的，但 Knowledge/Content 资产债务巨大。release gate 可以 `PASS_WITH_DEBT`，但没有把这些生命周期债务升级成明确 gate 项。

## 4. 目标一：定义统一资产生命周期状态

### 4.1 目标

给 Knowledge 文档和 ContentPackage 建立一致的源文件生命周期语义，不要求完全同表字段，但要求对外口径一致。

必须支持这些状态：

```text
source_available        源文件存在且未删除
source_recycled         源文件已进入回收站，可恢复
source_missing          源文件记录不存在或物理源缺失
source_permanently_deleted  源文件永久删除，派生数据只能归档或删除
archived_by_lifecycle   派生数据因源文件生命周期被归档
restored                源文件恢复后派生数据恢复可用或重新入队
```

### 4.2 实现要求

- Knowledge 可以继续使用现有 `source_file_deleted/source_file_missing`，但要在 service/API 层映射成统一生命周期状态。
- ContentPackage 必须新增等价生命周期识别与状态输出。
- 不要把源文件删除导致的状态显示为普通 failed。
- 不要把生命周期归档当作系统错误。

### 4.3 验收

- 对 deleted source file 的 Knowledge 文档，API 返回 lifecycle 状态，不再只表现为普通 failed。
- 对 deleted source file 的 ContentPackage，API/治理接口能识别 source lifecycle 状态。
- 用户文案能区分“系统处理失败”和“源文件已删除/在回收站”。

## 5. 目标二：补齐 ContentPackage 文件生命周期处理

### 5.1 目标

ContentPackage 必须响应源文件删除、恢复、永久删除。

### 5.2 必须实现

1. `file.deleted`
   - 找到 `source_file_id` 等于被删除文件的 active ContentPackage。
   - 将其标记为 lifecycle archived 或 source unavailable。
   - 不要物理删除 package/version。
   - 返回处理数量。

2. `file.restored`
   - 如果源文件恢复，则恢复对应 ContentPackage 的 source available 状态。
   - 如果需要重新验证/重建 preview，进入可重建/待验证状态。

3. `file.permanent_deleted` 或等价永久删除事件
   - 如果项目当前没有这个事件，请先查清实际永久删除入口。
   - 有事件则接入；没有事件则写清楚缺口，并至少在治理审计里识别 permanently missing。

4. 幂等
   - 同一事件重复触发不应重复污染状态。

### 5.3 修改范围建议

可能涉及：

```text
backend/app/services/content/package_service.py
backend/app/models/content*.py 或相关 ContentPackage model
backend/app/routers/content.py
backend/app/services/module_events.py 或事件注册入口
backend/tests/test_content_package_lifecycle.py  # 可新增
```

具体以当前代码为准，先 `code_explore` / `code_node` 查准。

### 5.4 验收

- 新增或补齐测试：删除源文件后 ContentPackage 状态变为 lifecycle archived/source unavailable。
- 恢复源文件后状态可恢复或进入待验证。
- 重复事件幂等。
- 不破坏现有 Content IR 架构测试。

## 6. 目标三：Knowledge 生命周期治理从“提示”升级为“批量治理队列”

### 6.1 目标

当前 Knowledge 已能显示 source unavailable，但历史 163 条债务还在。需要提供治理能力，而不是只在前端解释。

### 6.2 必须实现

提供一个 Knowledge 生命周期治理能力，至少支持 dry-run：

```text
knowledge:audit_lifecycle_debt
knowledge:archive_source_unavailable_documents
```

或同等 HTTP/API 入口。

要求：

- `audit_lifecycle_debt` 返回：
  - source_recycled_count
  - source_missing_count
  - candidate_document_ids
  - sample documents
  - recommended action
- `archive_source_unavailable_documents` 必须支持：
  - dry_run=true 默认；
  - limit；
  - reason；
  - confirm token，例如 `confirm="ARCHIVE_SOURCE_UNAVAILABLE"` 才真实执行。
- 真实执行只做软归档/标记 deleted，不物理删除 chunks/raw/fusions，除非另有明确清理任务。

### 6.3 验收

- dry-run 不改数据。
- confirm 不正确时拒绝执行。
- 真实执行可限制 limit，执行后 dashboard 的 failed/source_unavailable 应减少或转入 lifecycle archived 统计。
- 执行记录要可审计，至少返回 affected ids / count / reason。

## 7. 目标四：测试污染识别与门禁

### 7.1 目标

smoke/e2e/recycle 测试文件不能长期污染用户文件域、回收站、Knowledge、ContentPackage。

### 7.2 必须实现

新增一个只读审计能力，识别测试污染：

```text
workspace/test data pollution audit
```

可以放在 dev_toolkit 或 backend service，按项目现有风格选。

必须统计：

```text
active test files
recycled test files
knowledge documents from test files
content packages from test files
uploads test artifacts
```

测试 marker 至少包括：

```text
smoke-
e2e-
recycle-
test-
pytest-
```

### 7.3 清理策略

不要直接清真实数据。

清理能力必须：

- 默认 dry-run；
- 只清明确 marker 的测试数据；
- 支持 limit；
- 支持 confirm，例如 `confirm="CLEAN_TEST_DATA"`；
- 清理时同步处理 file/recycle/Knowledge/ContentPackage 派生状态。

### 7.4 release gate 要求

release gate 至少增加一个 DEBT 项：

```text
Test data pollution
```

当 active/recycled/knowledge/content 测试 marker 超过阈值时：

- 现有历史污染可以先记 DEBT；
- 新增污染必须 BLOCKER，或至少在 gate-run delta 里标红。

### 7.5 验收

- 审计接口/工具能返回当前测试污染计数。
- dry-run 不改数据。
- confirm 才清理。
- release gate 输出包含测试污染项。

## 8. 目标五：发布门禁接入资产生命周期债务

### 8.1 目标

release gate 不能只看 health/task queue/smoke。资产生命周期债务也必须进入机器可读结果。

### 8.2 必须接入的 gate 项

```text
Knowledge lifecycle debt
ContentPackage lifecycle debt
Test data pollution
```

每项输出：

```json
{
  "check": "Knowledge lifecycle debt",
  "level": "PASS|DEBT|BLOCKER",
  "detail": "source_unavailable=163, archived=0, new_since_baseline=0",
  "data": {...}
}
```

### 8.3 PASS_WITH_DEBT 语义

本任务不要求彻底改 release gate 语义，但要避免误读：

- `release_safe=true` 可以保留，但必须同时输出 `clean_release_ready=false`。
- 如果已有字段可表达，则不要重复造字段；如果没有，请补充。
- 文档说明：有生命周期债务时不是 clean release。

### 8.4 验收

- `release_gate --preflight --skip-ui` 输出包含 lifecycle/test pollution 三项。
- 当前历史债务至少进入 DEBT。
- 不出现 BLOCKER 时可保持 PASS_WITH_DEBT。
- 机器 JSON 可读，不只靠文本。
