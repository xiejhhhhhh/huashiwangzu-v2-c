## 3. 背景证据

产品化总审计结论：

- 当前产品化成熟度 6.1/10，桌面体验 6.2/10。
- 最大用户感知缺口是：不知道后台有没有在跑、哪里失败了、是否待确认、产物在哪里。

- Agent workflow ledger 已经有基础，但用户无感知体验不足。
- `partial/manual_required/needs_confirmation` 必须有明确下一步动作。

## 4. 功能目标

第一期只做“桌面反馈入口 + 聚合展示 + 可理解文案”，不要做复杂任务管理器。

至少实现：

1. 任务栏显示反馈中心入口。
2. 有未读通知、后台任务、失败、待确认时有轻量状态提示。
3. 点击入口打开通知/反馈面板。
4. 面板至少聚合：
   - 通知列表；
   - 任务审计摘要；
   - Agent 待确认/工作流摘要，如现有 API 可用；
   - 失败或部分完成的人话说明；
   - 可跳转入口，后端缺字段时先显示“去对应模块查看”。
5. 空状态要友好，不白屏，不显示技术堆栈。
6. 不使用 `any`、`as any`、`@ts-ignore` 绕过类型。

---

## 5. 用户文案口径

前端展示尽量用用户能懂的 6 类：

```text
等待中
处理中
需要确认
已完成
失败
部分完成
```

禁止把内部字段直接暴露给普通用户，例如 raw pending、stale、PASS_WITH_DEBT、workflow_run_id、tool_call_id 等。管理员展开详情可以看到更多技术字段。

## 6. API 使用建议

优先消费已有接口，先查准再写：

```text
routes(filter="notification")
routes(filter="task")
routes(filter="agent")
probe(GET /api/notifications)
probe(GET /api/tasks/worker/audit)
call_capability(agent:list_workflows)
```

如接口返回空，也要保证 UI 空状态正常。
