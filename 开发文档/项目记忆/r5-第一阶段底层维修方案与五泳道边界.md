# R5 第一阶段目标

以 R4 `PASS_WITH_DEBT` 后的剩余底层债务为边界，不继续无限深挖。第一阶段收口对象是 knowledge pipeline 生命周期债与多 agent 验收效率，不把媒体算法大工程混入同一批。

# 当前基线

- `framework_system_task_queues` 中 `module='knowledge' AND task_type='kb_pipeline'`: failed=769, completed=957。
- `classify_pipeline_debt(limit=500)` 样本：source_file_missing=294, source_file_deleted=162, doc_missing=24, file_row_live=13, parser_no_content_blocks=2, async_context_error=4, duplicate_or_stale_parse_lock=1。
- `kb_documents` 状态机债：status_doc_source_file_missing=845, status_doc_source_file_deleted=440。
- `kb_pipeline_runs`: running=4，均为 task_id 为空的 orphan diagnostic rows。

# 五条并行泳道

A. knowledge source-unavailable 状态机：源文件 missing/deleted 文档对外 `source_unavailable`，search_ready/deep_ready=false，历史 done 不再误判完成。

B. knowledge failed 队列 guarded archive：只归档 doc_missing/source_file_missing/source_file_deleted，保留 previous error/result，dry-run 与 task_ids 小批 apply 一致。

C. knowledge orphan running reconcile：独立能力 dry-run/apply `kb_pipeline_runs.status='running' AND task_id IS NULL` 的源文件不可用诊断行，不和队列归档混用。

D. dev_toolkit baseline 归因：worktree_guard/finish_task 支持 baseline，区分开工前 dirty 与本 agent 新增越界。

E. dev_toolkit 大响应裁剪：probe/call_capability 支持 selector/max_items/max_bytes，治理接口默认仍兼容旧输出。

# 第一阶段验收门禁

- 5 个泳道各自 focused tests + ruff 通过。
- knowledge live dry-run 可见 archive/reconcile/status_machine 收口；生产 DB mutation 只能由主会话小批验收后执行。
- `/api/health` ok；`release_gate.py --skip-ui` 无 BLOCKER。
- knowledge failed 数不得新增，若执行 archive 小批则必须下降且 result 保留审计链。
- 所有子代理落项目记忆；主会话验收后按稳定切片提交、推支线、合 main。

# 第二阶段候选

- media-intelligence 替换 `local_algorithms.placeholder`，先接真实 metadata/keyframe/OCR/object/embedding 事实层，VLM 只做裁判。
- knowledge parser quality、greenlet_spawn、duplicate parse lock 作为代码缺陷专项，不混入生命周期治理。
