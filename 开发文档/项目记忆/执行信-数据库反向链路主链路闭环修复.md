## 3. 总目标

把数据库里已经存在或正在产生的数据，按主链路分成三类处理：

1. **必须保留并闭环**：有用户价值，补 API、UI、治理页、测试和 release gate 检查。
2. **历史债务/运行痕迹**：补治理能力、归档策略、清理策略，不让它污染“成功/失败”判断。
3. **疑似废弃/无入口**：只做标记和报告，不直接删除生产数据；需要删除也必须有迁移/备份/确认。

## 4. 优先修复主链路

请基于 `数据库反向链路审计报告.md` 的 Top 风险，优先收口以下链路。

### 4.1 后台任务 / 队列债务链路

关注表和代码：

```text
framework_system_task_queues
backend/app/routers/tasks.py
backend/app/services/task_queue_audit_service.py
backend/app/services/task_debt_governance_service.py
frontend/src/... 任务中心相关页面
```

目标：

- 任务队列有统一审计 API。
- 历史 failed / stale pending 不再让 release gate 假红，也不能被忽略成假绿。
- 用户能在任务中心看到可理解的状态、失败原因、可操作建议。
- 管理员能区分：本次新增失败 vs 历史债务。
- `PASS_WITH_DEBT` 只能表示“可发布但有债”，不能当 clean PASS。

建议输出：

- `/api/tasks/audit` 或现有同类端点完善。
- stale pending / failed baseline / new failures 结构化返回。
- 前端任务中心能展示债务摘要。
- release gate 消费同一结构化数据。

### 4.2 通知 / 消息 / 用户可见反馈链路

如果数据库中存在通知、消息、系统事件类表，请查清：

```text
表 → service → router → frontend → 测试
```

目标：

- 不再只有落库没有入口。
- 用户能看到和自己相关的通知。
- 管理员能看到系统级异常/后台任务债务提醒。
- 无用表只标记，不盲删。

### 4.3 文件 / 上传 / 解析会话链路

重点检查：

```text
file upload session
parser diagnostics
framework_file_items
framework_file_recycle_items
```

要求：

- 任何按 `file_id` 读取文件内容的地方必须走 `check_file_access(db, file_id, user_id)`。
- parser 诊断信息不要只落库，要能被任务中心/文件详情/治理接口看到。
- 回收站、上传会话、解析状态不要互相矛盾。

### 4.4 记忆 / 知识库污染治理链路

关注：

```text
kb_* 表
开发文档/项目记忆/
uploads
knowledge_noise_report / cleanup
```

目标：

- 测试/烟雾/验收污染文件可识别、可清理、可报告。
- 不要把一次 workflow 的大量中间过程都写成项目记忆。
- 项目记忆控制：每个 WorkflowRun 最多 1 条 summary memory。
- 清理工具必须保守，不能误删用户真实资产。

### 4.5 release gate / smoke / sandbox 链路

关注：

```text
dev_toolkit/release_gate.py
dev_toolkit/smoke.py
dev_toolkit/module_sandbox_matrix.py
dev_toolkit/config.json
```

目标：

- release gate 输出机器可读 JSON。
- 记录 git sha、dirty worktree、backend/frontend 可用性、smoke 结果、sandbox 结果、历史债务摘要。
- 支持把 `module_sandbox_matrix` 的 `--jobs` / `--frontend-jobs` 从 release gate 透传或配置化。
- 不要在 `dev_toolkit/config.json` 保留明文账号密码和 DSN；改成：

```text
dev_toolkit/config.example.json
dev_toolkit/config.local.json   # gitignore
环境变量覆盖
```

- 兼容现有本地开发体验，不要让用户手动配一堆东西才能跑。

## 8. 配置安全要求

`dev_toolkit/config.json` 当前存在明文 PostgreSQL DSN 和测试账号密码风险。

本任务要改成更工程化的方式：

1. 提供 `dev_toolkit/config.example.json`，只放示例和默认非敏感值。
2. 本地真实值放 `dev_toolkit/config.local.json`，加入 `.gitignore`。
3. 支持环境变量覆盖。
4. release gate / probe / smoke 读取配置时保持向后兼容：如果没有 local 配置，也能使用安全默认或环境变量。
5. 不要把真实密码写入新文件。

注意：用户说这是局域网项目，不要把这个当 P0 危机夸大；它是工程化改进，不是阻塞用户使用的安全灾难。
