## 发现清单

### 🔴 级别 — 阻断级

#### F1. `system_trace_spans` 表命名违规
- **位置**: `backend/app/models/system_trace.py:8`
- **描述**: `__tablename__ = "system_trace_spans"` 缺少 `framework_` 前缀
- **规则**: 底层文档"数据库表命名"：`{owner}_{domain}_{sub_domain}`，框架表用 `framework_*`
- **修复**: 改 `__tablename__ = "framework_system_trace_spans"` + 做迁移重命名

#### F2. pytest 报错 — 模块表与框架 MetaData 冲突
- **位置**: `cd backend && pytest`
- **错误**: `Table 'agent_usage_daily' is already defined for this MetaData instance`
- **根因**: Agent/Knowledge/Memory/IM 模块的 ORM 模型直接继承框架 `Base`，测试初始化时重复注册。所有模块的 `Base` 实例共用同一 MetaData 空间
- **修复**: 模块需使用独立的 `declarative_base()` 实例或确保 `__table_args__ = {'extend_existing': True}`；同时检查 `init_db.py` 中的 `Base.metadata.create_all` 是否与框架冲突

#### F3. 前端构建失败 — IM 模块 Element Plus 依赖
- **位置**: `cd frontend && npm run build`
- **错误**: Rolldown 无法解析 `element-plus/es` 来自 `modules/im/frontend/index.vue`
- **根因**: IM 模块在 template 中使用 `<el-dialog>` `<el-button>`，主框架全局注册了 Element Plus，但 Rolldown 静态分析时发现模块直接引用了 Element Plus 内部路径
- **修复**: 检查 IM 模块是否有隐式导入 `element-plus/es`（可能通过 `auto-import` 插件）；或在 `vite.config.ts` 中添加 external 配置

### 🟡 级别 — 需修复

#### F4. `return ApiResponse(success=False)` 违反 API 契约（3 处）
- **位置**:
  - `backend/app/routers/workflow.py:80`
  - `backend/app/routers/workflow.py:121`
  - `backend/app/routers/file_shares.py:154`
- **描述**: 返回 HTTP 200 但 `success=False`，应该抛 `AppException` 子类让全局异常处理器返回正确状态码
- **规则**: 底层文档"API 契约"：禁止 `return ApiResponse(success=False)` 返回 200
- **修复**: 改为 `raise NotFound(...)` 或 `raise ValidationError(...)`

#### F5. `hello-world` 模块文档声明但磁盘不存在
- **位置**: `开发文档/README.md:85`
- **描述**: 文档列出的 `hello-world`（最小前端样板）模块在 `modules/` 中不存在
- **含义**: 要么补回模块，要么从文档中移除

#### F6. 模块使用框架未公开 API
- **位置**:
  - `modules/terminal-tools/backend/handlers/python.py:75` — 使用 `app.services.file_preview_service._resolve_storage_path`（**私有函数**）
  - `modules/docs-open/backend/handlers/content.py:17` — 使用 `app.schemas.document_ir`（未列入公开能力清单）
  - `modules/*-parser/backend/router.py` — 5 个文件解析模块直接 import `app.schemas.document_ir`
- **规则**: 框架能力清单以外的 `app.*` 均为内部实现，模块不应直接引用
- **修复**: 将 `document_ir` 加入框架公开接口清单，或模块自己定义；`_resolve_storage_path` 必须改为使用框架公开函数

### 🟢 级别 — 健康项目

#### G1. 跨模块导入零违规
- 零发现模块之间直接 `import modules/xxx`。跨模块调用全部经过 `register_capability` + `call_capability` 统一通路

#### G2. 无中文文件名违规
- 零发现 `开发文档/` 之外的目录/文件含中文名

#### G3. 前端零 `any` 绕过
- `frontend/src/` 中零 `as any`、零 `@ts-ignore`

#### G4. 文件 owner 隔离完整
- `file_service.py` 所有查询带 `owner_id` 过滤
- `check_file_access` 在下载、预览、文件读取等关键路径均有调用

#### G5. 模块表命名合规
- 所有模块业务表使用 `{key}_*` 前缀（`kb_*`, `agent_*`, `memory_*`, `im_*`）
- 零跨表 ForeignKey 到框架表或其他模块表
