type: audit — tags: [审计, 工程质量, 测试覆盖, 依赖, 构建, 状态管理, 事件系统, 运行时] — 2026-06-25

## 摘要

第四轮深度审计覆盖四个新维度：前端架构（组件大小/内存泄漏/Vue 模式）、测试覆盖与质量、依赖与构建配置、状态管理/数据流/事件系统。

---

## 🔴 CRITICAL

### C1. 无 CI/CD 流水线
- 无 `.github/workflows/`、`.gitlab-ci.yml`、`Jenkinsfile`
- **影响**: 测试不被持续验证，任何人都可以合入破坏性代码

### C2. 11 个后端包安装但完全未使用
- `backend/requirements.txt` 中 `docling`, `langchain`, `langchain-text-splitters`, `markitdown`, `easyocr`, `pandas`, `numpy`, `matplotlib`, `jieba`, `rapidfuzz`, `apscheduler` **在任何 `.py` 文件中无 import**
- **影响**: ~100MB+ 无用的安装包，增加攻击面和安全维护成本

### C3. 全局 Type 声明缺失 — 3 个 window 全局变量无类型
- `frontend/src/types/global.d.ts` 只声明了 `__HSWZ_CONFIG__` 和 `platform`
- **缺失**: `__HUASHI_RUNTIME__`、`__HSWZ_WINDOW_MANAGER__`、`__MODULE_OPEN_FILE_PAYLOAD__`
- **影响**: 所有通过 `(window as any)` 访问这些变量的地方（7+ 处）都绕过了类型检查

### C4. 模块设置断裂 — `module_settings` 始终为空
- `frontend/src/desktop/shell/use-desktop-app-loading.ts:51` — `module_settings: {}` 硬编码
- 后端 → 前端的模块设置通道**完全断裂**，模块的 `settings.get()` 永远返回默认值

### C5. 183 个测试是静态字符串搜索，非真正测试
- `backend/tests/test_agent_regression.py` (77)、`test_context_snapshot_lifecycle.py` (35)、`test_engine_batch2.py` (33)、`test_memory_core_paths.py` (38) — 合计 **183 个 `def test_` 函数**
- 这些文件只对 `.py` 源码做 `in` 字符串搜索（如 `"async def setup_global_hooks" in SRC`），**从不执行被测试代码**
- **影响**: 测试数虚高（全项目共 421 个测试定义，其中 183 个为假测试），核心逻辑零覆盖

## 🟡 HIGH

### H1. 无意义断言
- `backend/tests/test_agent_inline_tool_calls.py:36`:
  ```python
  assert clean == "查一下" or True  # 永远通过！
  ```
- **影响**: 该断言形同虚设，无论被测函数返回什么都通过

### H2. 20/24 模块无测试
- 只有 `codemap`、`knowledge`（2 个文件）、`office-gen` 有专用测试
- agent（后端有 30+ 集成测试但模块级零）、excel-engine、im、image-gen、terminal-tools、web-tools 等 20 个模块无任何测试

### H3. 13 处 `as never` 强制转换破坏事件系统类型安全
- 散布在 `use-app-event-bus.ts`、`request-response-channel.ts`、`desktop-app-handle-v2.ts`、`desktop/index.vue`、`recycle/index.vue`
- **影响**: mitt 事件总线的类型安全被完全绕过

### H4. `request-response-channel.ts` 注册全局监听器从不清理
- `frontend/src/desktop/app-registry/request-response-channel.ts:18,30`
- 两个 `emitter.on()` 监听器注册后**永远不从全局 emitter 移除**

### H5. 模块运行时 SDK 膨胀 — 25 份几乎相同的副本
- 每个 `modules/*/runtime/index.ts` 约 510 行，总计约 13,000 行几乎相同的代码
- `getApiUrl`、`authHeaders`、`_handle401`、`apiGet`、`apiPost`、`initRuntime` 在每个模块重复
- **影响**: 修改 API 层逻辑（如错误处理方式）需要改 25 个文件

### H6. 数据库迁移从未被测试
- `backend/migrations/` 有 3 个版本文件，零个对应测试
- **影响**: 生产环境 `alembic upgrade head` 可能失败且不被 CI 发现

### H7. Sandbox 构建工具版本与主前端严重脱节
- 所有 sandbox `package.json` 使用 `vite@^6.0`、`@vitejs/plugin-vue@^5.0`、`typescript@^5.0`
- 主前端使用 `vite@^8.0`、`@vitejs/plugin-vue@^6.0`、`typescript@~6.0`
- **影响**: sandbox 开发环境与主框架行为不一致

### H8. ESLint 配置过于简陋
- `eslint.config.mjs` 只启用了 `flat/essential` + `no-alert`
- **缺失关键规则**: `no-unused-vars`、`no-console`、`@typescript-eslint/no-explicit-any`、`vue/no-unused-components`
- **影响**: TypeScript 质量保证基本靠自觉

### H9. 组件直接调用 API 而非通过服务层
- `frontend/src/desktop/shell/index.vue` — 直接调用 `moveEntryRequest`、`emptyRecycleBinRequest`
- `frontend/src/platform/components/apps/desktop/index.vue` — 直接调用 `restoreRecycleBinEntry`、`permanentlyDeleteEntry`
- `frontend/src/shared/composables/use-notifications.ts` — 用 `api.get/post` 绕过模块 API 层

### H10. `knowledge/index.vue` 854 行 + `excel-engine/index.vue` 765 行
- 违反 Hard Rule 4（源文件不超过 600 行）
- 知识库前端: 模板嵌套 4 层条件分支，脚本逻辑密集，应拆为子组件

### H11. `scan-modules.js` 无错误处理
- `frontend/scripts/scan-modules.js:36` — `JSON.parse(fs.readFileSync(...))` 无 try-catch
- 任意 manifest.json 格式错误即导致构建脚本完全崩溃

### H12. 3 处内存泄漏风险
- `desktop-icon-grid.vue:93-94` — `mousedown` 时注册监听器，若在拖拽中卸载组件，监听器泄漏
- `fm-file-list.vue:100-101` — 同上模式
- `desktop-shell/index.vue:244-248` — 动态 `<input>` 及其 `change` 监听器从不移除
