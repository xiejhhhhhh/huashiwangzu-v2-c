## 4. 调研主线 A：用户真实业务场景闭环

## 5. 调研主线 B：桌面壳与用户体验

目标不是审美评价，而是判断它像不像一个可交付的桌面办公产品。

重点检查：

1. 启动器：用户能否快速找到模块/文件/命令？
2. 窗口系统：打开、切换、最小化、关闭、多窗口是否自然？
3. 任务栏：是否能表达正在运行的任务、后台任务、失败任务？
4. 通知中心：有没有真正承担“系统反馈”的作用？
5. 文件打开：不同类型文件能否打开到正确应用？失败时是否可理解？
6. 长任务：后台运行时用户是否无感但可追踪？
7. Agent：用户是否知道 Agent 在干什么，什么时候需要确认，什么时候已经完成？
8. 模块入口：模块多了以后是否像“应用生态”，还是像散乱按钮？
9. 新手路径：第一次打开产品，用户是否知道该从哪里开始？
10. 一致性：按钮、toast、弹窗、状态文案是否统一？

可以对照：

- Windows / macOS 桌面隐喻
- 飞书 / Notion / Slack / Linear 的任务与通知体验
- VS Code / Obsidian 的插件与工作区体验
- Open WebUI / Dify / AnythingLLM 等 AI 工作台体验

输出要求：

- 给出桌面体验成熟度评分。
- 列出 Top 10 桌面体验缺口。
- 每个缺口按“低成本快修 / 中等改造 / 大型升级”分类。
- 标注哪些建议依赖 Agent 工作流中枢或数据库反向链路先完成。

## 6. 调研主线 C：测试分层、发布门禁与交付效率

用户机器性能较强，但之前测试仍慢。请从工程化角度查原因。

重点检查：

1. pytest 配置是否重复、testpaths/addopts/markers 是否清楚？
2. 后端测试是否分层：unit / integration / live smoke / full regression？
3. Playwright 是否仍有串行、重复登录、重复截图、硬等待？
4. 前端测试是否每次启动 Vite，是否支持外部常驻服务？
5. module sandbox 是否可并发，哪些模块仍 skip？
6. release gate 是否能快速给出“能不能发”的结论？
7. smoke 是否存在假绿/假红风险？
8. 测试失败时是否能定位到责任模块？
9. 是否有机器可读 JSON 输出，方便多 Agent 合流？
10. 哪些测试应每天跑，哪些应提交前跑，哪些只在发布前跑？

特别注意：

- 当前已有数据库反向链路执行信在处理 `release_gate` / `smoke` / `module_sandbox_matrix` 的一部分。
- 本调研不要直接改这些文件，只输出建议和下一封执行信草案。

输出要求：

- 给出测试/发布效率成熟度评分。
- 列出 Top 10 慢测或发布门禁缺口。
- 给出一套推荐分层：

```text
quick check: 30-60 秒内
module check: 1-3 分钟
preflight release gate: 3-5 分钟
full regression: 可更久，但必须可并发、可解释
```

- 明确哪些是配置问题，哪些是测试设计问题，哪些是产品架构问题。
