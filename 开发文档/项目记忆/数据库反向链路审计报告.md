## 3. 最高风险问题 Top 10

1. **P1：模块通知接口权限过宽。** `POST /api/notifications/module` 只要求 viewer 即可创建 `published` 全局通知，且模块 runtime 暴露 `platform.notifications.notify()`，普通登录用户或模块可能发全员公告。
2. **P1：Agent 当前库没有主链路数据。** `agent_conversations/messages/events` 全部 0，release/smoke 又没有创建会话、发消息、验证落库，因此存在“Agent 绿了但主表没动”的假绿风险。
3. **P1：Memory 当前全空且无前端管理页。** `memory_records/chunks/links/experiences/stable_rules` 全部 0；Agent 代码会调用 memory，但当前库没有任何可召回内容。
4. **P1：知识库状态字段与派生产物漂移。** 画像、关系、图谱底表已有部分数据，但 `kb_documents.profile_status/graph_status/relation_status` 全 `pending`，前端若读底表会误判未完成。
5. **P2：任务队列 retry/cancel 缺状态限制。** 当前只按 owner/admin 判断，completed/running 任务可能被重排或取消，污染历史任务语义。
6. **P2：系统任务中心和 worker 治理缺前端入口。** `framework_system_task_queues` 有 156 条真实任务，但用户/管理员没有完整任务中心、worker audit、reconcile 页面。
7. **P2：内容包和资源是主链路但用户管理面弱。** `framework_content_packages`、versions、resources 有真实数据，前端主要通过 runtime 能力消费，缺通用包浏览、版本、资源、引用管理页面。
8. **P2：分享表有 API 但前端闭环不全。** `framework_file_shares` 当前 0，后端支持 create/sent/received/delete/check，但未见完整的发出分享、撤销、过期策略 UI。
9. **P2：日志/反馈/设置表存在但产品化不足。** `framework_system_logs` 有 1667 行，但管理查看与清理策略弱；`framework_system_feedbacks`、`framework_system_settings` 为 0，前端设置管理入口未接上。
10. **P2：多 worker 运行态仍有进程本地状态。** task worker 的 handler registry、last_active 等是 process-local，当前 smoke 未验证多 worker 注册一致性或重复消费。

## 13. 推荐修复路线

### Phase 1：主链路断链修复

1. 修 `POST /api/notifications/module` 权限，只允许系统身份、admin 或明确受限的模块服务身份发全局通知。
2. 给 release gate 增加真实 Agent 主链路：创建会话、发一条消息、验证 `agent_conversations/messages/events` 增量。
3. 给 release gate 增加真实 Memory 主链路：`memory:save` 后验证 records/chunks，再 `recall`。
4. 给 Knowledge release gate 增加上传/登记/pipeline 完成增量检查，至少验证 `kb_pipeline_runs`、`kb_chunks`、`kb_raw_data`。
5. 给任务队列 retry/cancel 加状态限制：只能 retry failed/cancelled，cancel 只能 pending/running 且要定义 running 取消语义。

### Phase 2：状态机统一

1. 统一 knowledge 状态枚举，删除 sandbox 中 `completed/processing` 旧口径。
2. 为 `kb_documents.profile_status/relation_status/graph_status` 建立派生产物回写或废弃这些字段，二选一。
3. 为 raw/fusion/profile/graph/relation 失败状态建立用户可见的 retry 入口。
4. 对文件 deleted 与 recycle item 建一致性 audit。
5. 给 notifications/feedback/settings 的状态字段加枚举约束和前端覆盖。

### Phase 3：前端消费补齐

1. 做系统任务中心：任务列表、详情、状态、重试/取消、worker audit、semantic failed。
2. 做分享管理：发出分享、收到分享、撤销、权限、过期。
3. 做设置/用户/角色矩阵管理页。
4. 做日志/反馈 admin 页。
5. 做 Memory 管理页：记忆列表、召回测试、删除、来源、embedding/chunk 状态。
6. 做内容包/资源浏览器：包、版本、资源引用、source file、权限。
7. 接入 imagegen history。

### Phase 4：死表/历史数据治理

1. 对所有 0 行但有代码引用的表打标：未启用、可选治理、必须主链路。
2. 对 `framework_file_recycle_items`、`framework_system_logs`、`kb_pipeline_stale` 建 retention 策略。
3. 明确 owner_id=0 的全局模板语义。
4. 对 artifact/workflow/private module 表决定近期是否产品化，不产品化就从主验收口径移出。

### Phase 5：release gate 纳入数据链路检查

1. 每个主模块至少一个“用户动作 -> DB 增量 -> 前端/API 可读”的黑箱检查。
2. release gate 输出每张主链路表的行数、关键状态分布、owner 分布。
3. 对 semantic failed、degraded、source_unavailable 建红线。
4. 对多 worker 任务消费做重复消费和 handler 注册一致性检查。
5. 对前端假闭环页面加 E2E：任务中心、知识库失败重试、Memory 管理、分享、反馈处理。
