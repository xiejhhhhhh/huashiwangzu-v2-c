## 🔴 阻断级

### D1. Element Plus 全局注册缺失 — 运行时崩溃
- **位置**: `frontend/src/main.ts`
- **描述**: 主框架的 Vue 入口**没有调用 `app.use(ElementPlus)`**，仅导入了 CSS。所有在 template 中使用的 `<el-config-provider>`、`<el-menu>`、`<el-dialog>`、`<el-button>`、`<el-input>` **在运行时不会渲染**。
- **对比**: sandbox 模块（`modules/*/sandbox/src/main.ts`）**正确**调用了 `app.use(ElementPlus)`，印证了主框架的遗漏。
- **受影响的组件**:
  - `frontend/src/App.vue:2` — `<el-config-provider>`
  - `frontend/src/app-entry/layouts/main-layout.vue:7-12` — `<el-menu>`, `<el-menu-item>`
  - `modules/im/frontend/index.vue:47-58` — `<el-dialog>`, `<el-button>`
  - `modules/knowledge/frontend/index.vue:153-159` — `<el-dialog>`, `<el-button>`
  - `modules/agent/frontend/admin/ApprovalPanel.vue:33-40` — `<el-dialog>`, `<el-input>`, `<el-button>`
- **修复**: 在 `frontend/src/main.ts` 添加 `app.use(ElementPlus)`（按需导入+全局注册）

### D2. MD5 去重 TOC/TOU 竞态条件 — 重复写盘/引用计数漂移
- **位置**:
  - `backend/app/services/file_upload_service.py:62-109`
  - `backend/app/services/file_create_service.py:132-162`
- **描述**: 内容去重流程先 SELECT 查 md5_hash，再决定 INSERT 还是 increment ref_count。两个步骤**没有 `FOR UPDATE` 保护**，多 worker 并发上传同一内容时：
  ```
  Worker A: SELECT md5='abc' → 未找到
  Worker B: SELECT md5='abc' → 未找到
  Worker A: 写磁盘, INSERT File(ref_count=1)
  Worker B: 写磁盘(同名), INSERT File(ref_count=1)  ← 重复
  ```
  结果：磁盘文件写两次，ref_count 永远不对称，永久删除时残留。
- **修复**: MD5 检查全部改用 `select(File).where(File.md5_hash == md5).with_for_update()`

### D3. 多 Worker 断路器互相不可见 — 降级失效
- **位置**: `backend/app/services/circuit_breaker.py:85` — `_breakers: dict[str, CircuitBreaker] = {}`
- **描述**: 断路器状态是进程内内存变量。`--workers 3` 下，某个 provider 连续失败 5 次只熔断 Worker A，Worker B 和 C 继续调用继续失败。熔断降级形同虚设。
- **修复**: 断路器状态持久化到 DB（`framework_circuit_breaker_states` 表）或用文件锁做协同

### D4. 前端构建失败
- **位置**: `cd frontend && npm run build`
- **错误**: Rolldown 无法解析 `element-plus/es` 来自 `modules/im/frontend/index.vue`
- **原因**: IM 模块在 template 中使用 `<el-dialog>` `<el-button>`，但 Rolldown 构建时静态分析发现 Element Plus 组件未在模块级注册。D1 是根本原因的一部分——即使全局注册，Rolldown 也可能因模块与主框架的构建隔离报错。
- **状态**: 第一轮已记录，第二轮确认仍存在。修复 D1 后仍需单独排查

### D5. 模块引用框架私有函数
- **位置**: `modules/terminal-tools/backend/handlers/python.py:75`
  ```python
  from app.services.file_preview_service import _resolve_storage_path
  ```
- **描述**: 使用了**下划线前缀的私有函数**。这是最强硬的违规——私有函数无契约、无稳定签名、随时可能被重构删除。
- **其他非私有但未公开的 import**: 见 M2 清单

## 🟡 中高危

### M1. `ref_count` 递减逻辑缺陷
- **位置**: `backend/app/services/file_upload_service.py:189-206`
- **描述**: `replace_file_content()` 中旧内容 ref_count 递减循环用 `break` 退出，**只减了第一个"其他"文件**。如果多个文件引用同一旧内容，只减一处，ref_count 逐渐漂高。
- **修复**: 去掉 `break`，对所有非自身文件都执行递减

### M3. `batch_delete` / `batch_move` 循环内 commit + rollback 导致部分提交
- **位置**: `backend/app/routers/files.py:93-129`
- **描述**: 循环内每项 `db.commit()`，失败时 `db.rollback()`。但前 N 项已提交无法回滚，产生**部分删除/移动**。rollback 后还可能导致"no transaction is active" SQLAlchemy 错误。
- **修复**: 改用 `db.begin_nested()` 逐项做 savepoint，或者全部收集后一次性 commit

### M4. `window as any` 全局桥接模式 — 7 处
- **位置**: 7 个模块的 `frontend/index.vue` 通过 `(window as any).__MODULE_OPEN_FILE_PAYLOAD__` 读取框架注入的文件打开参数
- **问题**: 绕过 TypeScript 类型检查，`__MODULE_OPEN_FILE_PAYLOAD__` 未在 `global.d.ts` 声明
- **修复**: 在 `global.d.ts` 添加 `Window` 类型增强；模块侧用 `window as Window` 而非 `window as any`

### M5. 多 Worker 模块加载错误 / 重试节流不共享
- **位置**: `backend/app/routers/registry.py:15,23`
- `_module_load_errors` — 每个 worker 独立，健康检查 endpoint 看到的错误不一致
- `_retry_last_attempt` — 每个 worker 独立，3 个 worker 会同时重试故障模块
- **修复**: 改为 DB 持久化或透传聚合

### M6. 文件锁跨进程竞态（codemap）
- **位置**: `modules/codemap/backend/locks/file_lock.py:76`
- **描述**: `threading.Lock()` 只保护同进程内的线程，不保护跨进程。3 个 worker 同时 acquire_lock 时，读-check-写三个步骤可以被交叉：
  ```
  Worker A: 读 locks.json → 空闲
  Worker B: 读 locks.json → 空闲
  Worker A: 写 locks.json → 锁住
  Worker B: 写 locks.json → 覆盖 A ← 锁丢失
  ```
- **缓解**: TTL 自动过期（600s），但丢失锁期间可能产生冲突写入
- **修复**: 对 `locks.json` 文件加 POSIX `fcntl.flock()`

### M7. 大量裸 SQL 在模块中（excel-engine / memory / agent）
- **位置**:
  - `modules/excel-engine/backend/state/db_ops.py` — 27 处 `text(f"DELETE FROM {Table.__tablename__} ...")`
  - `modules/memory/backend/` — 25 处 `text(...)` 调用（向量相似度 JOIN、统计）
  - `modules/agent/backend/` — 27 处 `text(...)`（含 `f"IN ({ids_str})"` 拼接, `context_snapshot.py:269,288`）
- **风险**: 虽目前表名来自 `__tablename__` 类属性（安全），`IN` 列表来自 ORM 查询结果（安全），但模式脆弱。`IN` 拼接尤其危险——未来若 input 溯源改变即产生 SQL 注入
- **修复**: `IN` 查询改用 `in_(...)` 或绑定参数数组；逐步将裸露 SQL 用 ORM 标准查询替代

### M8. 异步批量 operations 的 session 管理风险
- **位置**: `modules/agent/backend/engine/post_turn_hooks.py:270`
- `run_hooks()` 接受 `db` 参数但**从不使用**，每个 hook 自建 session。容易混淆维护者
  ```python
  async def run_hooks(self, db: AsyncSession, ...):  # db 参数未使用
  ```

## 🟢 健康部分

### G1. 跨模块 import 零违规（代码层）
所有模块 backend 代码**无一处** import 其他模块的代码，全部经过 `register_capability` / `call_capability` 统一通路。

### G2. 跨模块表访问零违规
模块只读写自己的 `{key}_*` 表。SQL 查询中未发现引用其他模块前缀表名。

### G3. ForeignKey 零违规
所有模块模型只使用 `Integer`/`BigInteger` 做跨实体关联，**无 ForeignKey 约束到框架表或其他模块表**。

### G4. 模块表名前缀合规
所有模块 `__tablename__` 使用模块 key 前缀（`agent_*`, `kb_*`, `memory_*`, `im_*`, `excel_*`, `codemap_*`, `docs_open_*`）

### G5. manifest 全合规
全部 24 个 manifest.json 的 `key`、`backend.router`、`public_actions`、`enabled` 均合规。router 文件存在性验证通过。

### G6. 后台任务 worker 正确使用 `FOR UPDATE SKIP LOCKED`
`task_worker.py` 中的 `_recover_stale_tasks()`、`_recover_orphan_running_tasks()`、`_claim_one_task()` 全部正确使用 `with_for_update(skip_locked=True)`，是标准的跨 worker 抢占模式。

### G7. `noqa` / `# type: ignore` 全部合法
9 处 `noqa` 均用于合法场景（模型注册、全局变量声明、能力注册延迟导入）。1 处 `# type: ignore[arg-type]` 用于 AST 类型兼容，正确。

### G8. `text()` vs `func.text()` 无混淆
未发现 `func.text()` 误用。memory 模块的模型使用 `sa_text("0")` + `sa_func.now()`，是正确做法。
